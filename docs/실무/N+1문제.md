# N+1 문제

## 왜 알아야 하는가?

JPA를 사용하는 거의 모든 프로젝트에서 발생하는 성능 함정이다. 코드상으로는 아무 문제가 없어 보이지만, 실제로 수십~수백 개의 불필요한 SQL 쿼리가 실행되어 애플리케이션 성능을 급격히 저하시킨다. 개발 환경에서는 데이터가 적어 눈치채지 못하다가 운영에서 터지는 경우가 많다.

---

## N+1 문제란?

**1번의 쿼리를 실행한 후, 그 결과 N개에 대해 각각 추가 쿼리가 N번 더 실행되는 현상.**

```
전체 목록 조회: 1번 쿼리
각 항목의 연관 데이터 조회: N번 쿼리
─────────────────────────────────
총 실행 쿼리: 1 + N 번
```

예: 주문 100개를 조회하면 → 1번 + 각 주문의 고객 정보 100번 = 총 101번 쿼리

---

## 발생 원인

JPA의 **Lazy Loading(지연 로딩)** 전략 때문에 발생한다.

```
[Eager Loading] - N+1 문제를 피하려다 다른 문제 발생
엔티티 로드 시 연관 엔티티를 즉시 함께 로드
→ 사용하지 않는 데이터도 항상 가져옴 (비효율)

[Lazy Loading] - JPA 기본 전략 (권장)
연관 엔티티는 실제 접근할 때 로드
→ 필요한 데이터만 가져오지만, 루프에서 접근 시 N+1 발생
```

---

## 발생 상황 예시

### 도메인 모델

```java
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members = new ArrayList<>();
}

@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "team_id")
    private Team team;
}
```

### N+1이 발생하는 코드

```java
// Repository
List<Team> teams = teamRepository.findAll(); // 쿼리 1번

// 루프에서 연관 엔티티 접근
for (Team team : teams) {
    System.out.println(team.getMembers().size()); // 각 Team마다 쿼리 1번씩
}
```

### 실제 실행되는 SQL

```sql
-- 1번: 팀 전체 조회
SELECT * FROM team;

-- N번: 각 팀의 멤버 조회 (team이 10개면 10번)
SELECT * FROM member WHERE team_id = 1;
SELECT * FROM member WHERE team_id = 2;
SELECT * FROM member WHERE team_id = 3;
-- ... 10개 team이면 10번 더 실행
```

```
팀 10개 → 쿼리 11번
팀 100개 → 쿼리 101번
팀 1000개 → 쿼리 1001번
```

## Hibernate SQL 로깅 설정

N+1 문제를 직접 확인하려면 SQL 로깅이 필수다.

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        format_sql: true
        show_sql: true

logging:
  level:
    org.hibernate.SQL: DEBUG
    org.hibernate.orm.jdbc.bind: TRACE   # 바인딩 파라미터 확인
```

더 정확한 쿼리 수 확인은 p6spy 라이브러리를 사용한다:

```gradle

```

```
logMessageFormat=com.p6spy.engine.spy.appender.CustomLineFormat
customLogMessageFormat=%(executionTime)ms|%(sql)
```

## 해결 방법 1: Fetch Join (JPQL)

연관 엔티티를 한 번의 JOIN 쿼리로 함께 가져온다.

```java
// Repository
public interface TeamRepository extends JpaRepository<Team, Long> {

    @Query("SELECT t FROM Team t JOIN FETCH t.members")
    List<Team> findAllWithMembers();
}
```

```sql
-- 실행되는 SQL: 단 1번
SELECT t.*, m.*
FROM team t
INNER JOIN member m ON t.id = m.team_id;
```

```java
// 사용
List<Team> teams = teamRepository.findAllWithMembers();
for (Team team : teams) {
    System.out.println(team.getMembers().size()); // 추가 쿼리 없음
}
```

### Fetch Join 주의사항: 페이징과의 충돌

```java
// 이렇게 하면 경고 발생 + 메모리에서 페이징 (매우 위험)
@Query("SELECT t FROM Team t JOIN FETCH t.members")
Page<Team> findAllWithMembersPageable(Pageable pageable);
// Hibernate 경고: HHH90003004: firstResult/maxResults specified with collection fetch; applying in memory
```

컬렉션 fetch join + 페이징은 함께 쓸 수 없다. 이 경우 BatchSize를 사용한다.

## 해결 방법 2: @EntityGraph

JPQL을 작성하지 않고 어노테이션으로 fetch join 효과를 낸다.

```java
public interface TeamRepository extends JpaRepository<Team, Long> {

    // attributePaths에 지정한 연관관계를 LEFT JOIN FETCH로 가져옴
    @EntityGraph(attributePaths = {"members"})
    List<Team> findAll();

    // 메서드 이름 쿼리와 함께 사용 가능
    @EntityGraph(attributePaths = {"members"})
    List<Team> findByName(String name);
}
```

```java
// Entity에 NamedEntityGraph 정의 후 사용
@Entity
@NamedEntityGraph(
    name = "Team.withMembers",
    attributeNodes = @NamedAttributeNode("members")
)
public class Team {
    // ...
}

// Repository
@EntityGraph("Team.withMembers")
List<Team> findAll();
```

## 해결 방법 3: Batch Size

N+1을 1+1로 줄인다. N번의 쿼리 대신 IN 절을 사용해 묶어서 처리한다.

```yaml
# application.yml (전역 설정)
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

```java
// 또는 특정 컬렉션에만 적용
@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;

    @BatchSize(size = 100)
    @OneToMany(mappedBy = "team", fetch = FetchType.LAZY)
    private List<Member> members = new ArrayList<>();
}
```

### 실행되는 SQL

```sql
-- 1번: 팀 전체 조회
SELECT * FROM team;

-- 1번: IN 절로 멤버 한 번에 조회 (100개씩 묶음)
SELECT * FROM member WHERE team_id IN (1, 2, 3, ..., 100);
```

```
팀 100개 → 쿼리 2번 (팀 1번 + 멤버 IN절 1번)
팀 250개 → 쿼리 4번 (팀 1번 + 멤버 100, 100, 50씩 3번)
```

**페이징과 함께 사용 가능**하다는 것이 큰 장점이다.

## 해결 방법 4: DTO 프로젝션

엔티티 대신 필요한 컬럼만 담은 DTO로 직접 조회한다.

```java
// DTO
public class TeamMemberDto {
    private Long teamId;
    private String teamName;
    private Long memberId;
    private String memberName;

    // JPQL 생성자 표현식용 생성자
    public TeamMemberDto(Long teamId, String teamName, Long memberId, String memberName) {
        this.teamId = teamId;
        this.teamName = teamName;
        this.memberId = memberId;
        this.memberName = memberName;
    }
}
```

```java
// Repository
public interface TeamRepository extends JpaRepository<Team, Long> {

    @Query("""
        SELECT new com.example.dto.TeamMemberDto(
            t.id, t.name, m.id, m.name
        )
        FROM Team t
        JOIN t.members m
        """)
    List<TeamMemberDto> findAllTeamMemberDtos();
}
```

```java
// 또는 interface 기반 프로젝션
public interface TeamMemberView {
    Long getTeamId();
    String getTeamName();
    Long getMemberId();
    String getMemberName();
}

@Query("""
    SELECT t.id as teamId, t.name as teamName,
           m.id as memberId, m.name as memberName
    FROM Team t JOIN t.members m
    """)
List<TeamMemberView> findAllAsView();
```

## 해결 방법 비교 표

| 방법 | 쿼리 수 | 페이징 가능 | 복잡도 | 적합한 상황 |
|------|---------|------------|--------|-------------|
| Fetch Join | 1번 | X (컬렉션) | 낮음 | 페이징 없는 단순 조회 |
| @EntityGraph | 1번 | X (컬렉션) | 낮음 | 어노테이션 선호 시 |
| Batch Size | 1+a번 | O | 낮음 | 페이징 + 컬렉션 조회 |
| DTO 프로젝션 | 1번 | O | 중간 | 특정 컬럼만 필요할 때 |

## CartesianProduct 문제 (컬렉션 2개 Fetch Join)

컬렉션을 2개 이상 fetch join하면 카테시안 곱이 발생한다.

```java
// 위험한 코드
@Query("SELECT t FROM Team t JOIN FETCH t.members JOIN FETCH t.projects")
List<Team> findAll(); // MultipleBagFetchException 또는 중복 데이터 폭발
```

### 발생하는 문제

```
Team 1 - Members: [A, B] - Projects: [P1, P2]

카테시안 곱 결과:
Team1 | A | P1
Team1 | A | P2
Team1 | B | P1
Team1 | B | P2
→ 4행 반환 (실제는 1개 팀, 멤버 2명, 프로젝트 2개)
```

### 해결 방법

```java
// 방법 1: 하나만 Fetch Join, 나머지는 BatchSize
@Query("SELECT DISTINCT t FROM Team t JOIN FETCH t.members")
List<Team> findAllWithMembers();
// projects는 default_batch_fetch_size로 처리

// 방법 2: 쿼리 분리
List<Team> teams = teamRepository.findAllWithMembers();
// 별도 쿼리로 프로젝트 조회 후 매핑

// 방법 3: DTO 프로젝션 사용
```

## @ManyToOne에서의 N+1

컬렉션뿐 아니라 @ManyToOne에서도 발생한다.

```java
// N+1 발생
List<Member> members = memberRepository.findAll(); // 쿼리 1번
for (Member member : members) {
    System.out.println(member.getTeam().getName()); // 각 멤버마다 team 조회
}
```

```java
// 해결: Fetch Join
@Query("SELECT m FROM Member m JOIN FETCH m.team")
List<Member> findAllWithTeam();
```

---

## 발생 케이스 정리

| 상황 | N+1 발생 여부 | 설명 |
|------|--------------|------|
| findAll() 후 컬렉션 접근 | O | 가장 흔한 케이스 |
| findAll() 후 @ManyToOne 접근 | O | 단건이지만 N번 반복 |
| @OneToOne LAZY | O | 영속성 컨텍스트에 없는 경우 |
| Fetch Join 사용 | X | 해결됨 |
| BatchSize 설정 | X (감소) | IN절로 묶음 |
| Eager Loading | X (다른 문제) | 불필요한 데이터도 로드 |

## 실무 권장 패턴

```
1. 기본은 모든 연관관계에 LAZY 설정
2. 성능이 중요한 조회는 Fetch Join 또는 DTO 프로젝션
3. 페이징이 필요하면 BatchSize + Fetch Join 조합
4. 개발 단계에서 p6spy로 쿼리 수 모니터링
5. 운영에서는 SlowQuery 로그로 감시
```

```java
// 실무에서 자주 쓰는 패턴: 목록은 DTO, 상세는 Fetch Join
public interface OrderRepository extends JpaRepository<Order, Long> {

    // 목록 조회: 필요한 컬럼만
    @Query("""
        SELECT new com.example.dto.OrderListDto(
            o.id, o.status, o.totalAmount, u.name
        )
        FROM Order o JOIN o.user u
        """)
    Page<OrderListDto> findOrderList(Pageable pageable);

    // 상세 조회: 연관 엔티티 모두 로드
    @Query("""
        SELECT o FROM Order o
        JOIN FETCH o.user
        JOIN FETCH o.orderItems oi
        JOIN FETCH oi.product
        WHERE o.id = :orderId
        """)
    Optional<Order> findDetailById(@Param("orderId") Long orderId);
}
```

## 요약 정리

```
N+1 문제 = 1번 조회 후 N개 연관 데이터를 N번 따로 조회하는 현상

원인: JPA Lazy Loading + 루프에서 연관 엔티티 접근

해결:
├── 페이징 없음: Fetch Join / @EntityGraph
├── 페이징 있음: Batch Size (default_batch_fetch_size: 100)
└── 읽기 전용: DTO 프로젝션

주의:
├── 컬렉션 2개 Fetch Join → MultipleBagFetchException
├── 컬렉션 Fetch Join + 페이징 → 메모리 페이징 (위험)
└── Eager Loading은 해결책이 아님
```
