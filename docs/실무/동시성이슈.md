# 동시성 이슈 (Concurrency Issues)

---

## 왜 공부하는가?

실무에서 재고 차감, 포인트 적립/차감, 좋아요 카운트, 선착순 이벤트 처리 등 여러 사용자가 동시에 같은 자원에 접근하는 상황은 매우 흔하다.
이를 제대로 처리하지 않으면 **데이터 정합성이 깨지는 심각한 버그**가 발생한다.

---

## 동시성 문제란?

**여러 스레드 또는 프로세스가 동시에 공유 자원(메모리, DB 레코드 등)에 접근할 때** 실행 순서에 따라 결과가 달라지는 문제.

```
Thread A: 재고 읽기(10) → 차감 연산(10-1=9) → 저장(9)
Thread B: 재고 읽기(10) → 차감 연산(10-1=9) → 저장(9)

기대 결과: 8
실제 결과: 9  ← 데이터 정합성 오류!
```

---

## Race Condition (경쟁 조건)

### 정의

두 개 이상의 스레드가 공유 자원에 동시에 접근하여 **읽기-수정-쓰기(Read-Modify-Write)** 를 수행할 때, 실행 순서에 따라 결과가 달라지는 현상.

### 발생하는 대표 시나리오

| 시나리오 | 증상 |
|---|---|
| 재고 차감 | 재고가 0 이하로 내려감 (오버셀링) |
| 포인트 차감 | 잔고보다 많이 차감됨 |
| 좋아요 카운트 | 동시 클릭 시 카운트가 누락됨 |
| 선착순 쿠폰 발급 | 정해진 수량보다 많이 발급됨 |
| 회원 가입 중복 | 같은 이메일로 중복 가입됨 |

### Java에서 Race Condition 발생 예시

```java
// 재고 차감 - Race Condition 발생 코드
@Service
public class StockService {
    private int stock = 100; // 공유 자원

    public void decrease() {
        if (stock <= 0) {
            throw new IllegalStateException("재고 부족");
        }
        stock--; // 원자적 연산이 아님! (읽기 → 감소 → 쓰기)
    }
}
```

```java
// 테스트: 100개 스레드가 동시에 차감
@Test
void raceConditionTest() throws InterruptedException {
    int threadCount = 100;
    ExecutorService executorService = Executors.newFixedThreadPool(32);
    CountDownLatch latch = new CountDownLatch(threadCount);

    for (int i = 0; i < threadCount; i++) {
        executorService.submit(() -> {
            try {
                stockService.decrease();
            } finally {
                latch.countDown();
            }
        });
    }
    latch.await();

    // 기대값: 0, 실제값: 0이 아닐 가능성 높음
    System.out.println("최종 재고: " + stockService.getStock());
}
```

## 해결 방법

### 1. synchronized (단일 서버, JVM 레벨)

**왜 사용하는가?**
JVM 내에서 한 번에 하나의 스레드만 해당 메서드/블록에 접근하도록 강제한다.

**어떻게 사용하는가?**

```java
@Service
public class StockService {
    private int stock = 100;

    // 메서드 레벨 동기화
    public synchronized void decrease() {
        if (stock <= 0) {
            throw new IllegalStateException("재고 부족");
        }
        stock--;
    }

    // 블록 레벨 동기화 (더 세밀한 제어)
    public void decreaseBlock() {
        synchronized (this) {
            if (stock <= 0) {
                throw new IllegalStateException("재고 부족");
            }
            stock--;
        }
    }
}
```

> **주의:** `@Transactional`과 함께 쓸 때 문제 발생 가능
> Spring의 `@Transactional`은 프록시 기반이라 `synchronized`가 트랜잭션 커밋 전에 해제될 수 있다.

**장점**
- 구현 간단
- 추가 의존성 없음

**단점**
- 다중 서버(분산 환경)에서 동작하지 않음
- 성능 저하 (병목)
- `@Transactional`과 함께 사용 시 주의 필요

**어떨 때 쓰는가?**
단일 서버, 간단한 카운터, 내부 캐시 등 JVM 내에서만 공유되는 자원.

---

### 2. ReentrantLock (단일 서버, 더 유연한 락)

**왜 사용하는가?**
`synchronized`보다 더 세밀한 제어 (타임아웃, 공정성 설정, tryLock 등)가 필요할 때.

```java
@Service
public class StockService {
    private int stock = 100;
    private final ReentrantLock lock = new ReentrantLock(true); // true = 공정 락

    public void decrease() {
        lock.lock();
        try {
            if (stock <= 0) {
                throw new IllegalStateException("재고 부족");
            }
            stock--;
        } finally {
            lock.unlock(); // 반드시 finally에서 해제
        }
    }

    // 타임아웃이 있는 락 획득 시도
    public boolean tryDecrease() throws InterruptedException {
        if (lock.tryLock(3, TimeUnit.SECONDS)) {
            try {
                stock--;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 3초 내 락 획득 실패
    }
}
```

**장점**
- tryLock으로 데드락 방지
- 공정 락(fairness) 설정 가능
- 조건 변수(Condition) 사용 가능

**단점**
- synchronized보다 코드가 복잡
- unlock()을 finally에서 반드시 호출해야 함
- 분산 환경 미지원

---

### 3. Atomic 클래스 (AtomicInteger, AtomicLong)

**왜 사용하는가?**
CAS(Compare-And-Swap) 연산을 사용해 락 없이 원자적 연산을 보장한다. 성능이 좋다.

```java
@Service
public class StockService {
    private AtomicInteger stock = new AtomicInteger(100);

    public void decrease() {
        int current;
        do {
            current = stock.get();
            if (current <= 0) {
                throw new IllegalStateException("재고 부족");
            }
        } while (!stock.compareAndSet(current, current - 1));
        // CAS: current값이 변경되지 않았으면 current-1로 설정
    }

    // 더 간단한 방법
    public void simpleDecrease() {
        if (stock.decrementAndGet() < 0) {
            stock.incrementAndGet(); // 롤백
            throw new IllegalStateException("재고 부족");
        }
    }
}
```

**장점**
- 락보다 성능이 좋음 (Lock-free)
- 구현 간단

**단점**
- 분산 환경 미지원
- 복잡한 비즈니스 로직엔 부적합 (단순 카운터에 적합)
- ABA 문제 발생 가능 (AtomicStampedReference로 해결)

### 4. DB 비관적 락 (Pessimistic Lock, SELECT FOR UPDATE)

**왜 사용하는가?**
충돌이 자주 발생하는 환경에서 데이터를 읽는 시점에 락을 걸어 다른 트랜잭션의 접근을 막는다.

**어떻게 사용하는가?**

```java
// Repository
public interface StockRepository extends JpaRepository<Stock, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT s FROM Stock s WHERE s.id = :id")
    Optional<Stock> findByIdWithPessimisticLock(@Param("id") Long id);
}
```

```sql
-- 실제 실행되는 SQL
SELECT * FROM stock WHERE id = 1 FOR UPDATE;
-- 다른 트랜잭션은 해당 레코드의 락이 해제될 때까지 대기
```

```java
@Service
@RequiredArgsConstructor
public class StockService {
    private final StockRepository stockRepository;

    @Transactional
    public void decrease(Long stockId, Long quantity) {
        Stock stock = stockRepository.findByIdWithPessimisticLock(stockId)
            .orElseThrow(() -> new EntityNotFoundException("재고 없음"));
        stock.decrease(quantity);
    }
}
```

**장점**
- 충돌이 많은 환경에서 신뢰성 높음
- 데이터 정합성 보장이 확실함

**단점**
- 성능 저하 (락 대기 시간 발생)
- 데드락 위험
- 분산 DB 환경에서는 성능 문제 심화

**어떨 때 쓰는가?**
재고 차감처럼 충돌 가능성이 높고, 정합성이 매우 중요한 경우.

---

### 5. DB 낙관적 락 (Optimistic Lock, @Version)

**왜 사용하는가?**
충돌이 적은 환경에서 버전 번호를 이용해 동시 수정을 감지한다. 실제 락을 걸지 않아 성능이 좋다.

```java
@Entity
public class Stock {
    @Id
    @GeneratedValue
    private Long id;

    private Long quantity;

    @Version // 낙관적 락의 핵심: 업데이트 시마다 자동 증가
    private Long version;

    public void decrease(Long quantity) {
        if (this.quantity < quantity) {
            throw new IllegalStateException("재고 부족");
        }
        this.quantity -= quantity;
    }
}
```

```java
// Repository
public interface StockRepository extends JpaRepository<Stock, Long> {

    @Lock(LockModeType.OPTIMISTIC)
    @Query("SELECT s FROM Stock s WHERE s.id = :id")
    Optional<Stock> findByIdWithOptimisticLock(@Param("id") Long id);
}
```

```java
// 서비스: 충돌 시 재시도 로직 필요
@Service
@RequiredArgsConstructor
public class OptimisticLockStockFacade {
    private final StockRepository stockRepository;

    public void decrease(Long id, Long quantity) throws InterruptedException {
        while (true) {
            try {
                stockRepository.findByIdWithOptimisticLock(id)
                    .ifPresent(stock -> {
                        stock.decrease(quantity);
                        stockRepository.save(stock);
                    });
                break;
            } catch (OptimisticLockingFailureException e) {
                // 버전 충돌 시 재시도
                Thread.sleep(50);
            }
        }
    }
}
```

```sql
-- 실제 실행되는 UPDATE SQL (version 조건 포함)
UPDATE stock SET quantity = 9, version = 2 WHERE id = 1 AND version = 1;
-- 다른 트랜잭션이 먼저 수정했다면 version이 이미 2이므로 0 rows updated → 예외 발생
```

**장점**
- 락을 걸지 않아 성능 좋음
- 충돌이 적은 환경에 적합

**단점**
- 충돌 시 재시도 로직을 직접 구현해야 함
- 충돌이 잦으면 재시도가 많아져 성능 저하

**어떨 때 쓰는가?**
동시 수정 가능성이 낮고, 충돌 시 실패해도 괜찮은 경우 (예: 게시글 수정).

### 6. Redis 분산 락 (Distributed Lock)

**왜 사용하는가?**
다중 서버(분산 환경)에서 JVM 레벨의 락은 효과가 없다. Redis를 중앙 락 저장소로 사용하면 여러 서버 간 동기화가 가능하다.

#### Redisson 사용 (추천)

```java
// build.gradle
// implementation 'org.redisson:redisson-spring-boot-starter:3.23.4'

@Service
@RequiredArgsConstructor
public class RedissonLockStockService {
    private final RedissonClient redissonClient;
    private final StockRepository stockRepository;

    public void decrease(Long key, Long quantity) throws InterruptedException {
        RLock lock = redissonClient.getLock("stock:" + key);

        try {
            // 락 획득 시도: 최대 5초 대기, 락 유지 시간 3초
            boolean available = lock.tryLock(5, 3, TimeUnit.SECONDS);
            if (!available) {
                throw new RuntimeException("락 획득 실패");
            }

            Stock stock = stockRepository.findById(key)
                .orElseThrow(() -> new EntityNotFoundException("재고 없음"));
            stock.decrease(quantity);
            stockRepository.save(stock);

        } finally {
            lock.unlock(); // 반드시 해제
        }
    }
}
```

#### Lettuce 사용 (Spin Lock 방식)

```java
// Lettuce는 기본적으로 spin lock (CPU 낭비) 방식
@Component
@RequiredArgsConstructor
public class RedisLockRepository {
    private final RedisTemplate<String, String> redisTemplate;

    public Boolean lock(Long key) {
        return redisTemplate.opsForValue()
            .setIfAbsent("lock:" + key, "lock", Duration.ofMillis(3_000));
    }

    public Boolean unlock(Long key) {
        return redisTemplate.delete("lock:" + key);
    }
}

@Service
@RequiredArgsConstructor
public class LettuceLockStockFacade {
    private final RedisLockRepository redisLockRepository;
    private final StockRepository stockRepository;

    public void decrease(Long id, Long quantity) throws InterruptedException {
        while (!redisLockRepository.lock(id)) {
            Thread.sleep(100); // spin lock: Redis 부하 주의
        }
        try {
            Stock stock = stockRepository.findById(id)
                .orElseThrow();
            stock.decrease(quantity);
            stockRepository.save(stock);
        } finally {
            redisLockRepository.unlock(id);
        }
    }
}
```

**Redisson vs Lettuce 비교**

| 항목 | Redisson | Lettuce |
|---|---|---|
| 락 방식 | Pub/Sub (효율적) | Spin Lock (CPU 낭비) |
| 타임아웃 설정 | 기본 지원 | 직접 구현 |
| 재시도 로직 | 자동 처리 | 직접 구현 |
| 구현 복잡도 | 낮음 | 높음 |
| 추천 여부 | 추천 | 간단한 경우만 |

### 7. 메시지 큐 (순차 처리)

**왜 사용하는가?**
동시성 문제를 근본적으로 없애는 방법. 모든 요청을 큐에 넣고 순차적으로 처리.

```
요청 A → [Message Queue] → Consumer → 처리
요청 B → [Message Queue] → Consumer → 처리 (A 완료 후)
요청 C → [Message Queue] → Consumer → 처리 (B 완료 후)
```

```java
// Kafka Producer
@Service
@RequiredArgsConstructor
public class StockProducer {
    private final KafkaTemplate<String, StockDecreaseEvent> kafkaTemplate;

    public void sendDecreaseEvent(Long stockId, Long quantity) {
        kafkaTemplate.send("stock-decrease", new StockDecreaseEvent(stockId, quantity));
    }
}

// Kafka Consumer (단일 파티션 = 순차 처리 보장)
@Service
@RequiredArgsConstructor
public class StockConsumer {
    private final StockRepository stockRepository;

    @KafkaListener(topics = "stock-decrease", groupId = "stock-group")
    public void consume(StockDecreaseEvent event) {
        stockRepository.findById(event.getStockId())
            .ifPresent(stock -> {
                stock.decrease(event.getQuantity());
                stockRepository.save(stock);
            });
    }
}
```

**장점**
- 동시성 문제가 구조적으로 사라짐
- 시스템 부하 분산 효과

**단점**
- 처리 지연 발생 (즉각적인 결과 반환 어려움)
- 인프라 복잡도 증가
- 실패 처리(DLQ) 별도 구현 필요

## 분산 환경에서의 동시성

### 단일 서버 락이 분산 환경에서 동작하지 않는 이유

```
[서버 A] - synchronized(lock) → 재고 읽기(10) → 차감 → 저장(9)
[서버 B] - synchronized(lock) → 재고 읽기(10) → 차감 → 저장(9)
                                                         ↑
                         서버 B는 서버 A의 JVM 락을 모른다!
```

각 서버는 독립적인 JVM을 가지므로, `synchronized`나 `ReentrantLock`은 **같은 JVM 내에서만** 유효하다.

분산 환경에서는 반드시 **외부 공유 저장소(Redis, ZooKeeper, DB)** 를 통한 락이 필요하다.

## 각 방법 비교 표

| 방법 | 적용 환경 | 성능 | 구현 복잡도 | 데이터 정합성 | 비고 |
|---|---|---|---|---|---|
| synchronized | 단일 서버 | 낮음 | 낮음 | 높음 | JVM 레벨 |
| ReentrantLock | 단일 서버 | 낮음 | 중간 | 높음 | tryLock 지원 |
| AtomicInteger | 단일 서버 | 높음 | 낮음 | 높음 | 단순 카운터 |
| DB 비관적 락 | 분산 가능 | 낮음 | 중간 | 높음 | 충돌 多 환경 |
| DB 낙관적 락 | 분산 가능 | 중간 | 중간 | 중간 | 재시도 필요 |
| Redis 분산 락 | 분산 환경 | 높음 | 높음 | 높음 | 인프라 필요 |
| 메시지 큐 | 분산 환경 | 중간 | 높음 | 높음 | 비동기 처리 |

## 어떨 때 어떤 방법을 선택하는가?

```
동시성 문제 발생
        │
        ▼
단일 서버인가?
  ├── Yes → 충돌 빈도가 높은가?
  │           ├── Yes → synchronized / ReentrantLock
  │           └── No  → AtomicInteger / 낙관적 락
  │
  └── No (분산 환경) → 외부 저장소 사용 가능한가?
                  ├── Redis 있음 → Redisson 분산 락 (추천)
                  ├── Redis 없음 → DB 비관적 락
                  └── 비동기 처리 가능 → 메시지 큐
```

### 실무 선택 가이드

| 상황 | 추천 방법 |
|---|---|
| 단일 서버, 간단한 카운터 | AtomicInteger |
| 단일 서버, 복잡한 비즈니스 로직 | synchronized + @Transactional 분리 |
| 분산 서버, Redis 인프라 있음 | Redisson 분산 락 |
| 분산 서버, Redis 없음 | DB 비관적 락 |
| 충돌 빈도 낮음 | DB 낙관적 락 (@Version) |
| 비동기 허용, 높은 처리량 필요 | Kafka/RabbitMQ 메시지 큐 |
| 선착순 이벤트, 쿠폰 발급 | Redis 분산 락 + DB 비관적 락 |

## 특이점 및 주의사항

1. **@Transactional + synchronized 함께 쓸 때 주의**
   - 트랜잭션 커밋 전에 synchronized 블록이 종료되어 다른 스레드가 아직 커밋 안 된 데이터를 읽을 수 있음
   - 해결책: 트랜잭션 범위를 synchronized 메서드를 호출하는 외부 클래스에 두기

2. **Redis 분산 락의 만료 시간 설정 주의**
   - 락 만료 시간이 비즈니스 로직 처리 시간보다 짧으면 만료 후 다른 스레드가 진입
   - Redisson의 Watchdog 기능: 락 보유 중 자동으로 만료 시간 연장

3. **낙관적 락의 재시도 폭발 주의**
   - 동시 요청이 많을수록 재시도가 기하급수적으로 증가
   - Exponential Backoff + Jitter 전략 활용

4. **데드락(Deadlock) 주의**
   - 비관적 락 사용 시 항상 같은 순서로 락 획득
   - 타임아웃 설정 필수
