# 트러블슈팅: 커넥션 풀 고갈

## 커넥션 풀 고갈이란?

DB 커넥션 풀(Connection Pool)의 모든 커넥션이 사용 중이어서 새로운 요청이 커넥션을 얻지 못하고 대기하다 타임아웃되는 현상.

```
정상 상태:
  ┌─────────────────────────────────┐
  │         HikariCP Pool           │
  │  [conn1] [conn2] [conn3] [conn4] [conn5] ...│
  │    사용중  사용중   유휴    유휴    유휴  │
  └─────────────────────────────────┘
  새 요청 → 유휴 커넥션 즉시 획득 → 처리 → 반납

커넥션 풀 고갈 상태:
  ┌─────────────────────────────────┐
  │         HikariCP Pool           │
  │  [conn1] [conn2] [conn3] [conn4] [conn5]    │
  │    사용중  사용중  사용중  사용중  사용중     │
  └─────────────────────────────────┘
  새 요청 → 대기... → 30초 타임아웃 → 에러!
```

## 증상

```
에러 로그:
  HikariPool-1 - Connection is not available,
  request timed out after 30000ms.

  java.sql.SQLTransientConnectionException: HikariPool-1 -
  Connection is not available, request timed out after 30000ms

  Caused by: org.springframework.dao.DataAccessResourceFailureException:
  Unable to acquire JDBC Connection
```

```
운영 증상:
  - API 응답 시간이 30초 근처로 급증
  - 타임아웃 에러 대량 발생
  - DB 연결 수는 최대치인데 쿼리 처리가 안 됨
  - 헬스체크 실패 → 로드밸런서에서 서버 제거
```

## 원인

### 1. 트랜잭션 내에서 오래 걸리는 작업

```java
// 문제: 트랜잭션 안에서 외부 API 호출 (수 초 이상 소요)
@Transactional
public void processOrder(Long orderId) {
    Order order = orderRepository.findById(orderId).orElseThrow();

    // DB 커넥션을 붙잡은 채 외부 API 호출!
    PaymentResult result = paymentClient.charge(order);  // 2~5초 소요

    order.complete(result);
    orderRepository.save(order);
    // 트랜잭션 끝날 때까지 커넥션 반납 안 됨
}

// 10개의 동시 요청 → 모든 커넥션(10개) 점유 → 11번째 요청 타임아웃!
```

### 3. 커넥션 반납 누락

```java
// 문제: JPA를 사용하지 않고 직접 Connection 획득 후 close 안 함
public void badQuery() throws SQLException {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT ...");
    // conn.close() 없음! 커넥션 영구 점유
}

// 해결: try-with-resources 반드시 사용
public void goodQuery() throws SQLException {
    try (Connection conn = dataSource.getConnection();
         Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery("SELECT ...")) {
        // 자동으로 close() 호출
    }
}
```

## 해결 방법

### 1. 트랜잭션 범위 최소화

```java
// 잘못된 패턴: 트랜잭션이 너무 넓음
@Transactional
public OrderResult processOrder(OrderRequest request) {
    // 1. 재고 확인 (DB)
    inventoryService.checkStock(request.getProductId());

    // 2. 외부 결제 API 호출 (2~5초!) ← 커넥션 점유 중!
    PaymentResult payment = paymentClient.charge(request);

    // 3. 주문 저장 (DB)
    Order order = orderRepository.save(new Order(request, payment));

    // 4. 알림 발송 (외부 API) ← 커넥션 점유 중!
    notificationClient.send(order);

    return new OrderResult(order);
}

// 올바른 패턴: 트랜잭션 범위 분리
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final PaymentClient paymentClient;
    private final NotificationClient notificationClient;
    private final InventoryService inventoryService;

    // 트랜잭션 없이 전체 흐름 조율
    public OrderResult processOrder(OrderRequest request) {
        // 1. 재고 확인 (짧은 트랜잭션)
        inventoryService.checkStock(request.getProductId());

        // 2. 외부 결제 API 호출 (트랜잭션 밖!)
        PaymentResult payment = paymentClient.charge(request);

        // 3. 주문 저장 (짧은 트랜잭션)
        Order order = saveOrder(request, payment);

        // 4. 알림 발송 (트랜잭션 밖!)
        notificationClient.send(order);

        return new OrderResult(order);
    }

    @Transactional  // DB 작업만 트랜잭션
    public Order saveOrder(OrderRequest request, PaymentResult payment) {
        return orderRepository.save(new Order(request, payment));
    }
}
```

### 2. 외부 API 호출을 트랜잭션 밖으로

```java
// 잘못된 패턴
@Transactional
public void syncUserData(Long userId) {
    User user = userRepository.findById(userId).orElseThrow();

    // 외부 CRM API 호출 (느릴 수 있음) - 트랜잭션 안!
    CrmUser crmUser = crmClient.getUser(user.getEmail());

    user.updateFromCrm(crmUser);
    userRepository.save(user);
}

// 올바른 패턴
public void syncUserData(Long userId) {
    // 1. 필요한 데이터 조회 (빠른 트랜잭션)
    String email = getUserEmail(userId);

    // 2. 외부 API 호출 (트랜잭션 밖)
    CrmUser crmUser = crmClient.getUser(email);

    // 3. DB 업데이트 (빠른 트랜잭션)
    updateUserFromCrm(userId, crmUser);
}

@Transactional(readOnly = true)
public String getUserEmail(Long userId) {
    return userRepository.findById(userId)
        .map(User::getEmail)
        .orElseThrow();
}

@Transactional
public void updateUserFromCrm(Long userId, CrmUser crmUser) {
    User user = userRepository.findById(userId).orElseThrow();
    user.updateFromCrm(crmUser);
}
```

### 3. 풀 크기 조정 (공식)

```
HikariCP 풀 크기 공식 (PostgreSQL 문서 기준):
  pool size = Tn × (Cm - 1) + 1

  Tn: 스레드 수 (동시 요청 처리 스레드)
  Cm: 쿼리당 최대 동시 커넥션 수

예시:
  - 서버 스레드 200개
  - 요청 하나당 커넥션 1개 사용
  pool size = 200 × (1 - 1) + 1 = 1 → 이론값이지만 실제론 더 필요

실무 공식 (단순화):
  pool size = (서버 코어 수 × 2) + 유효 스핀들 수

권장 접근법:
  1. 처음에는 작게 시작 (10~20)
  2. 모니터링으로 pending 발생 여부 확인
  3. 점진적으로 증가
  4. DB 서버의 max_connections 고려 (여러 앱 서버 합산)
```

```yaml
spring:
  datasource:
    hikari:
      # CPU 4코어 기준 예시
      maximum-pool-size: 20      # (4 × 2) + 유휴 여유
      minimum-idle: 5             # 유휴 커넥션 최소 유지
      connection-timeout: 30000   # 30초 대기 후 예외
      idle-timeout: 600000        # 10분 유휴 시 커넥션 반납
      max-lifetime: 1800000       # 30분 후 커넥션 교체
```

## 코드 예제: 잘못된 패턴 vs 올바른 패턴

### 패턴 1: 파일 처리

```java
// 잘못된 패턴: 파일 처리를 트랜잭션 안에서
@Transactional
public void importProducts(MultipartFile file) {
    List<Product> products = parseCsv(file);  // 파일 파싱 (오래 걸릴 수 있음)
    // 파싱하는 동안 커넥션 점유!

    for (Product product : products) {
        productRepository.save(product);
    }
}

// 올바른 패턴: 파일 처리를 트랜잭션 밖으로
public void importProducts(MultipartFile file) {
    // 1. 트랜잭션 밖: 파일 파싱
    List<Product> products = parseCsv(file);

    // 2. 트랜잭션 안: DB 저장만
    saveProducts(products);
}

@Transactional
public void saveProducts(List<Product> products) {
    productRepository.saveAll(products);
}
```

### 패턴 2: N+1 문제로 인한 커넥션 오래 사용

```java
// 잘못된 패턴: N+1 문제 → 쿼리 N번 = 오래 걸림
@Transactional
public List<OrderResponse> getOrders(Long userId) {
    List<Order> orders = orderRepository.findByUserId(userId);
    return orders.stream()
        .map(order -> {
            // 각 주문마다 아이템 조회 (N번의 추가 쿼리!)
            List<OrderItem> items = orderItemRepository.findByOrderId(order.getId());
            return new OrderResponse(order, items);
        })
        .collect(Collectors.toList());
}

// 올바른 패턴: Fetch Join으로 한 번에 조회
@Transactional(readOnly = true)
public List<OrderResponse> getOrders(Long userId) {
    List<Order> orders = orderRepository.findByUserIdWithItems(userId);
    return orders.stream()
        .map(OrderResponse::new)
        .collect(Collectors.toList());
}

// Repository
@Query("SELECT DISTINCT o FROM Order o LEFT JOIN FETCH o.items WHERE o.userId = :userId")
List<Order> findByUserIdWithItems(@Param("userId") Long userId);
```

## 정리

```
커넥션 풀 고갈 예방 체크리스트:

1. 트랜잭션 범위
   □ @Transactional 범위가 최소한인가?
   □ 외부 API 호출이 트랜잭션 밖에 있는가?
   □ 파일 처리, 대용량 연산이 트랜잭션 밖에 있는가?

2. 쿼리 성능
   □ N+1 문제는 없는가? (EXPLAIN으로 확인)
   □ 슬로우 쿼리는 없는가? (slow query log 확인)
   □ 인덱스가 적절히 설정되어 있는가?

3. 모니터링
   □ hikaricp.connections.pending 메트릭 알람 설정
   □ leak-detection-threshold 설정
   □ Grafana 대시보드로 실시간 모니터링

4. 풀 크기
   □ maximum-pool-size가 트래픽에 맞는가?
   □ DB 서버의 max_connections 초과하지 않는가?
   □ connection-timeout이 적절한가? (5초 권장)

핵심 원칙:
  커넥션 점유 시간 = 트랜잭션 시간
  트랜잭션 시간 최소화 = 커넥션 풀 고갈 방지
```
