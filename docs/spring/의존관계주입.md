# 의존관계 자동 주입 (@Autowired)

---

## @Autowired 기본 동작

`@Autowired`는 스프링 컨테이너에서 **타입이 일치하는 빈**을 찾아 주입한다.

```markdown
@Autowired
private MemberRepository memberRepository;
// 내부적으로 ac.getBean(MemberRepository.class)와 동일하게 동작
```

주입 흐름:

```
@Autowired 발견
    ↓
스프링 컨테이너에서 타입으로 빈 조회
    ↓
빈 1개  → 주입 성공
빈 0개  → NoSuchBeanDefinitionException
          (@Autowired(required=false) 설정 시 주입 생략)
빈 2개↑ → NoUniqueBeanDefinitionException → 추가 처리 필요
```

---

## @Autowired 옵션 처리 (빈이 없을 때)

```markdown
// 방법 1: required = false — 빈이 없으면 메서드 호출 자체가 안 됨
@Autowired(required = false)
public void setMailSender(MailSender mailSender) {
    this.mailSender = mailSender;
}

// 방법 2: @Nullable — 빈이 없으면 null 주입
@Autowired
public void setMailSender(@Nullable MailSender mailSender) {
    this.mailSender = mailSender; // null 허용
}

// 방법 3: Optional — 빈이 없으면 Optional.empty()
@Autowired
public void setMailSender(Optional<MailSender> mailSender) {
    this.mailSender = mailSender.orElse(null);
}
```

| 방법 | 빈 없을 때 | 빈 있을 때 |
|---|---|---|
| `required=false` | 메서드 자체 호출 안 함 | 정상 주입 |
| `@Nullable` | `null` 주입 | 정상 주입 |
| `Optional<T>` | `Optional.empty()` | `Optional.of(빈)` |

---

## 빈이 2개 이상일 때 문제

같은 인터페이스를 구현한 빈이 두 개 등록되면 주입 시 예외가 발생한다.

```markdown
public interface DiscountPolicy { int discount(Member member, int price); }

@Component
public class FixDiscountPolicy implements DiscountPolicy { ... }

@Component
public class RateDiscountPolicy implements DiscountPolicy { ... }
```

```markdown
@Autowired
private DiscountPolicy discountPolicy;
// → NoUniqueBeanDefinitionException:
//   expected single matching bean but found 2: fixDiscountPolicy, rateDiscountPolicy
```

해결 방법이 세 가지 있다.

---

## 해결 방법 1: 필드명/파라미터명 매칭

`@Autowired`는 타입 매칭 실패 시 **필드명 또는 파라미터명으로 빈 이름을 추가 매칭**한다.

```markdown
// 필드명을 빈 이름과 일치시키면 rateDiscountPolicy 빈이 주입됨
@Autowired
private DiscountPolicy rateDiscountPolicy; // 필드명 = 빈 이름

// 생성자 파라미터명으로도 가능
@Autowired
public OrderServiceImpl(DiscountPolicy rateDiscountPolicy) { // 파라미터명 = 빈 이름
    this.discountPolicy = rateDiscountPolicy;
}
```

단점: 필드명이나 파라미터명이 특정 구현체 이름에 묶이게 되어 리팩터링할 때 놓치기 쉽다.

## 해결 방법 2: @Qualifier (추가 구별자)

빈에 추가 이름표를 붙여서 주입 시 구분한다.

```markdown
// 빈에 @Qualifier 붙이기
@Component
@Qualifier("mainDiscountPolicy")
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Component
@Qualifier("fixDiscountPolicy")
public class FixDiscountPolicy implements DiscountPolicy { ... }
```

```markdown
// 주입할 때 @Qualifier로 지정
@Autowired
public OrderServiceImpl(MemberRepository memberRepository,
                        @Qualifier("mainDiscountPolicy") DiscountPolicy discountPolicy) {
    this.discountPolicy = discountPolicy;
}
```

`@Qualifier` 매칭 순서:
```
1. @Qualifier("mainDiscountPolicy") 이름으로 빈 탐색
2. 없으면 "mainDiscountPolicy"라는 이름의 빈 탐색
3. 없으면 NoSuchBeanDefinitionException
```

### 커스텀 어노테이션으로 개선 (타입 안전)

`@Qualifier("mainDiscountPolicy")`는 문자열이라 오타가 생겨도 컴파일 에러가 나지 않는다.
커스텀 어노테이션으로 만들면 오타를 컴파일 타임에 잡을 수 있다.

```markdown
// 커스텀 어노테이션 정의
@Target({ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER,
         ElementType.TYPE, ElementType.ANNOTATION_TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Qualifier("mainDiscountPolicy") // 내부에 @Qualifier 포함
public @interface MainDiscountPolicy {}
```

```markdown
// 적용
@Component
@MainDiscountPolicy // 오타 시 컴파일 에러 → 안전
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Autowired
public OrderServiceImpl(MemberRepository memberRepository,
                        @MainDiscountPolicy DiscountPolicy discountPolicy) {
    this.discountPolicy = discountPolicy;
}
```

## 해결 방법 3: @Primary (우선순위)

여러 빈 중 하나에 `@Primary`를 붙이면 타입 매칭 시 우선으로 선택된다.

```markdown
@Component
@Primary // 타입으로 주입 시 이 빈이 선택됨
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Component
public class FixDiscountPolicy implements DiscountPolicy { ... }
```

```markdown
// @Primary가 있으면 타입 그대로 주입 가능
@Autowired
private DiscountPolicy discountPolicy; // RateDiscountPolicy 주입됨
```

### @Primary vs @Qualifier 우선순위

```
@Qualifier > @Primary

@Qualifier를 명시하면 @Primary보다 우선 적용된다.
```

### 실무 활용 패턴

```markdown
// 주 DB — @Primary (평소에 항상 쓰는 것)
@Bean
@Primary
public DataSource mainDataSource() {
    return new HikariDataSource(); // 주 DB
}

// 서브 DB — @Qualifier (특별히 필요할 때만)
@Bean
@Qualifier("subDataSource")
public DataSource subDataSource() {
    return new HikariDataSource(); // 통계용 서브 DB
}
```

## 모든 구현체가 필요할 때: List, Map

전략 패턴처럼 런타임에 구현체를 선택해야 할 때 사용한다.

```markdown
@Service
public class DiscountService {

    // DiscountPolicy를 구현한 빈 전부를 Map으로 받음
    // key: 빈 이름, value: 구현체
    private final Map<String, DiscountPolicy> policyMap;
    private final List<DiscountPolicy> policies;

    @Autowired
    public DiscountService(Map<String, DiscountPolicy> policyMap,
                           List<DiscountPolicy> policies) {
        this.policyMap = policyMap;
        this.policies = policies;
        // policyMap = {"fixDiscountPolicy": Fix..., "rateDiscountPolicy": Rate...}
        // policies  = [FixDiscountPolicy, RateDiscountPolicy]
    }

    // 클라이언트가 선택한 정책 이름으로 동적으로 처리
    public int discount(Member member, int price, String discountCode) {
        DiscountPolicy discountPolicy = policyMap.get(discountCode);
        return discountPolicy.discount(member, price);
    }
}
```

```markdown
// 사용
discountService.discount(member, 10000, "fixDiscountPolicy");  // 정액 할인
discountService.discount(member, 10000, "rateDiscountPolicy"); // 정률 할인
```

## @Autowired 매칭 전체 흐름

```
1. 타입 매칭 시도
        ↓
2. 타입 매칭 결과가 2개 이상
        ↓
3. @Qualifier 확인 → 있으면 매칭
        ↓ (없으면)
4. @Primary 확인 → 있으면 매칭
        ↓ (없으면)
5. 필드명/파라미터명으로 빈 이름 매칭
        ↓ (없으면)
6. NoUniqueBeanDefinitionException 발생
```

## 자동 vs 수동 등록 선택 기준

| 상황 | 권장 방식 |
|---|---|
| 일반 비즈니스 로직 | 자동 (`@ComponentScan`) |
| 기술 지원 빈 (AOP, DataSource 등) | 수동 (`@Bean`) — 명확히 드러내야 함 |
| 다형성 활용 (구현체 여러 개) | 수동 or `@Qualifier` — 어떤 빈인지 한눈에 보여야 함 |
| 외부 라이브러리 빈 | 수동 (`@Bean`) — 어노테이션 추가 불가 |

```markdown
// 다형성 빈 관리 — 수동 등록이 명확함
@Configuration
public class DiscountPolicyConfig {
    // 어떤 할인 정책 빈이 있는지 한눈에 파악 가능
    @Bean
    public DiscountPolicy rateDiscountPolicy() { return new RateDiscountPolicy(); }

    @Bean
    public DiscountPolicy fixDiscountPolicy() { return new FixDiscountPolicy(); }
}
```
