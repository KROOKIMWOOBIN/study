# 스프링 컨테이너

---

## BeanFactory vs ApplicationContext

### BeanFactory

스프링 컨테이너의 최상위 인터페이스. 빈을 관리하고 조회하는 기능을 제공한다.

```java
public interface BeanFactory {
    Object getBean(String name);
    <T> T getBean(String name, Class<T> requiredType);
    boolean containsBean(String name);
    boolean isSingleton(String name);
    // ...
}
```

- `getBean()` 등 기본 기능만 제공
- 빈을 **지연 로딩(Lazy Loading)**: `getBean()` 호출 시점에 빈 생성

### ApplicationContext

`BeanFactory`를 상속하고 부가 기능을 추가한 인터페이스. **실제로 주로 사용하는 것**.

```
ApplicationContext 상속 계층:
ApplicationContext
├── BeanFactory                   (빈 조회, 관리)
├── EnvironmentCapable            (환경변수 - local/dev/prod 구분)
├── MessageSource                 (국제화 - i18n, messages.properties)
├── ApplicationEventPublisher     (이벤트 발행/구독)
└── ResourceLoader                (파일/클래스패스/URL 리소스 로딩)
```

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

// BeanFactory 기능
OrderService orderService = ac.getBean("orderService", OrderService.class);

// 부가 기능들
String profile = ac.getEnvironment().getActiveProfiles()[0]; // 환경변수
String message = ac.getMessage("hello", null, Locale.KOREA);  // 국제화
```

### 비교

| 구분 | BeanFactory | ApplicationContext |
|---|---|---|
| 빈 관리 | O | O |
| 국제화 | X | O |
| 환경변수 | X | O |
| 이벤트 | X | O |
| 리소스 | X | O |
| 빈 로딩 | 지연(Lazy) | 즉시(Eager) — 기본 |
| **사용 여부** | 직접 잘 안 씀 | **항상 이것 사용** |

## 스프링 컨테이너 생성 과정

### 1단계: 스프링 컨테이너 생성

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
```

```
[스프링 컨테이너]
    빈 저장소 (비어있음)
    빈 이름  →  빈 객체
    ─────────────────────
    (아직 없음)
```

### 2단계: 스프링 빈 등록

`AppConfig.class`를 읽어 `@Bean`이 붙은 메서드를 모두 호출하여 빈 등록.

```java
@Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() { return new MemberServiceImpl(memberRepository()); }
    @Bean
    public MemberRepository memberRepository() { return new MemoryMemberRepository(); }
    @Bean
    public OrderService orderService() { return new OrderServiceImpl(memberRepository(), discountPolicy()); }
    @Bean
    public DiscountPolicy discountPolicy() { return new RateDiscountPolicy(); }
}
```

```
[스프링 컨테이너]
    빈 이름          →  빈 객체
    ─────────────────────────────────────
    memberService    →  MemberServiceImpl@x01
    memberRepository →  MemoryMemberRepository@x02
    orderService     →  OrderServiceImpl@x03
    discountPolicy   →  RateDiscountPolicy@x04
```

**빈 이름**: 기본적으로 메서드 이름. 직접 지정 가능: `@Bean(name="myService")`
**주의**: 빈 이름이 겹치면 안 됨 (설정에 따라 오버라이딩 또는 오류)

### 3단계: 스프링 빈 의존관계 주입

설정 정보를 참고해서 의존관계를 주입(DI).

```
[빈 객체들 연결]
MemberServiceImpl@x01
    └── memberRepository → MemoryMemberRepository@x02

OrderServiceImpl@x03
    ├── memberRepository → MemoryMemberRepository@x02
    └── discountPolicy   → RateDiscountPolicy@x04
```

> 생성자 주입의 경우 빈 생성과 의존관계 주입이 동시에 발생.

## 다양한 설정 형식 지원

### XML 방식 (레거시)

```xml
<!-- applicationContext.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="...">

    <bean id="memberService" class="hello.core.member.MemberServiceImpl">
        <constructor-arg name="memberRepository" ref="memberRepository"/>
    </bean>

    <bean id="memberRepository" class="hello.core.member.MemoryMemberRepository"/>

    <bean id="orderService" class="hello.core.order.OrderServiceImpl">
        <constructor-arg name="memberRepository" ref="memberRepository"/>
        <constructor-arg name="discountPolicy" ref="discountPolicy"/>
    </bean>

    <bean id="discountPolicy" class="hello.core.discount.RateDiscountPolicy"/>
</beans>
```

```java
// XML 기반 컨테이너
ApplicationContext ac = new ClassPathXmlApplicationContext("applicationContext.xml");
```

### Java Config 방식 (현재 주류)

```java
@Configuration
public class AppConfig {
    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }
    // ...
}

ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
```

### Component Scan 방식 (Spring Boot 기본)

```java
@SpringBootApplication // @ComponentScan 포함
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

## BeanDefinition 추상화

### 왜 다양한 설정 형식을 지원하는가?

스프링이 XML도, Java Config도 지원하는 이유는 **BeanDefinition** 추상화 덕분이다.

```
XML 파일
    → XmlBeanDefinitionReader
    → BeanDefinition (빈 메타정보)
    → 스프링 컨테이너

Java Config
    → AnnotatedBeanDefinitionReader
    → BeanDefinition (빈 메타정보)
    → 스프링 컨테이너
```

스프링 컨테이너는 `BeanDefinition`만 알면 된다. 설정 형식이 뭐든 상관없다.

### BeanDefinition 메타정보

```java
AnnotationConfigApplicationContext ac =
    new AnnotationConfigApplicationContext(AppConfig.class);

String[] beanDefinitionNames = ac.getBeanDefinitionNames();
for (String beanDefinitionName : beanDefinitionNames) {
    BeanDefinition beanDefinition = ac.getBeanDefinition(beanDefinitionName);

    if (beanDefinition.getRole() == BeanDefinition.ROLE_APPLICATION) {
        // 개발자가 직접 등록한 빈만 출력
        System.out.println("beanDefinitionName=" + beanDefinitionName
                + " beanDefinition=" + beanDefinition);
    }
}
```

| BeanDefinition 속성 | 설명 |
|---|---|
| `beanClassName` | 빈 클래스 이름 |
| `scope` | 스코프 (기본: 싱글톤) |
| `lazyInit` | 지연 로딩 여부 |
| `initMethodName` | 초기화 메서드 이름 |
| `destroyMethodName` | 소멸 메서드 이름 |
| `constructorArgumentValues` | 생성자 파라미터 정보 |
| `propertyValues` | setter 주입 정보 |

## 빈 조회

### 기본 조회

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

// 이름으로 조회
MemberService memberService = ac.getBean("memberService", MemberService.class);

// 타입으로 조회 (같은 타입이 여러 개면 예외 발생!)
MemberService memberService = ac.getBean(MemberService.class);

// 구체 타입으로 조회 (비권장 - 구현에 의존)
MemberServiceImpl impl = ac.getBean("memberService", MemberServiceImpl.class);
```

### 동일 타입 빈이 여러 개인 경우

```java
// 타입으로 조회 시 예외: NoUniqueBeanDefinitionException
ac.getBean(MemberRepository.class); // 여러 개면 오류!

// 해결 1: 이름 지정
ac.getBean("memberRepository", MemberRepository.class);

// 해결 2: 타입의 모든 빈 조회
Map<String, MemberRepository> beansOfType = ac.getBeansOfType(MemberRepository.class);
```

### 상속 관계 조회

```java
// 부모 타입으로 조회하면 자식 타입도 함께 조회
ac.getBean(DiscountPolicy.class);
// FixDiscountPolicy, RateDiscountPolicy 모두 조회됨 → NoUniqueBeanDefinitionException!

// 이름 지정 필요
ac.getBean("rateDiscountPolicy", DiscountPolicy.class);

// 모든 자식 조회
Map<String, DiscountPolicy> beansOfType = ac.getBeansOfType(DiscountPolicy.class);
```

## 컨테이너에 빈 등록하는 방법 비교

| 방법 | 특징 | 사용 시기 |
|---|---|---|
| `@Bean` (Java Config) | 명시적 등록, 외부 라이브러리 빈 등록 가능 | 수동 등록이 필요한 빈 |
| `@Component` + `@ComponentScan` | 자동 등록, 간편 | 대부분의 빈 |
| XML | 레거시, 컴파일 없이 변경 가능 | 레거시 시스템 |
| `@Import` | 다른 설정 클래스 포함 | 설정 모듈화 |

```java
// @Import 예시
@Configuration
@Import({AppConfig.class, DbConfig.class})
public class RootConfig {
}
```
