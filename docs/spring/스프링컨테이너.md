# 스프링 컨테이너

---

## 스프링 컨테이너란?

스프링 컨테이너는 **빈(Bean)의 생성·관리·의존관계 주입을 담당하는 핵심 관리자**다.

개발자가 `AppConfig`에 설정 정보를 작성하면, 스프링 컨테이너가 이를 읽어서 빈을 등록하고 DI를 처리한다.

```markdown
// 스프링 컨테이너 생성 — AppConfig 설정 정보를 읽어서 빈을 등록
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

// 등록된 빈 꺼내 쓰기
OrderService orderService = ac.getBean("orderService", OrderService.class);
```

`ApplicationContext`가 스프링 컨테이너의 인터페이스다.

---

## BeanFactory vs ApplicationContext

### BeanFactory

스프링 컨테이너의 최상위 인터페이스. 빈을 관리하고 조회하는 기본 기능을 제공한다.

- 빈을 **지연 로딩(Lazy Loading)**: `getBean()` 호출 시점에 생성
- 실무에서 직접 쓸 일은 거의 없다

### ApplicationContext

`BeanFactory`를 상속하고 실무에 필요한 부가 기능을 추가한 인터페이스. **항상 이것을 사용한다.**

```
ApplicationContext 상속 계층:
├── BeanFactory              (빈 조회, 관리)
├── EnvironmentCapable       (환경변수 — local/dev/prod 구분)
├── MessageSource            (국제화 — messages.properties)
├── ApplicationEventPublisher (이벤트 발행/구독)
└── ResourceLoader           (파일/클래스패스/URL 리소스 로딩)
```

| 기능 | BeanFactory | ApplicationContext |
|---|---|---|
| 빈 관리 | O | O |
| 환경변수 (프로파일) | X | O |
| 국제화 | X | O |
| 이벤트 | X | O |
| 리소스 로딩 | X | O |
| 빈 로딩 방식 | 지연(Lazy) | 즉시(Eager) — 기본 |

## 스프링 컨테이너 생성 과정

### 1단계: 컨테이너 생성

```markdown
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
```

```
[스프링 컨테이너]
    빈 저장소 (아직 비어있음)
    빈 이름  →  빈 객체
    ─────────────────────
    (없음)
```

### 2단계: 빈 등록

`AppConfig`를 읽어서 `@Bean`이 붙은 메서드를 모두 호출하고, 반환된 객체를 빈으로 등록한다.

```markdown
@Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() { return new MemberServiceImpl(memberRepository()); }

    @Bean
    public MemberRepository memberRepository() { return new MemoryMemberRepository(); }

    @Bean
    public OrderService orderService() { return new OrderServiceImpl(memberRepository(), discountPolicy()); }

    @Bean
    public DiscountPolicy discountPolicy() { return new RateDiscountPolicy(); }
}
```

```
[스프링 컨테이너]
    빈 이름           →  빈 객체
    ─────────────────────────────────────────
    memberService     →  MemberServiceImpl@x01
    memberRepository  →  MemoryMemberRepository@x02
    orderService      →  OrderServiceImpl@x03
    discountPolicy    →  RateDiscountPolicy@x04
```

빈 이름은 기본적으로 **메서드 이름**이다.

```markdown
@Bean(name = "myOrderService") // 이름 직접 지정도 가능
public OrderService orderService() { ... }
```

주의: 빈 이름이 겹치면 예외가 발생하거나 오버라이딩된다. 항상 다른 이름을 써야 한다.

### 3단계: 의존관계 주입

설정 정보를 참고하여 빈들의 의존관계를 연결한다.

```
[빈 객체 연결 결과]
MemberServiceImpl@x01
    └── memberRepository → MemoryMemberRepository@x02

OrderServiceImpl@x03
    ├── memberRepository → MemoryMemberRepository@x02
    └── discountPolicy   → RateDiscountPolicy@x04
```

생성자 주입의 경우 빈 생성과 DI가 동시에 일어난다.

## 빈 설정 방법 3가지

### Java Config 방식 (현재 주류)

```markdown
@Configuration
public class AppConfig {
    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }
    // ...
}

ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
```

### Component Scan 방식 (Spring Boot 기본)

```markdown
@SpringBootApplication // @ComponentScan이 포함되어 있음
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

`@Component`, `@Service`, `@Repository`, `@Controller`가 붙은 클래스를 자동으로 빈 등록.

### XML 방식 (레거시)

```markdown
<!-- applicationContext.xml -->
<beans>
    <bean id="memberRepository" class="hello.core.member.MemoryMemberRepository"/>
    <bean id="orderService" class="hello.core.order.OrderServiceImpl">
        <constructor-arg name="memberRepository" ref="memberRepository"/>
    </bean>
</beans>
```

오래된 프로젝트에서 볼 수 있지만, 신규 프로젝트에선 거의 쓰지 않는다.

## 빈 조회

### 기본 조회

```markdown
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

// 이름 + 타입으로 조회 (가장 명확)
MemberService memberService = ac.getBean("memberService", MemberService.class);

// 타입만으로 조회 (같은 타입이 2개 이상이면 예외 발생!)
MemberService memberService = ac.getBean(MemberService.class);
```

### 같은 타입 빈이 여러 개인 경우

```markdown
// MemberRepository 구현체가 두 개 등록되어 있다면
ac.getBean(MemberRepository.class);
// → NoUniqueBeanDefinitionException: expected single matching bean but found 2

// 해결 1: 이름 지정
ac.getBean("memoryMemberRepository", MemberRepository.class);

// 해결 2: 해당 타입의 모든 빈 조회
Map<String, MemberRepository> beansOfType = ac.getBeansOfType(MemberRepository.class);
beansOfType.forEach((name, bean) -> System.out.println("name=" + name + " bean=" + bean));
```

### 상속 관계 조회 주의

부모 타입으로 조회하면 자식 타입 빈도 모두 조회된다.

```markdown
// DiscountPolicy를 구현한 빈이 Fix, Rate 두 개 있을 때
ac.getBean(DiscountPolicy.class);
// → NoUniqueBeanDefinitionException!

// 이름 지정 필요
ac.getBean("rateDiscountPolicy", DiscountPolicy.class);
```

Object로 조회하면 스프링이 등록한 모든 빈이 나온다. (내부 스프링 빈 포함)

## 빈 등록 방법 비교

| 방법 | 특징 | 사용 시기 |
|---|---|---|
| `@Bean` (Java Config) | 명시적 등록 | 외부 라이브러리, 공통 기술 빈 |
| `@Component` + `@ComponentScan` | 자동 등록 | 일반 비즈니스 빈 |
| XML | 레거시 | 오래된 프로젝트 유지보수 |
