# 컴포넌트 스캔 (@ComponentScan)

---

## 왜 필요한가

`@Bean`으로 빈을 등록하면 클래스마다 직접 작성해야 한다.

```markdown
@Configuration
public class AppConfig {
    @Bean public MemberService memberService() { ... }
    @Bean public MemberRepository memberRepository() { ... }
    @Bean public OrderService orderService() { ... }
    @Bean public DiscountPolicy discountPolicy() { ... }
    // 빈이 수십, 수백 개가 되면 일일이 등록해야 함
    // 실수로 누락할 수도 있음
}
```

`@ComponentScan`은 이 문제를 해결한다.
클래스패스를 탐색해서 `@Component`가 붙은 클래스를 **자동으로 빈 등록**한다.

---

## 동작 방식

```
1. @ComponentScan 선언 발견
        ↓
2. basePackages 위치부터 하위 패키지 전체 탐색
        ↓
3. @Component가 붙은 클래스 발견
        ↓
4. 스프링 빈으로 자동 등록
   (빈 이름: 클래스명 첫 글자 소문자, 예: MemberServiceImpl → memberServiceImpl)
        ↓
5. @Autowired 의존관계 자동 주입
```

```markdown
// 클래스에 @Component를 붙이면 자동 등록됨
@Component
public class MemberServiceImpl implements MemberService {
    private final MemberRepository memberRepository;

    @Autowired // 스프링이 타입으로 빈을 찾아서 주입
    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}
```

빈 이름 직접 지정도 가능하다.

```markdown
@Component("myMemberService") // 빈 이름: myMemberService
public class MemberServiceImpl implements MemberService { ... }
```

## @Component 계열 어노테이션

실무에서는 `@Component` 직접보다 역할을 표현하는 특화 어노테이션을 쓴다.
내부적으로 모두 `@Component`를 포함한다.

| 어노테이션 | 사용 계층 | 추가 기능 |
|---|---|---|
| `@Component` | 범용 | 없음 |
| `@Controller` | 웹 컨트롤러 | Spring MVC에서 컨트롤러로 인식 |
| `@Service` | 비즈니스 로직 | 없음 (역할 명시 목적) |
| `@Repository` | 데이터 접근 | DB 예외를 `DataAccessException`으로 변환 |
| `@Configuration` | 설정 | CGLIB 싱글톤 처리 |

```markdown
// @Service 내부를 보면 @Component가 들어있다
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Component // ← 포함!
public @interface Service { ... }
```

### 계층별 어노테이션 사용 예

```markdown
// 웹 계층
@Controller
public class MemberController {
    private final MemberService memberService;

    @Autowired
    public MemberController(MemberService memberService) {
        this.memberService = memberService;
    }
}

// 서비스 계층
@Service
public class MemberServiceImpl implements MemberService {
    private final MemberRepository memberRepository;

    @Autowired
    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}

// 레포지토리 계층
@Repository
public class MemoryMemberRepository implements MemberRepository {
    // ...
}
```

## 스캔 범위 설정

### 기본값 (권장)

`@ComponentScan`이 선언된 클래스의 패키지부터 하위 전체를 탐색한다.

```markdown
package hello.core;

@ComponentScan // hello.core 패키지부터 탐색
public class AppConfig { ... }
```

### 직접 지정

```markdown
@ComponentScan(basePackages = "hello.core")
// hello.core와 그 하위 패키지 탐색

@ComponentScan(basePackages = {"hello.core", "hello.service"})
// 여러 패키지 지정
```

### 권장 프로젝트 구조

설정 클래스를 **프로젝트 최상위 패키지**에 두면 하위 전체가 자동 탐색된다.

```
hello.core                  ← AppConfig, Application 위치 (최상위)
├── AppConfig.java           ← @ComponentScan
├── member/
│   ├── MemberController.java
│   ├── MemberService.java
│   └── MemoryMemberRepository.java
└── order/
    ├── OrderController.java
    └── OrderService.java
```

### Spring Boot의 경우

`@SpringBootApplication`이 `@ComponentScan`을 포함한다.
별도 설정 없이 `main` 클래스가 위치한 패키지부터 전체를 스캔한다.

```markdown
@SpringBootApplication // @ComponentScan + @Configuration + @EnableAutoConfiguration
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

## 필터 설정

### includeFilters (스캔 대상 추가)

```markdown
// 커스텀 어노테이션
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface MyIncludeComponent {}

// @MyIncludeComponent가 붙은 클래스도 자동 등록
@ComponentScan(
    includeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION, classes = MyIncludeComponent.class
    )
)
```

### excludeFilters (스캔 대상 제외)

```markdown
// 특정 어노테이션이 붙은 클래스 제외
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION, classes = Configuration.class
    )
)

// 특정 클래스 제외
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ASSIGNABLE_TYPE, classes = SomeBean.class
    )
)
```

## 중복 등록 문제

### 자동 vs 자동 (항상 오류)

같은 이름의 빈이 두 곳에서 자동 등록되면 `ConflictingBeanDefinitionException`이 발생한다.

```markdown
@Component
public class MemoryMemberRepository { ... } // 빈 이름: memoryMemberRepository

@Component("memoryMemberRepository") // 같은 이름 충돌!
public class OtherRepository { ... }
// → ConflictingBeanDefinitionException
```

### 수동 vs 자동 (Spring Boot에서는 오류)

```markdown
// 자동 등록
@Component
public class MemoryMemberRepository implements MemberRepository { ... }

// 수동 등록 (같은 이름)
@Configuration
public class AppConfig {
    @Bean(name = "memoryMemberRepository")
    public MemberRepository memberRepository() { return new MemoryMemberRepository(); }
}
// → 수동이 자동을 오버라이딩 (과거 Spring 기본 동작)
// → Spring Boot 기본 설정: 오류 발생!
```

Spring Boot에서 허용하려면:

```markdown
// application.yml
spring:
  main:
    allow-bean-definition-overriding: true
```

> **권장**: 수동/자동 빈 이름이 겹치지 않도록 명확하게 구분한다.

## 언제 자동 등록, 언제 수동 등록?

| 상황 | 권장 방식 |
|---|---|
| 비즈니스 로직 빈 (서비스, 레포지토리) | 자동 (`@Service`, `@Repository`) |
| 외부 라이브러리 빈 등록 | 수동 (`@Bean`) — 내 코드가 아니라 어노테이션 추가 불가 |
| AOP, 트랜잭션 등 공통 기술 빈 | 수동 (`@Bean`) — 명확히 드러내는 게 좋음 |
| DataSource, 외부 API 클라이언트 | 수동 (`@Bean`) |

```markdown
// 실무 패턴: 비즈니스 빈은 자동, 인프라 빈은 수동
@Configuration
public class InfraConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(); // 외부 라이브러리 → 수동 등록
    }
}

// 비즈니스 로직은 컴포넌트 스캔
@Service
public class MemberService { ... }

@Repository
public class MemberRepository { ... }
```
