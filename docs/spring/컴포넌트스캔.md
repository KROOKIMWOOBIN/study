# 컴포넌트 스캔 (@ComponentScan)

---

## @ComponentScan이란? 왜 필요한가

### 문제: @Bean 수동 등록의 한계

```java
@Configuration
public class AppConfig {
    @Bean public MemberService memberService() { ... }
    @Bean public MemberRepository memberRepository() { ... }
    @Bean public OrderService orderService() { ... }
    @Bean public DiscountPolicy discountPolicy() { ... }
    // 빈이 수십, 수백 개면 일일이 등록해야 함 → 귀찮고 누락 가능성
}
```

### 해결: @ComponentScan

스프링이 클래스패스를 탐색하여 `@Component` 어노테이션이 붙은 클래스를 **자동으로 빈 등록**한다.

```java
@Configuration
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(type = FilterType.ANNOTATION,
                                           classes = Configuration.class)
)
public class AutoAppConfig {
    // @Bean이 없어도 됨!
}
```

---

## @ComponentScan 동작 방식

### 탐색 과정

```
1. @ComponentScan 선언된 설정 클래스 발견
    ↓
2. basePackages 위치부터 하위 패키지 모두 탐색
    ↓
3. @Component가 붙은 클래스 발견
    ↓
4. 스프링 빈으로 등록
    (기본 빈 이름: 클래스명의 첫 글자를 소문자로)
    ↓
5. @Autowired 의존관계 자동 주입
```

### 빈 이름 규칙

```java
@Component
public class MemberServiceImpl implements MemberService { ... }
// 빈 이름: memberServiceImpl (첫 글자 소문자)

@Component("myMemberService") // 직접 지정
public class MemberServiceImpl implements MemberService { ... }
// 빈 이름: myMemberService
```

## @Component 계열 어노테이션

| 어노테이션 | 설명 | 추가 기능 |
|---|---|---|
| `@Component` | 기본 컴포넌트 | 없음 |
| `@Controller` | 스프링 MVC 컨트롤러 | 스프링 MVC에서 컨트롤러로 인식 |
| `@Service` | 비즈니스 로직 | 특별한 처리 없음 (의미론적 역할) |
| `@Repository` | 데이터 접근 계층 | 데이터 예외를 `DataAccessException`으로 변환 |
| `@Configuration` | 설정 정보 | 스프링 빈으로 등록 + CGLIB 싱글톤 처리 |

### 내부 구조 (모두 @Component를 포함)

```java
// @Service 내부
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Component // ← @Component가 포함되어 있음!
public @interface Service {
    @AliasFor(annotation = Component.class)
    String value() default "";
}
```

### 계층별 어노테이션 사용

```
[웹 계층]           @Controller
[서비스 계층]        @Service
[레포지토리 계층]    @Repository
[설정]              @Configuration
```

```java
// 컨트롤러
@Controller
public class MemberController {
    private final MemberService memberService;

    @Autowired
    public MemberController(MemberService memberService) {
        this.memberService = memberService;
    }
}

// 서비스
@Service
public class MemberServiceImpl implements MemberService {
    private final MemberRepository memberRepository;

    @Autowired
    public MemberServiceImpl(MemberRepository memberRepository) {
        this.memberRepository = memberRepository;
    }
}

// 레포지토리
@Repository
public class MemoryMemberRepository implements MemberRepository {
    // ...
}
```

## 스캔 범위 설정

### basePackages

```java
@ComponentScan(basePackages = "hello.core")
// hello.core 패키지와 그 하위 패키지 탐색

@ComponentScan(basePackages = {"hello.core", "hello.service"})
// 여러 패키지 지정 가능
```

### basePackageClasses

```java
@ComponentScan(basePackageClasses = AppConfig.class)
// AppConfig 클래스가 위치한 패키지부터 탐색 (더 안전한 방법)
```

### 기본값 (권장 방법)

```java
// 지정하지 않으면 @ComponentScan이 선언된 클래스의 패키지부터 탐색
@ComponentScan
public class AppConfig { ... }
// AppConfig가 hello.core 패키지에 있으면 hello.core부터 탐색
```

```
[권장 프로젝트 구조]
hello.core              ← @SpringBootApplication (최상위 패키지)
├── AppConfig.java        ← @ComponentScan (설정 클래스)
├── member/
│   ├── MemberController.java
│   ├── MemberService.java
│   └── MemoryMemberRepository.java
└── order/
    ├── OrderController.java
    └── OrderService.java
```

**설정 클래스(AppConfig)를 프로젝트 최상위에 두면** 하위 패키지 전체가 자동 탐색됨.

### Spring Boot의 경우

```java
@SpringBootApplication // = @ComponentScan + @Configuration + @EnableAutoConfiguration
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

`@SpringBootApplication`이 위치한 패키지부터 하위 전체를 스캔하므로, 별도 설정 불필요.

## 필터 설정

### includeFilters (스캔에 추가)

```java
// 커스텀 어노테이션 만들기
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface MyIncludeComponent {
}

// 설정
@ComponentScan(
    includeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION,
        classes = MyIncludeComponent.class
    )
)
```

### excludeFilters (스캔에서 제외)

```java
// 특정 어노테이션 제외
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ANNOTATION,
        classes = Configuration.class // @Configuration 붙은 클래스 제외
    )
)

// 특정 클래스 제외
@ComponentScan(
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ASSIGNABLE_TYPE,
        classes = BeanA.class
    )
)
```

### FilterType 종류

| FilterType | 설명 | 예시 |
|---|---|---|
| `ANNOTATION` | 어노테이션 기준 | `@MyAnnotation` |
| `ASSIGNABLE_TYPE` | 클래스/인터페이스 기준 | `BeanA.class` |
| `ASPECTJ` | AspectJ 패턴 | `"hello..*Service+"` |
| `REGEX` | 정규식 | `"hello\\.Default.*"` |
| `CUSTOM` | 커스텀 TypeFilter 구현체 | `MyTypeFilter.class` |

## 중복 등록 문제

### 자동 빈 등록 vs 자동 빈 등록

같은 이름의 빈이 2개 자동 등록되면 `ConflictingBeanDefinitionException` 발생.

```java
// 같은 빈 이름이 두 곳에서 등록되면 오류
@Component
public class MemoryMemberRepository implements MemberRepository { ... }

@Component("memoryMemberRepository") // 이름 충돌!
public class SomeOtherRepository { ... }
```

### 수동 빈 등록 vs 자동 빈 등록

```java
// 자동 등록 (컴포넌트 스캔)
@Component
public class MemoryMemberRepository implements MemberRepository { ... }

// 수동 등록 (@Bean)
@Configuration
public class AppConfig {
    @Bean(name = "memoryMemberRepository") // 같은 이름!
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository();
    }
}
```

이 경우 **수동 등록 빈이 자동 등록 빈을 오버라이딩**한다.

- 스프링 부트 기본 설정: **오류 발생** (의도치 않은 오버라이딩 방지)
  ```
  Consider renaming one of the beans or enabling overriding by setting
  spring.main.allow-bean-definition-overriding=true
  ```

- 허용하려면:
  ```yaml
  spring:
    main:
      allow-bean-definition-overriding: true
  ```

### 권장사항

> **실무에서는 수동/자동 빈이 겹치지 않도록** 명확하게 구분하는 것이 좋다.
> 오버라이딩은 의도하지 않은 버그를 유발할 수 있다.

## @ComponentScan 어떨 때 사용하는가

| 상황 | 권장 방식 |
|---|---|
| 일반 애플리케이션 빈 (서비스, 레포지토리 등) | `@ComponentScan` + `@Component` 계열 |
| 외부 라이브러리 빈 등록 | `@Bean` 수동 등록 (내 코드가 아니므로 어노테이션 불가) |
| 비즈니스 로직 빈 | 자동 등록 (`@Service`, `@Repository` 등) |
| 공통 기술 지원 빈 (AOP, 트랜잭션 등) | 수동 등록 (명확하게 드러내는 것이 좋음) |
| DataSource, 외부 API 클라이언트 등 | 수동 등록 |

```java
// 실무 패턴: 비즈니스 로직은 자동, 인프라/기술 설정은 수동
@Configuration
public class InfraConfig {
    // DataSource, TransactionManager 등 인프라 빈은 수동 등록
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
}

// 비즈니스 로직 빈은 컴포넌트 스캔
@Service
public class MemberService { ... }

@Repository
public class MemberRepository { ... }
```
