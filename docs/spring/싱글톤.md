# 싱글톤 (Singleton)

---

## 왜 싱글톤이 필요한가

웹 애플리케이션은 수많은 사용자가 동시에 요청을 보낸다.
만약 요청마다 새 객체를 생성하면 어떻게 될까?

```markdown
// 스프링 없이 순수 자바로 구현한다면
public class AppConfig {
    public OrderService orderService() {
        return new OrderServiceImpl(...); // 매번 새 객체 생성
    }
}
```

```
클라이언트 A 요청 → new OrderServiceImpl() 생성 (1개)
클라이언트 B 요청 → new OrderServiceImpl() 생성 (1개)
클라이언트 C 요청 → new OrderServiceImpl() 생성 (1개)
...
초당 1,000개 요청 → 1,000개의 OrderServiceImpl 생성, 사용 후 GC 대상
```

`OrderServiceImpl`은 상태가 없는 객체다. 굳이 매번 만들 필요가 없다.
**하나만 만들어서 공유하면 메모리 낭비 없이 효율적이다.**

---

## 순수 자바 싱글톤 패턴

자바로 직접 싱글톤을 구현하면 이렇게 작성한다.

```markdown
public class SingletonService {

    // 1. 클래스 로딩 시점에 인스턴스를 딱 1개 생성
    private static final SingletonService instance = new SingletonService();

    // 2. 생성자를 private으로 막아 외부에서 new 불가
    private SingletonService() {}

    // 3. 유일한 인스턴스를 반환하는 정적 메서드
    public static SingletonService getInstance() {
        return instance;
    }

    public void logic() {
        System.out.println("싱글톤 객체 로직 호출");
    }
}
```

```markdown
// 테스트
SingletonService s1 = SingletonService.getInstance();
SingletonService s2 = SingletonService.getInstance();

assertThat(s1).isSameAs(s2); // true — 같은 인스턴스!
```

---

## 순수 싱글톤 패턴의 문제점

직접 구현하는 싱글톤에는 여러 문제가 있다.

| 문제 | 설명 |
|---|---|
| 부가 코드 필요 | `private` 생성자, `getInstance()` 등 매번 작성해야 함 |
| DIP 위반 | 클라이언트가 `getInstance()`를 직접 호출 → 구체 클래스에 의존 |
| 테스트 어려움 | Mock 교체가 불가능, 테스트 간 상태가 공유됨 |
| 유연성 부족 | 자식 클래스 만들기 어려움 |
| 멀티스레드 위험 | 구현 방식에 따라 스레드 안전하지 않을 수 있음 |

```markdown
// DIP 위반 예
public class ClientA {
    private SingletonService service = SingletonService.getInstance(); // 구체 클래스 직접 참조
    // 인터페이스 기반으로 바꾸기 어려움
}
```

## 스프링의 싱글톤 컨테이너

스프링 컨테이너는 이 문제를 모두 해결하면서도 싱글톤을 보장해준다.

```markdown
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

MemberService s1 = ac.getBean("memberService", MemberService.class);
MemberService s2 = ac.getBean("memberService", MemberService.class);

assertThat(s1).isSameAs(s2); // true — 스프링이 같은 인스턴스를 반환
```

스프링은 싱글톤을 위해 `private` 생성자 같은 부가 코드 없이도 동작한다.
DI로 주입받기 때문에 클라이언트는 인터페이스에만 의존하면 된다.

| 장점 | 설명 |
|---|---|
| 자동 관리 | 싱글톤 구현 코드 불필요 |
| DIP 준수 | 인터페이스 기반으로 주입 |
| 테스트 용이 | Mock 객체 주입 가능 |
| 유연성 | 상속, 다형성 지원 |

## 싱글톤 주의사항: 무상태 설계 (Stateless)

> **싱글톤 객체는 상태를 가지면 안 된다.**

하나의 객체를 여러 스레드가 동시에 사용한다.
인스턴스 변수에 값을 쓰면 서로의 데이터를 덮어쓰는 심각한 버그가 발생한다.

### 문제 예시

```markdown
// 위험한 코드
public class StatefulService {
    private int price; // 공유 필드 → 여러 스레드가 동시에 쓸 수 있음

    public void order(String name, int price) {
        System.out.println("name=" + name + " price=" + price);
        this.price = price; // 상태 변경!
    }

    public int getPrice() {
        return price;
    }
}
```

```markdown
// 스레드 A: 김철수가 10,000원 주문
// 스레드 B: 동시에 김영희가 20,000원 주문

StatefulService service = ...; // 싱글톤 — 하나의 객체

// [스레드 A] 주문
service.order("김철수", 10000); // price = 10000

// [스레드 B] 스레드 A 사이에 끼어들어서 주문
service.order("김영희", 20000); // price = 20000 (덮어씀!)

// [스레드 A] 가격 조회
int price = service.getPrice();
// 기대: 10000  실제: 20000 → 버그!
```

실제 서비스에서 이런 버그가 생기면 다른 사람의 결제 금액이 내 영수증에 찍히는 상황이 된다.

### 올바른 무상태 설계

```markdown
// 안전한 코드 — 인스턴스 변수 없음
public class StatelessService {

    public int order(String name, int price) {
        System.out.println("name=" + name + " price=" + price);
        return price; // 지역 변수로 처리 — 스레드마다 독립적
    }
}
```

지역 변수, 파라미터, 반환값은 스레드마다 독립적인 스택에 저장된다. 공유 문제가 없다.

### 무상태 설계 체크리스트

| 코드 패턴 | 안전 여부 |
|---|---|
| 인스턴스 변수에 값을 쓴다 | **위험** |
| 인스턴스 변수를 읽기만 한다 (설정값 등) | 안전 |
| 지역 변수, 파라미터, 반환값을 사용한다 | 안전 |
| `ThreadLocal`을 사용한다 | 안전 (스레드마다 독립) |

### 스레드별 독립 상태가 필요할 때 — ThreadLocal

```markdown
// ThreadLocal: 스레드마다 다른 값을 저장
public class UserContext {
    private static final ThreadLocal<String> currentUser = new ThreadLocal<>();

    public static void set(String user) { currentUser.set(user); }
    public static String get() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); } // 요청 처리 후 반드시 제거!
}
```

`ThreadLocal`을 쓸 때는 **요청 처리가 끝난 후 반드시 `remove()`를 호출**해야 한다.
스레드 풀 환경에서는 스레드가 재사용되므로, 제거하지 않으면 이전 요청의 데이터가 남는다.

## @Configuration과 CGLIB — 싱글톤을 어떻게 보장하는가

### 문제: 자바 코드 그대로라면 싱글톤이 깨진다

```markdown
@Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        return new MemberServiceImpl(memberRepository()); // memberRepository() 호출 1
    }

    @Bean
    public OrderService orderService() {
        return new OrderServiceImpl(memberRepository(), discountPolicy()); // memberRepository() 호출 2
    }

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository(); // new!
    }
}
```

자바 코드라면 `memberRepository()`가 2번 호출되어 `MemoryMemberRepository`가 2개 생길 것 같다.
하지만 실제로 실행해보면 **1개만 생성된다**. 어떻게 된 것일까?

### @Configuration의 CGLIB 바이트코드 조작

스프링은 `@Configuration`이 붙은 클래스를 CGLIB 라이브러리로 **프록시 클래스**로 만들어 교체한다.

```
AppConfig 원본
    ↓ CGLIB이 프록시 클래스 생성
AppConfig$$EnhancerBySpringCGLIB$$xxxx (프록시)
    → @Bean 메서드를 오버라이딩
    → 이미 컨테이너에 등록된 빈이면 기존 빈 반환
    → 없으면 새로 생성 후 등록
```

```markdown
// CGLIB이 생성하는 코드 (개념적으로)
@Bean
public MemberRepository memberRepository() {
    if (스프링 컨테이너에 memberRepository 빈이 이미 있다면) {
        return 기존 빈; // 싱글톤 보장
    }
    MemberRepository repo = new MemoryMemberRepository();
    스프링 컨테이너에 등록;
    return repo;
}
```

```markdown
// 확인 방법
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
AppConfig bean = ac.getBean(AppConfig.class);
System.out.println(bean.getClass());
// 출력: class hello.core.AppConfig$$EnhancerBySpringCGLIB$$12ab34cd
// → 원본 클래스가 아니라 CGLIB이 만든 프록시 클래스
```

### @Configuration 없이 @Bean만 쓰면?

```markdown
// @Configuration 없이 사용
public class AppConfig { // @Configuration 없음!
    @Bean
    public MemberService memberService() {
        return new MemberServiceImpl(memberRepository()); // new MemoryMemberRepository()
    }

    @Bean
    public OrderService orderService() {
        return new OrderServiceImpl(memberRepository(), ...); // 또 new MemoryMemberRepository()!
    }
}
```

- CGLIB이 적용되지 않아 `memberRepository()`가 매번 새 인스턴스를 생성
- **싱글톤이 깨진다**

```
결론:
@Configuration + @Bean → CGLIB 프록시 적용 → 싱글톤 보장 ✓
@Bean만 (단독)         → 순수 자바 → 싱글톤 미보장 ✗
```

> **항상 `@Configuration`과 `@Bean`을 함께 사용할 것.**
