# 싱글톤 (Singleton)

---

## 싱글톤 패턴이란?

클래스의 인스턴스가 **딱 1개만 생성**되는 것을 보장하는 디자인 패턴.

### 왜 싱글톤이 필요한가?

웹 애플리케이션에서 수많은 요청이 들어올 때마다 객체를 새로 생성하면:

```
클라이언트 A 요청 → MemberService 생성 (객체 1개)
클라이언트 B 요청 → MemberService 생성 (객체 1개)
클라이언트 C 요청 → MemberService 생성 (객체 1개)
...
초당 100개 요청 → 100개의 MemberService 생성
```

메모리 낭비가 심각해진다. 대부분의 서비스 객체는 **상태 없이(Stateless)** 사용하므로 하나만 만들어서 공유하는 것이 합리적이다.

---

## 순수 자바 싱글톤 패턴 구현

```java
public class SingletonService {

    // 1. static 필드에 딱 1개만 생성
    private static final SingletonService instance = new SingletonService();

    // 2. 외부에서 new 불가
    private SingletonService() {}

    // 3. 인스턴스를 반환하는 static 메서드
    public static SingletonService getInstance() {
        return instance;
    }

    public void logic() {
        System.out.println("싱글톤 객체 로직 호출");
    }
}
```

```java
// 테스트
SingletonService s1 = SingletonService.getInstance();
SingletonService s2 = SingletonService.getInstance();
assertThat(s1).isSameAs(s2); // 같은 인스턴스!
```

---

## 순수 싱글톤 패턴의 문제점

| 문제 | 설명 |
|---|---|
| 구현 코드 증가 | `getInstance()`, `private` 생성자 등 부가 코드 작성 필요 |
| DIP 위반 | 클라이언트가 `getInstance()`를 직접 호출 → 구체 클래스에 의존 |
| OCP 위반 | 변경이 어려움 |
| 테스트 어려움 | Mock 교체 불가, 테스트 간 상태 공유로 격리 어려움 |
| 유연성 부족 | 자식 클래스 만들기 어려움 |
| 멀티스레드 위험 | 구현 방식에 따라 스레드 안전하지 않을 수 있음 |

```java
// DIP 위반 예시
public class ClientA {
    // getInstance()를 통해 구체 클래스를 직접 참조
    private SingletonService service = SingletonService.getInstance();
}
```

## 스프링의 싱글톤 컨테이너 (싱글톤 레지스트리)

스프링 컨테이너는 싱글톤 패턴의 단점을 해결하면서 싱글톤으로 객체를 관리한다.

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);

MemberService s1 = ac.getBean("memberService", MemberService.class);
MemberService s2 = ac.getBean("memberService", MemberService.class);

// 스프링 컨테이너가 같은 인스턴스 반환!
assertThat(s1).isSameAs(s2);
```

### 스프링 싱글톤의 장점

| 장점 | 설명 |
|---|---|
| 자동 관리 | `private` 생성자 등 부가 코드 불필요 |
| DIP 준수 | 클라이언트가 인터페이스에만 의존 |
| OCP 준수 | 스프링이 의존관계 주입 |
| 테스트 용이 | Mock 객체 주입 가능 |
| 유연성 | 상속, 다형성 지원 |

## 싱글톤 방식의 주의사항: 무상태 설계

### 핵심 원칙

> 싱글톤 객체는 **상태를 가지면 안 된다 (Stateless)**.

여러 스레드가 하나의 객체를 공유하므로, 상태(필드)가 있으면 **공유 필드 문제**가 발생한다.

### 문제 예시

```java
// 위험한 코드: 공유 필드(상태) 있음
public class StatefulService {
    private int price; // 공유 필드 → 위험!!!

    public void order(String name, int price) {
        System.out.println("name=" + name + " price=" + price);
        this.price = price; // 상태 변경!
    }

    public int getPrice() {
        return price;
    }
}
```

```java
// 스레드 A: 김철수가 10000원 주문
// 스레드 B: 김영희가 20000원 주문 (거의 동시)
StatefulService service = ... // 싱글톤

// 스레드 A
service.order("김철수", 10000);

// 스레드 B (스레드 A 사이에 끼어듦)
service.order("김영희", 20000);

// 스레드 A가 가격 조회
int price = service.getPrice();
// 기대: 10000, 실제: 20000 ← 버그!
```

### 올바른 무상태 설계

```java
// 안전한 코드: 상태를 지역 변수나 반환값으로 처리
public class StatelessService {
    // 공유 필드 없음!

    public int order(String name, int price) {
        System.out.println("name=" + name + " price=" + price);
        return price; // 반환값으로 처리 (지역 변수)
    }
}
```

```java
// ThreadLocal을 사용해야 하는 경우 (스레드마다 독립적인 상태가 필요)
public class UserContext {
    private static final ThreadLocal<String> currentUser = new ThreadLocal<>();

    public static void setCurrentUser(String user) { currentUser.set(user); }
    public static String getCurrentUser() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); } // 요청 처리 후 반드시 제거!
}
```

### 무상태 설계 체크리스트

- 인스턴스 변수에 값을 쓰는가? → **위험**
- 해당 필드가 클라이언트마다 다른 값이어야 하는가? → **위험**
- 읽기 전용으로만 사용하는가? → 안전
- 지역 변수, 파라미터, 반환값을 활용하는가? → 안전

## @Configuration과 CGLIB

### 문제: 자바 코드 그대로면 싱글톤이 깨짐

```java
@Configuration
public class AppConfig {
    @Bean
    public MemberService memberService() {
        return new MemberServiceImpl(memberRepository()); // memberRepository() 호출
    }

    @Bean
    public OrderService orderService() {
        return new OrderServiceImpl(memberRepository(), discountPolicy()); // memberRepository() 호출
    }

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository(); // new!
    }
}
```

단순히 자바 코드라면 `memberRepository()`가 2번 호출되어 `MemoryMemberRepository`가 2개 생성되어야 할 것 같다. 하지만 실제로는 **1개만 생성**된다.

### @Configuration의 CGLIB 바이트코드 조작

스프링은 `@Configuration`이 붙은 클래스를 바이트코드 조작 라이브러리(CGLIB)를 사용해 **프록시 클래스로 교체**한다.

```
AppConfig 원본
    ↓ CGLIB
AppConfig$$EnhancerBySpringCGLIB$$abcd (프록시)
    → @Bean 메서드 오버라이딩
    → 이미 스프링 컨테이너에 등록된 빈이면 기존 것 반환
    → 없으면 새로 생성 후 등록
```

```java
// CGLIB이 생성하는 코드 (개념적)
@Bean
public MemberRepository memberRepository() {
    if (스프링 컨테이너에 memberRepository 빈이 이미 있다면) {
        return 기존 빈; // 싱글톤 보장
    }
    MemberRepository repo = new MemoryMemberRepository();
    스프링 컨테이너에 등록;
    return repo;
}
```

### 확인

```java
ApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);
AppConfig bean = ac.getBean(AppConfig.class);

// AppConfig$$EnhancerBySpringCGLIB$$... 출력됨 (프록시 클래스)
System.out.println(bean.getClass());
```

### @Configuration 없이 @Bean만 사용하면?

```java
// @Configuration 없이 @Bean만 사용
public class AppConfig {
    @Bean
    public MemberService memberService() {
        return new MemberServiceImpl(memberRepository()); // new MemoryMemberRepository()
    }

    @Bean
    public OrderService orderService() {
        return new OrderServiceImpl(memberRepository(), ...); // 또 new MemoryMemberRepository()
    }

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository(); // 매번 새로 생성!
    }
}
```

- CGLIB 적용 안 됨 → `memberRepository()`가 매번 새 인스턴스를 생성
- **싱글톤 깨짐!**

```
결론:
- @Configuration 있음 → CGLIB 프록시 → 싱글톤 보장 ✓
- @Configuration 없음 (@Bean만) → 순수 자바 → 싱글톤 미보장 ✗
```

> **항상 `@Configuration`과 `@Bean`을 함께 사용**하거나, `@Component`와 `@Autowired`를 사용.
