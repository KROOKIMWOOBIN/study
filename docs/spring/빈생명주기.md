# 빈 생명주기 콜백

---

## 빈 생명주기란? 왜 필요한가

애플리케이션 시작 시점에 DB 커넥션 풀, 네트워크 소켓 등을 **미리 연결**해두고, 종료 시점에 **안전하게 연결을 끊어야** 하는 경우가 있다.

스프링은 빈의 **생성(초기화)** 과 **소멸** 시점에 콜백을 제공한다.

---

## 스프링 빈 생명주기 순서

```
스프링 컨테이너 생성
    ↓
스프링 빈 생성 (객체 new)
    ↓
의존관계 주입 (DI)
    ↓
[초기화 콜백] ← 여기서 초기화 작업 수행
    ↓
사용
    ↓
[소멸 전 콜백] ← 여기서 정리 작업 수행
    ↓
스프링 종료
```

### 생성자 주입의 경우

```
빈 생성과 DI가 동시에 발생 (생성자에서 의존관계 주입)
    ↓
초기화 콜백
    ↓
사용
    ↓
소멸 전 콜백
```

### 왜 생성자에서 초기화를 하지 않는가?

```java
// 생성자에서 초기화하는 방법은 권장하지 않음
public class NetworkClient {
    private String url;

    public NetworkClient(String url) {
        this.url = url;
        connect(); // 생성자에서 초기화 — 비권장
    }
}
```

**이유**: 생성자는 객체 생성에만 집중해야 한다. (단일 책임 원칙)
- 생성자에 무거운 초기화 코드를 넣으면 유지보수가 어려워짐
- 초기화 작업이 외부와 연동되는 경우 단위 테스트가 어려워짐

---

## 초기화/소멸 콜백 3가지 방법

### 방법 1: @PostConstruct / @PreDestroy (권장)

```java
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;

@Component
public class NetworkClient {
    private String url;

    @Autowired
    public void setUrl(String url) {
        this.url = url;
    }

    @PostConstruct // 의존관계 주입 완료 후 호출
    public void init() {
        System.out.println("NetworkClient.init url=" + url);
        connect();
        call("초기화 연결 메시지");
    }

    @PreDestroy // 빈 소멸 전 호출
    public void close() {
        System.out.println("NetworkClient.close");
        disconnect();
    }

    private void connect() {
        System.out.println("connect: " + url);
    }

    private void call(String message) {
        System.out.println("call: " + url + " message=" + message);
    }

    private void disconnect() {
        System.out.println("close: " + url);
    }
}
```

**@PostConstruct/@PreDestroy 권장 이유**:

| 이유 | 설명 |
|---|---|
| 표준 기술 | `javax.annotation` (Jakarta EE 표준) — 스프링에 종속되지 않음 |
| 간결함 | 어노테이션 하나로 해결 |
| 컴포넌트 스캔 친화적 | 별도 설정 불필요 |
| 코드 가독성 | 초기화/소멸 메서드가 클래스 안에 명확히 드러남 |

**단점**:
- 외부 라이브러리에는 적용 불가 (내 코드에 어노테이션을 달 수 없음)

---

### 방법 2: InitializingBean / DisposableBean (레거시)

```java
import org.springframework.beans.factory.DisposableBean;
import org.springframework.beans.factory.InitializingBean;

public class NetworkClient implements InitializingBean, DisposableBean {

    private String url;

    public NetworkClient() {
        System.out.println("생성자 호출, url=" + url);
    }

    public void setUrl(String url) {
        this.url = url;
    }

    @Override
    public void afterPropertiesSet() throws Exception {
        // 의존관계 주입 완료 후 호출
        System.out.println("NetworkClient.afterPropertiesSet");
        connect();
        call("초기화 연결 메시지");
    }

    @Override
    public void destroy() throws Exception {
        // 빈 소멸 전 호출
        System.out.println("NetworkClient.destroy");
        disconnect();
    }
}
```

**단점**:

| 단점 | 설명 |
|---|---|
| 스프링 전용 인터페이스 | 스프링에 의존하게 됨 |
| 메서드 이름 변경 불가 | `afterPropertiesSet()`, `destroy()` 고정 |
| 외부 라이브러리 적용 불가 | 마찬가지로 내 코드에만 적용 가능 |

> 이 방법은 스프링 초창기 방법이며, 현재는 더 나은 방법이 있으므로 사용 권장하지 않음.

---

### 방법 3: @Bean initMethod / destroyMethod

```java
public class NetworkClient {
    private String url;

    public void setUrl(String url) {
        this.url = url;
    }

    // 초기화 메서드 (이름 자유롭게 지정)
    public void init() {
        System.out.println("NetworkClient.init");
        connect();
        call("초기화 연결 메시지");
    }

    // 소멸 메서드 (이름 자유롭게 지정)
    public void close() {
        System.out.println("NetworkClient.close");
        disconnect();
    }
}
```

```java
@Configuration
public class LifeCycleConfig {
    @Bean(initMethod = "init", destroyMethod = "close")
    public NetworkClient networkClient() {
        NetworkClient networkClient = new NetworkClient();
        networkClient.setUrl("http://hello.dev");
        return networkClient;
    }
}
```

**장점**:

| 장점 | 설명 |
|---|---|
| 메서드 이름 자유 | 원하는 이름으로 지정 가능 |
| 스프링 코드 무관 | `NetworkClient`가 스프링에 의존하지 않음 |
| **외부 라이브러리 적용 가능** | 소스 코드를 수정할 수 없는 외부 라이브러리에도 사용 가능 |

**destroyMethod의 기본값 추론 기능**:

```java
@Bean(initMethod = "init")
// destroyMethod를 지정하지 않으면 기본값 "(inferred)"
// "close" 또는 "shutdown" 이름의 메서드를 자동으로 찾아서 호출
// 대부분의 외부 라이브러리가 close() 또는 shutdown()을 가지고 있어서 편리
```

```java
// destroyMethod 기본 추론 비활성화
@Bean(destroyMethod = "") // 빈 문자열로 추론 기능 끔
```

## 세 가지 방법 비교

| 구분 | @PostConstruct/@PreDestroy | InitializingBean/DisposableBean | @Bean initMethod/destroyMethod |
|---|---|---|---|
| 권장 여부 | **강력 권장** | 비권장 (레거시) | 외부 라이브러리에 사용 |
| 스프링 의존 | 없음 (Jakarta 표준) | 있음 | 없음 |
| 적용 위치 | 클래스 내 어노테이션 | 인터페이스 구현 | @Bean 속성 |
| 외부 라이브러리 | 불가 | 불가 | **가능** |
| 메서드명 자유 | 자유 | 고정 | 자유 |
| 코드 간결성 | 가장 간결 | 중간 | 설정 코드 필요 |

## 실제 활용 예제

### DB 커넥션 풀 초기화

```java
@Component
public class DatabaseConnectionPool {
    private List<Connection> connectionPool = new ArrayList<>();

    @PostConstruct
    public void init() {
        // 애플리케이션 시작 시 커넥션 미리 생성
        for (int i = 0; i < 10; i++) {
            connectionPool.add(createConnection());
        }
        System.out.println("커넥션 풀 초기화 완료: " + connectionPool.size() + "개");
    }

    @PreDestroy
    public void destroy() {
        // 애플리케이션 종료 시 모든 커넥션 닫기
        for (Connection conn : connectionPool) {
            closeConnection(conn);
        }
        System.out.println("모든 커넥션 정리 완료");
    }
}
```

### 외부 라이브러리 (DataSource) — @Bean 방식

```java
@Configuration
public class DataSourceConfig {
    @Bean(destroyMethod = "close") // HikariDataSource.close() 자동 호출
    public DataSource dataSource() {
        HikariDataSource dataSource = new HikariDataSource();
        dataSource.setJdbcUrl("jdbc:mysql://localhost:3306/db");
        // ...
        return dataSource;
    }
}
```

### 캐시 워밍업 (ApplicationReadyEvent)

```java
@Component
public class CacheWarmup {

    private final ProductRepository productRepository;

    public CacheWarmup(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    // @PostConstruct 보다 늦은 시점 (모든 초기화 완료 후)
    @EventListener(ApplicationReadyEvent.class)
    public void warmupCache() {
        List<Product> popularProducts = productRepository.findTop100ByOrderBySalesDesc();
        // 캐시에 저장...
        System.out.println("캐시 워밍업 완료: " + popularProducts.size() + "건");
    }
}
```

## 싱글톤 빈 소멸 시점

```
스프링 컨테이너 종료 (ac.close())
    ↓
싱글톤 빈 소멸 전 콜백 (@PreDestroy, destroy() 등)
    ↓
빈 소멸

※ 프로토타입 빈은 스프링이 소멸 콜백을 호출하지 않음!
   프로토타입 빈의 소멸은 클라이언트가 직접 관리해야 함.
```
