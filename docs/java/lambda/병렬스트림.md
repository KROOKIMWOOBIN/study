## 병렬 스트림 (Parallel Stream)

### 왜 쓰는지

일반 스트림은 단일 스레드로 순차 처리한다. 데이터가 많을 때 CPU 코어를 하나만 쓰는 것은 낭비다. 병렬 스트림은 내부적으로 `ForkJoinPool`을 사용해 작업을 여러 스레드로 분할·병합하여 처리 속도를 높인다.

### 언제 쓰는지

| 적합 | 부적합 |
|------|--------|
| 데이터가 충분히 많을 때 (수만 건 이상) | 데이터가 적을 때 (스레드 분할 오버헤드 > 이득) |
| 연산 비용이 클 때 (CPU-intensive) | I/O 대기가 주된 작업일 때 |
| 요소 순서가 중요하지 않을 때 | 순서 보장이 필요할 때 |
| 상태 없는(stateless) 연산일 때 | 공유 상태 변경이 있을 때 |

### 사용법

```markdown
// 순차 스트림 → 병렬 스트림
list.parallelStream()
    .filter(n -> n % 2 == 0)
    .mapToLong(Long::valueOf)
    .sum();

// 기존 스트림을 병렬로 전환
list.stream()
    .parallel()
    .map(String::toUpperCase)
    .toList();
```

### 내부 동작

```markdown
// ForkJoinPool.commonPool() 을 공유 사용
// 기본 스레드 수 = CPU 코어 수 - 1
ForkJoinPool.commonPool().getParallelism(); // 예: 7 (8코어 기준)

// 커스텀 풀로 실행하려면
ForkJoinPool customPool = new ForkJoinPool(4);
customPool.submit(() ->
    list.parallelStream().forEach(System.out::println)
).get();
```

### 장점

- CPU 코어를 모두 활용해 대용량 데이터 처리 속도 향상
- 코드 변경이 거의 없음 (`stream()` → `parallelStream()`)
- `reduce`, `collect` 같은 종단 연산이 자동으로 결과를 합산

### 단점

- 스레드 분할·합산 오버헤드로 소량 데이터에서는 오히려 느림
- 공유 `ForkJoinPool`을 사용하므로 다른 병렬 작업에 영향
- 순서 보장 안 됨 (`forEachOrdered`로 강제 가능하지만 성능 손해)
- 디버깅이 어려움

### 주의할 점

**1. 공유 상태 변경 금지**

```markdown
// 잘못된 예 - 레이스 컨디션 발생
List<Integer> result = new ArrayList<>();
list.parallelStream().forEach(result::add); // 위험!

// 올바른 예 - 스레드 안전한 수집
List<Integer> result = list.parallelStream()
    .filter(n -> n > 0)
    .collect(Collectors.toList()); // 내부적으로 안전하게 합산
```

**2. 순서 의존 연산 주의**

```markdown
// findFirst()는 순서 보장 → 병렬에서 성능 손해 가능
list.parallelStream().findFirst();

// 순서 상관없으면 findAny() 사용
list.parallelStream().findAny(); // 더 빠름
```

**3. 성능 측정 필수**

병렬 스트림이 항상 빠르지 않다. 반드시 벤치마크(JMH 등)로 실제 이득을 확인한 후 적용한다.

### 특징 요약

| 항목 | 순차 스트림 | 병렬 스트림 |
|------|-------------|-------------|
| 실행 방식 | 단일 스레드 순서대로 | ForkJoinPool 멀티 스레드 |
| 순서 보장 | O | X (기본) |
| 소량 데이터 | 빠름 | 느림 (오버헤드) |
| 대량 데이터 | 느림 | 빠름 |
| 안전성 | 항상 안전 | 공유 상태 주의 필요 |
