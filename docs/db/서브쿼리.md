# 서브쿼리

<div class="concept-box" markdown="1">

**서브쿼리(Subquery)**: 쿼리 안에 중첩된 또 다른 쿼리. 외부 쿼리(메인 쿼리)가 실행될 때 내부 쿼리(서브쿼리)의 결과를 값 또는 집합으로 활용한다.

</div>

## 왜 쓰는지

단일 쿼리로 표현하기 어려운 **다단계 조회**를 하나의 SQL로 처리할 때 사용한다.

```sql
-- "평균 주문 금액보다 많이 주문한 회원" 조회
-- → 먼저 평균을 구하고, 그 값으로 필터링해야 함
SELECT name
FROM member
WHERE total_amount > (SELECT AVG(total_amount) FROM member);
```

평균을 먼저 별도로 조회한 뒤 WHERE에 대입하는 2단계 작업을 서브쿼리 하나로 표현한다.

## 위치별 종류

### SELECT 절 — 스칼라 서브쿼리

행마다 **단일 값(스칼라)** 을 반환하는 서브쿼리. 컬럼처럼 사용한다.

```sql
-- 회원별로 주문 건수를 함께 조회
SELECT
    m.name,
    (SELECT COUNT(*) FROM orders o WHERE o.member_id = m.id) AS order_count
FROM member m;
```

- 행마다 서브쿼리가 실행되므로 **상관 서브쿼리**와 같은 특성을 가진다.
- 반드시 **단일 행, 단일 컬럼**을 반환해야 한다. 여러 행 반환 시 오류.

### FROM 절 — 인라인 뷰

서브쿼리 결과를 **임시 테이블처럼** 사용한다. 반드시 별칭(alias)을 붙여야 한다.

```sql
-- 회원별 총 주문 금액을 먼저 집계한 뒤, 상위 3명 조회
SELECT t.name, t.total
FROM (
    SELECT m.name, SUM(o.amount) AS total
    FROM member m
    JOIN orders o ON m.id = o.member_id
    GROUP BY m.id, m.name
) AS t
ORDER BY t.total DESC
LIMIT 3;
```

- GROUP BY + 집계 결과에 다시 조건을 걸 때 유용하다.
- MySQL 8.0 이상에서는 CTE(`WITH`)로 대체하면 더 가독성이 좋다.

```sql
-- CTE 버전 (권장)
WITH order_totals AS (
    SELECT m.name, SUM(o.amount) AS total
    FROM member m
    JOIN orders o ON m.id = o.member_id
    GROUP BY m.id, m.name
)
SELECT name, total FROM order_totals ORDER BY total DESC LIMIT 3;
```

### WHERE 절

#### 단일 행 서브쿼리

`=`, `>`, `<` 등 **비교 연산자**와 함께 사용. 서브쿼리가 **정확히 1행**을 반환해야 한다.

```sql
-- 가장 최근 주문한 회원 조회
SELECT name
FROM member
WHERE id = (SELECT member_id FROM orders ORDER BY created_at DESC LIMIT 1);
```

#### 다중 행 서브쿼리

`IN`, `NOT IN`, `ANY`, `ALL` 등과 함께 사용. 서브쿼리가 **여러 행**을 반환한다.

```sql
-- 한 번이라도 주문한 적 있는 회원 조회
SELECT name
FROM member
WHERE id IN (SELECT DISTINCT member_id FROM orders);

-- 한 번도 주문하지 않은 회원 조회
SELECT name
FROM member
WHERE id NOT IN (SELECT DISTINCT member_id FROM orders);
```

<div class="warning-box" markdown="1">

**NOT IN + NULL 주의** — 서브쿼리 결과에 `NULL`이 하나라도 있으면 `NOT IN`은 항상 FALSE를 반환해 결과가 비어버린다.

```sql
-- ❌ orders.member_id에 NULL이 있으면 결과가 0행
SELECT name FROM member WHERE id NOT IN (SELECT member_id FROM orders);

-- ✅ NULL 제거 후 사용하거나 NOT EXISTS로 대체
SELECT name FROM member WHERE id NOT IN (SELECT member_id FROM orders WHERE member_id IS NOT NULL);
```

</div>

## 상관 서브쿼리 (Correlated Subquery)

<div class="concept-box" markdown="1">

**상관 서브쿼리**: 외부 쿼리의 컬럼을 서브쿼리 내부에서 참조하는 방식. 외부 쿼리의 **행마다** 서브쿼리가 새로 실행된다.

</div>

```sql
-- 자신의 부서 평균 급여보다 많이 받는 직원 조회
SELECT e.name, e.salary, e.dept_id
FROM employee e
WHERE e.salary > (
    SELECT AVG(salary)
    FROM employee
    WHERE dept_id = e.dept_id  -- 외부 쿼리의 e.dept_id 참조
);
```

```text
외부 쿼리 행 1 (e.dept_id = 10) → 서브쿼리 실행: dept_id=10 평균 계산
외부 쿼리 행 2 (e.dept_id = 20) → 서브쿼리 실행: dept_id=20 평균 계산
...
→ 행 수만큼 반복 실행
```

### EXISTS / NOT EXISTS

상관 서브쿼리의 대표적인 활용. **존재 여부**만 확인하므로 `IN`보다 효율적인 경우가 많다.

```sql
-- 주문이 있는 회원 조회 (EXISTS)
SELECT m.name
FROM member m
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.member_id = m.id
);

-- 주문이 없는 회원 조회 (NOT EXISTS)
SELECT m.name
FROM member m
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.member_id = m.id
);
```

- `SELECT 1` — 존재 여부만 확인하므로 실제 컬럼 값은 중요하지 않다. 관례적으로 `1`을 사용한다.
- 조건에 맞는 행을 찾는 순간 즉시 멈추므로(**short-circuit**) 대량 데이터에서 `IN`보다 빠를 수 있다.

## IN vs EXISTS 비교

| | `IN` | `EXISTS` |
|---|---|---|
| 동작 방식 | 서브쿼리 결과 전체를 메모리에 올린 뒤 비교 | 행마다 서브쿼리 실행, 조건 맞으면 즉시 중단 |
| NULL 처리 | NULL 포함 시 `NOT IN` 오작동 | NULL 영향 없음 |
| 적합한 상황 | 서브쿼리 결과가 소량일 때 | 서브쿼리 테이블이 크고 인덱스가 있을 때 |
| 가독성 | 단순 목록 비교에 직관적 | 존재 여부 확인에 의미가 명확 |

## 언제 JOIN 대신 서브쿼리를 쓰는지

| 상황 | 이유 |
|------|------|
| 집계 결과로 필터링 | `HAVING`으로 해결 안 될 때, 다른 테이블 집계값과 비교 |
| 존재 여부만 확인 | `EXISTS`가 JOIN보다 간결하고 빠름 |
| 결과 행 수를 유지해야 할 때 | JOIN은 to-many 관계에서 행이 증가하지만 서브쿼리는 유지됨 |
| 가독성이 중요할 때 | 복잡한 조건을 단계별로 표현 |

**JOIN — 행 증가 문제**

```sql
-- 주문한 회원의 이름만 조회하려 했지만
SELECT DISTINCT m.name
FROM member m
JOIN orders o ON m.id = o.member_id;
-- DISTINCT 없으면 중복 행 발생
```

**EXISTS — 행 수 유지**

```sql
-- 행 증가 없이 존재 여부만 확인
SELECT m.name
FROM member m
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.member_id = m.id
);
```

## 주의할 점

<div class="warning-box" markdown="1">

**상관 서브쿼리 성능 주의** — 외부 쿼리 행 수만큼 반복 실행되므로 대량 데이터에서 느려질 수 있다. 가능하면 JOIN 또는 CTE로 대체한다.

```sql
-- ❌ 100만 행이면 서브쿼리 100만 번 실행
SELECT name, (SELECT COUNT(*) FROM orders WHERE member_id = m.id) AS cnt
FROM member m;

-- ✅ JOIN + GROUP BY로 대체
SELECT m.name, COUNT(o.id) AS cnt
FROM member m
LEFT JOIN orders o ON m.id = o.member_id
GROUP BY m.id, m.name;
```

</div>

<div class="danger-box" markdown="1">

**중첩 서브쿼리 남용 금지** — 서브쿼리를 3단계 이상 중첩하면 가독성과 성능 모두 나빠진다. CTE(`WITH`)로 단계를 분리한다.

```sql
-- ❌ 3단계 중첩 → 읽기 어렵고 최적화도 어려움
SELECT * FROM (
    SELECT * FROM (
        SELECT * FROM orders WHERE amount > 10000
    ) AS t1 WHERE ...
) AS t2 WHERE ...;

-- ✅ CTE로 분리
WITH filtered AS (SELECT * FROM orders WHERE amount > 10000),
     ranked   AS (SELECT *, ROW_NUMBER() OVER (...) AS rn FROM filtered)
SELECT * FROM ranked WHERE rn <= 10;
```

</div>
