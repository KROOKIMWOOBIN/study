# 인덱스 (Index)

<div class="concept-box" markdown="1">

**인덱스**: 테이블의 특정 컬럼에 대해 빠른 검색을 가능하게 하는 자료구조. 책의 "색인"처럼 원하는 데이터를 전체 테이블을 읽지 않고도 빠르게 찾을 수 있게 한다.

</div>

## 왜 쓰는지

인덱스 없이 `WHERE id = 100` 같은 쿼리를 실행하면 DBMS는 테이블의 **모든 행을 처음부터 끝까지 스캔(Full Table Scan)** 한다. 행이 100만 건이면 100만 번 비교한다.

인덱스를 만들면 B-Tree 구조에 정렬된 값이 저장되어 **O(log n)** 으로 찾을 수 있다.

| 상황 | 인덱스 없음 | 인덱스 있음 |
|------|------------|------------|
| 1,000만 건에서 특정 회원 조회 | ~10,000,000 행 비교 | ~23 번 비교 (log₂ 10M) |
| 복잡한 JOIN + WHERE | 매우 느림 | 대폭 개선 |

## 내부 구조 — B-Tree

대부분의 RDBMS(MySQL InnoDB, PostgreSQL)는 **B-Tree(Balanced Tree)** 를 기본 인덱스 구조로 사용한다.

```text
        [50]
       /    \
   [25]      [75]
   /  \      /  \
 [10][30] [60][90]
```

- 모든 리프 노드까지의 깊이가 같다. (Balanced)
- 범위 검색(`BETWEEN`, `>`, `<`)에도 효율적이다.
- 리프 노드끼리 연결 리스트로 이어져 있어 순차 접근도 빠르다.

<div class="tip-box" markdown="1">

**팁**: MySQL InnoDB는 B-Tree가 아닌 **B+Tree**를 사용한다. 내부 노드에는 키만, 리프 노드에 키+데이터를 모두 저장하여 범위 스캔이 더 빠르다.

</div>

## 다른 인덱스 구조

B-Tree가 기본이지만, 용도에 따라 다른 구조의 인덱스도 사용할 수 있다.

### Hash Index

<div class="concept-box" markdown="1">

**Hash Index**: 키를 해시 함수로 변환하여 **O(1)** 에 접근하는 인덱스. 등호(`=`) 비교에 최적화되어 있다.

</div>

- **범위 검색 불가**: `BETWEEN`, `>`, `<` 사용 불가
- **정렬 불가**: `ORDER BY`에 활용할 수 없음
- **부분 일치 불가**: `LIKE 'abc%'` 사용 불가
- MySQL InnoDB는 Hash Index를 직접 생성할 수 없다. 내부적으로 **Adaptive Hash Index**를 자동 생성하여 자주 조회되는 B-Tree 리프를 캐싱한다.

### Full-Text Index

<div class="concept-box" markdown="1">

**Full-Text Index**: 텍스트 내용을 단어 단위로 분해하여 **역인덱스(Inverted Index)** 로 저장하는 인덱스. 자연어 검색에 사용한다.

</div>

`LIKE '%keyword%'`는 앞쪽 와일드카드 때문에 항상 Full Scan이지만, Full-Text Index를 사용하면 대용량 텍스트에서도 빠르게 검색할 수 있다.

```sql
-- Full-Text 인덱스 생성
CREATE FULLTEXT INDEX ft_article_content ON article (title, content);

-- 검색
SELECT * FROM article
WHERE MATCH(title, content) AGAINST('인덱스 최적화');
```

<div class="tip-box" markdown="1">

**팁**: MySQL InnoDB는 5.6부터 Full-Text Index를 지원한다. 한국어·중국어·일본어(CJK)는 공백 기반 토큰화가 불가능하므로 **ngram parser**를 설정해야 한다.

```sql
CREATE FULLTEXT INDEX ft_article_content ON article (content) WITH PARSER ngram;
```

</div>

### 인덱스 구조 비교

| | B-Tree | Hash | Full-Text |
|--|--------|------|-----------|
| **등호 검색** | O(log n) | O(1) | - |
| **범위 검색** | 가능 | 불가 | - |
| **정렬** | 가능 | 불가 | - |
| **텍스트 검색** | LIKE 'abc%'만 | 불가 | 자연어 검색 |
| **InnoDB 지원** | 기본 | 자동 생성만 | 5.6+ |

## 클러스터형 vs 논클러스터형

| | 클러스터형 (Clustered) | 논클러스터형 (Non-Clustered) |
|--|----------------------|---------------------------|
| **저장 방식** | 인덱스 순서대로 실제 데이터 정렬 | 별도 인덱스 구조에 포인터 저장 |
| **개수** | 테이블당 1개 (PK가 클러스터형) | 여러 개 가능 |
| **조회 속도** | 빠름 (데이터 직접 접근) | 상대적으로 느림 (포인터 → 데이터) |
| **삽입 비용** | 순서 유지 필요 → 비쌈 | 상대적으로 저렴 |

MySQL InnoDB에서는 **PK가 클러스터형 인덱스**이고, 나머지는 논클러스터형(Secondary Index)이다. Secondary Index의 리프에는 실제 데이터 주소 대신 **PK 값**이 저장되어, PK로 다시 클러스터 인덱스를 조회한다 (==Double Lookup==).

## 커버링 인덱스 (Covering Index)

<div class="concept-box" markdown="1">

**커버링 인덱스**: 쿼리가 필요로 하는 모든 컬럼이 인덱스에 포함되어 있어, ==테이블 데이터에 접근하지 않고 인덱스만으로 결과를 반환==하는 것. Secondary Index의 Double Lookup을 제거하여 디스크 I/O를 대폭 줄인다.

</div>

<div class="compare-grid" markdown="1">
<div class="before" markdown="1">
**Bad — Double Lookup 발생**

```sql
-- idx_order_member_date: (member_id, created_at)
SELECT * FROM orders
WHERE member_id = 1;
```

인덱스로 PK를 찾고 → PK로 다시 클러스터 인덱스를 조회한다.

</div>
<div class="after" markdown="1">
**Good — 커버링 인덱스**

```sql
-- idx_order_member_date: (member_id, created_at)
SELECT member_id, created_at FROM orders
WHERE member_id = 1;
```

필요한 컬럼이 모두 인덱스에 있으므로 테이블 접근 없이 응답한다.

</div>
</div>

EXPLAIN으로 확인할 때 `Extra` 컬럼에 ==**Using index**==가 표시되면 커버링 인덱스가 사용된 것이다.

```sql
EXPLAIN SELECT member_id, created_at FROM orders WHERE member_id = 1;
-- Extra: Using index  ← 커버링 인덱스 사용 중
```

<div class="warning-box" markdown="1">

**주의**: `SELECT *`를 남발하면 커버링 인덱스를 활용할 수 없다. 필요한 컬럼만 명시적으로 SELECT하는 습관이 중요하다.

</div>

## 카디널리티와 선택도

<div class="concept-box" markdown="1">

**카디널리티(Cardinality)**: 해당 컬럼의 고유 값 개수.
**선택도(Selectivity)**: 카디널리티 / 전체 행 수. ==선택도가 1에 가까울수록 인덱스 효과가 크다.==

</div>

| 컬럼 | 전체 행 수 | 카디널리티 | 선택도 | 인덱스 효과 |
|------|-----------|-----------|--------|------------|
| email (UNIQUE) | 1,000,000 | 1,000,000 | 1.0 | 매우 높음 |
| created_at | 1,000,000 | 365,000 | 0.365 | 높음 |
| gender (M/F) | 1,000,000 | 2 | 0.000002 | 거의 없음 |

```sql
-- 카디널리티 확인
SHOW INDEX FROM member;
-- Cardinality 컬럼에 추정값이 표시된다 (InnoDB는 통계 기반 추정)

-- 통계 갱신
ANALYZE TABLE member;
```

<div class="warning-box" markdown="1">

**주의**: 인덱스로 조회 시 전체 행의 약 **5~10% 이하**만 읽을 때 인덱스가 효과적이다. 그 이상이면 옵티마이저가 오히려 Full Scan을 선택할 수 있다.

</div>

<div class="tip-box" markdown="1">

**팁**: 단일 컬럼의 카디널리티가 낮더라도 ==복합 인덱스로 조합하면 결합 카디널리티가 높아질 수 있다==. 예를 들어 `gender`(2종류)와 `age`(100종류)를 조합하면 최대 200종류까지 올라간다.

</div>

## 언제 쓰는지

인덱스를 걸면 좋은 컬럼:

- `WHERE` 조건에 자주 등장하는 컬럼
- `JOIN` 조건에 쓰이는 컬럼 (FK 컬럼)
- `ORDER BY`, `GROUP BY` 에 쓰이는 컬럼
- **카디널리티(Cardinality)가 높은 컬럼** — 값의 종류가 많을수록 효과적

<div class="warning-box" markdown="1">

**주의**: `성별(M/F)`, `상태(Y/N)` 같이 값의 종류가 2~3개뿐인 컬럼은 인덱스 효과가 거의 없다. 전체 행의 20~30% 이상을 읽어야 한다면 Full Scan이 더 빠를 수 있다.

</div>

## 사용 방법

### 인덱스 생성

```sql
-- 단일 컬럼 인덱스
CREATE INDEX idx_member_email ON member (email);

-- 복합 인덱스 (다중 컬럼)
CREATE INDEX idx_order_member_date ON orders (member_id, created_at);

-- UNIQUE 인덱스
CREATE UNIQUE INDEX uq_member_email ON member (email);
```

### 인덱스 조회·삭제

```sql
-- MySQL
SHOW INDEX FROM member;
DROP INDEX idx_member_email ON member;

-- PostgreSQL
\d member           -- 인덱스 포함 테이블 정보
DROP INDEX idx_member_email;
```

### 실행 계획 확인 (EXPLAIN)

```sql
EXPLAIN SELECT * FROM member WHERE email = 'test@test.com';
```

EXPLAIN 결과에서 `type` 컬럼은 인덱스 사용 방식을 나타낸다. <span class="text-green">위에 있을수록 좋다.</span>

| type | 설명 | 예시 |
|------|------|------|
| `const` | PK 또는 UNIQUE로 **1건** 조회. 상수 취급 | `WHERE id = 1` |
| `eq_ref` | JOIN에서 PK/UNIQUE로 정확히 1건 매칭 | `JOIN member m ON m.id = o.member_id` |
| `ref` | 논유니크 인덱스로 여러 건 매칭 | `WHERE member_id = 1` (FK 인덱스) |
| `range` | 인덱스 범위 스캔 | `WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'` |
| `index` | 인덱스 전체 스캔 (Full Index Scan) | `SELECT member_id FROM orders` (커버링이지만 전체 스캔) |
| `ALL` | 테이블 전체 스캔 (Full Table Scan) | 인덱스 미사용 |

<div class="danger-box" markdown="1">

**위험**: `type`이 `ALL`이면 Full Table Scan이다. 반드시 인덱스 추가 또는 쿼리 개선을 검토해야 한다. `index`도 인덱스 전체를 읽는 것이므로 대용량 테이블에서는 느릴 수 있다.

</div>

<div class="tip-box" markdown="1">

**팁**: `Extra` 컬럼도 함께 확인하자.

| Extra 값 | 의미 |
|----------|------|
| `Using index` | 커버링 인덱스 사용 (테이블 접근 없음) |
| `Using where` | 인덱스로 찾은 후 추가 필터링 |
| `Using filesort` | 정렬에 인덱스를 사용하지 못해 별도 정렬 수행 |
| `Using temporary` | 임시 테이블 생성 (GROUP BY, DISTINCT 등) |

</div>

## 복합 인덱스 설계

### 선두 컬럼 규칙

```sql
-- idx_order_member_date: (member_id, created_at) 순서로 생성
-- ✅ 사용됨
WHERE member_id = 1
WHERE member_id = 1 AND created_at > '2024-01-01'

-- ❌ 사용 안 됨 (선두 컬럼 없음)
WHERE created_at > '2024-01-01'
```

복합 인덱스는 **왼쪽부터 순서대로** 사용한다. 선두 컬럼(첫 번째 컬럼)이 `WHERE`에 없으면 인덱스를 타지 않는다.

### ESR 규칙 (Equality → Sort → Range)

<div class="concept-box" markdown="1">

**ESR 규칙**: 복합 인덱스의 컬럼 순서를 정하는 실전 규칙.
==**E**quality 조건 컬럼 → **S**ort(ORDER BY) 컬럼 → **R**ange 조건 컬럼== 순서로 배치한다.

</div>

- **Equality**(등호 `=`): 인덱스 노드를 정확히 좁힌다
- **Sort**(ORDER BY): 정렬된 순서로 읽으면 별도 정렬이 필요 없다
- **Range**(범위 `>`, `<`, `BETWEEN`): ==범위 조건 이후 컬럼은 인덱스 탐색이 중단==된다

```sql
-- 쿼리: 특정 회원의 주문을 최근 3개월 이내, 날짜 내림차순
SELECT * FROM orders
WHERE member_id = 1              -- Equality
  AND created_at > '2024-01-01'  -- Range
ORDER BY created_at DESC;        -- Sort

-- 최적 인덱스: (member_id, created_at)
-- member_id(E) → created_at이 Sort와 Range를 동시에 처리
```

### Range 이후 인덱스 중단

B-Tree에서 범위 조건이 적용되면, 그 이후 컬럼은 정렬 순서가 보장되지 않아 인덱스 탐색이 불가능하다.

<div class="compare-grid" markdown="1">
<div class="before" markdown="1">
**Bad — Range가 앞에 위치**

```sql
-- INDEX(age, name, created_at)
SELECT * FROM member
WHERE age > 20 AND name = '김철수';
```

`age`에서 Range가 걸려 `name`은 인덱스를 사용하지 못한다.

</div>
<div class="after" markdown="1">
**Good — Equality가 앞에 위치**

```sql
-- INDEX(name, age, created_at)
SELECT * FROM member
WHERE name = '김철수' AND age > 20;
```

`name`(Equality)이 선두이므로 정확히 좁힌 후 `age`(Range)를 처리한다.

</div>
</div>

<div class="warning-box" markdown="1">

**주의**: `IN` 연산자는 MySQL 옵티마이저가 등호 조건 목록으로 처리하므로 ==Range가 아닌 Equality처럼 동작==한다. `BETWEEN`, `>`, `<`와 다르다.

```sql
-- IN은 Equality처럼 동작 → 이후 컬럼도 인덱스 사용 가능
WHERE status IN ('ACTIVE', 'PENDING') AND created_at > '2024-01-01'
```

</div>

## 장점

- `WHERE`, `JOIN`, `ORDER BY` 쿼리 속도 대폭 향상
- 대용량 테이블에서 Full Scan 방지

## 단점

- **쓰기 성능 저하**: `INSERT`, `UPDATE`, `DELETE` 시 인덱스도 함께 갱신해야 함
- **저장 공간 추가**: 인덱스 자체가 별도 공간을 차지함
- 인덱스가 너무 많으면 오히려 성능 역효과

## 주의할 점

<div class="danger-box" markdown="1">

**인덱스를 무력화하는 패턴** — 인덱스가 있어도 사용되지 않는 경우:

```sql
-- ❌ 컬럼에 함수 적용
WHERE YEAR(created_at) = 2024
-- ✅ 대신
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'

-- ❌ 앞쪽 LIKE 와일드카드
WHERE name LIKE '%철수'
-- ✅ 뒤쪽 와일드카드는 OK
WHERE name LIKE '김%'

-- ❌ 묵시적 형변환
WHERE member_id = '100'  -- member_id가 INT인데 문자열 비교

-- ❌ NOT / 부정 조건
WHERE status != 'DELETED'
WHERE status NOT IN ('DELETED', 'EXPIRED')
-- ✅ 긍정 조건으로 전환
WHERE status IN ('ACTIVE', 'PENDING')

-- ❌ OR 조건 (서로 다른 컬럼)
WHERE name = '김철수' OR age > 30
-- 각 컬럼이 다른 인덱스를 사용해야 하므로 Full Scan 가능
-- ✅ UNION ALL로 분리
SELECT * FROM member WHERE name = '김철수'
UNION ALL
SELECT * FROM member WHERE age > 30 AND name != '김철수';
```

</div>

<div class="tip-box" markdown="1">

**팁**: MySQL InnoDB에서 `IS NULL`은 인덱스를 사용할 수 있다(`ref_or_null`). 반면 `IS NOT NULL`은 대부분의 행을 조회하게 되어 옵티마이저가 Full Scan을 선택하는 경우가 많다.

```sql
WHERE email IS NULL      -- ✅ 인덱스 사용 가능
WHERE email IS NOT NULL  -- ⚠️ 대부분 행 조회 → Full Scan 가능
```

</div>
