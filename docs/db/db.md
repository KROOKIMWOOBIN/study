# DB (Database)

데이터베이스 핵심 개념 정리.

<div class="concept-box" markdown="1">

**읽는 목표**: DB편은 SQL 문법만 외우는 문서가 아니라, 애플리케이션이 데이터를 저장·조회·변경할 때 DB가 어떤 규칙으로 안전하게 처리하는지 이해하기 위한 문서다.

</div>

---

## 먼저 잡고 가는 큰 그림

DB를 처음 보면 `SQL`, `트랜잭션`, `인덱스`, `정규화`가 따로 노는 개념처럼 보인다. 실제로는 아래 흐름으로 이어진다.

```text
데이터를 어디에 저장할까?
  -> 테이블, 컬럼, 데이터 타입, 제약 조건

데이터를 어떻게 읽고 바꿀까?
  -> SQL, JOIN, GROUP BY, 서브쿼리, CASE

데이터를 안전하게 바꿀 수 있을까?
  -> 트랜잭션, ACID, 격리 수준

데이터가 많아져도 빠르게 찾을 수 있을까?
  -> 인덱스, 실행 계획, 쿼리 튜닝
```

<div class="tip-box" markdown="1">

**신입 개발자 기준 핵심 질문**

- 테이블을 만들 때 어떤 컬럼과 제약 조건을 둬야 하는가?
- `SELECT` 결과가 왜 예상보다 많거나 적게 나오는가?
- `UPDATE`, `DELETE`를 안전하게 실행하려면 무엇을 확인해야 하는가?
- 데이터가 늘어났을 때 어떤 쿼리가 느려지는가?
- 여러 요청이 동시에 들어오면 데이터 정합성은 어떻게 지켜지는가?

</div>

## 신입이 자주 헷갈리는 구분

| 헷갈리는 개념 | 구분 기준 |
|---------------|-----------|
| DB vs DBMS | DB는 저장된 데이터 묶음이고, DBMS는 그 데이터를 관리하는 소프트웨어다. |
| 스키마 vs 데이터 | 스키마는 테이블 구조이고, 데이터는 실제 저장된 행이다. |
| PK vs FK | PK는 내 테이블의 행 식별자이고, FK는 다른 테이블의 행을 참조하는 값이다. |
| WHERE vs HAVING | `WHERE`는 그룹화 전 행 필터링, `HAVING`은 그룹화 후 그룹 필터링이다. |
| JOIN vs 서브쿼리 | JOIN은 테이블을 옆으로 붙여 조회하고, 서브쿼리는 쿼리 안에 다른 조회를 넣는다. |
| DELETE vs TRUNCATE | `DELETE`는 행을 삭제하는 DML, `TRUNCATE`는 테이블 데이터를 비우는 DDL 성격이 강하다. |
| 트랜잭션 vs 격리 수준 | 트랜잭션은 작업 묶음이고, 격리 수준은 동시에 실행될 때 서로를 얼마나 분리할지 정하는 정책이다. |
| 인덱스 vs 제약 조건 | 인덱스는 빠른 탐색을 위한 구조이고, 제약 조건은 데이터 무결성을 지키는 규칙이다. |
