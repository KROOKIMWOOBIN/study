# Kafka 베스트 프랙티스

Kafka 베스트 프랙티스의 핵심은 **유실보다 중복을 견디는 설계, partition 기준 순서 이해, topic 운영 기준 명확화, lag와 DLQ 관찰**입니다.

## 전체 기준

| 영역 | 권장 방식 |
|------|-----------|
| Topic | owner, retention, partition, RF, min ISR을 문서화 |
| Producer | 중요한 이벤트는 `acks=all`, idempotence, retry 설정 |
| Event | `eventId`, `eventType`, `schemaVersion`, `aggregateId` 포함 |
| Key | 순서가 필요한 aggregate 기준으로 선택 |
| Consumer | 처리 성공 후 offset commit, 멱등성 확보 |
| Error | retry 횟수 제한, DLQ, 재처리 절차 |
| Schema | optional 추가 중심, 호환성 테스트 |
| 운영 | partition별 lag, ISR, disk, DLQ 알림 |
| 보안 | 최소 권한 ACL, 민감정보 최소화 |

## Producer

| 권장 방식 | 이유 |
|-----------|------|
| `acks=all` 사용 검토 | leader 단독 저장 후 유실 위험 감소 |
| `enable.idempotence=true` | producer 재시도 중 log 중복 감소 |
| key를 명시적으로 설정 | 순서와 partition 분산 제어 |
| send 실패를 무시하지 않기 | 이벤트 유실 방지 |
| outbox pattern 검토 | DB 저장과 이벤트 발행 불일치 완화 |
| batch와 compression은 측정 후 조정 | 처리량과 지연 균형 |

## Event

| 권장 방식 | 이유 |
|-----------|------|
| `eventId` 포함 | 중복 처리 방지와 추적 |
| `eventType` 포함 | consumer가 의미를 판단 |
| `schemaVersion` 포함 | schema evolution 대응 |
| `occurredAt` 포함 | 지연과 순서 분석 |
| `aggregateId` 포함 | key와 추적 기준 |
| payload 의미를 명확히 함 | consumer 오해 방지 |

```json
{
  "eventId": "evt-20260425-000001",
  "eventType": "ORDER_CREATED",
  "schemaVersion": 1,
  "occurredAt": "2026-04-25T10:15:00Z",
  "aggregateId": "order-1001",
  "payload": {}
}
```

## Consumer

| 권장 방식 | 이유 |
|-----------|------|
| 처리 후 offset commit | 처리 전 commit으로 인한 유실 방지 |
| business 로직을 멱등하게 작성 | 중복 소비 대비 |
| DB unique 제약 사용 | 중복 반영 차단 |
| 상태 전이 검증 | 순서 지연과 중복에 안전 |
| 긴 작업은 poll interval 고려 | rebalance 방지 |
| DLQ와 재처리 도구 준비 | poison pill 격리 |

## 운영

| 권장 방식 | 이유 |
|-----------|------|
| RF 3, min ISR 2 검토 | broker 한 대 장애 대비 |
| retention은 업무 복구 시간 기준 | replay 가능 기간 확보 |
| partition별 lag 관찰 | hot partition 탐지 |
| DLQ 증가 알림 | schema/poison pill 조기 대응 |
| disk 사용률 알림 | broker 장애 예방 |
| offset reset은 dry-run 우선 | 대량 재처리 사고 방지 |
| topic 변경 승인 절차 | retention/partition 변경 사고 방지 |

## 보안

| 권장 방식 | 이유 |
|-----------|------|
| topic별 최소 권한 | 불필요한 read/write 방지 |
| 운영 계정 분리 | audit와 사고 추적 |
| 민감정보 payload 최소화 | 보관과 노출 위험 감소 |
| DLQ도 보안 대상에 포함 | 실패 메시지에 민감정보가 남을 수 있음 |
| 인증 정보 secret 관리 | credential 유출 방지 |

## Bad / Good

<div class="compare-grid" markdown="1">
<div class="before" markdown="1">
**Bad - 중복을 고려하지 않는 consumer**

```java
@KafkaListener(topics = "order-events")
void consume(OrderCreatedEvent event) {
    inventoryService.decrease(event.orderId());
}
```
</div>
<div class="after" markdown="1">
**Good - eventId로 멱등 처리**

```java
@KafkaListener(topics = "order-events")
void consume(OrderCreatedEvent event, Acknowledgment ack) {
    inventoryService.applyOnce(event.eventId(), event);
    ack.acknowledge();
}
```
</div>
</div>

## 언제 적용하는지

| 상황 | 우선 적용 |
|------|-----------|
| 핵심 업무 이벤트 | `acks=all`, idempotence, outbox, 멱등 consumer |
| 알림 이벤트 | 중복 발송 방지 key, retry, DLQ |
| 로그 수집 | batch, compression, partition 분산 |
| 재처리 요구 | retention, offset reset 절차 |
| 민감정보 포함 | ACL, encryption, payload 최소화 |
| schema 변경 잦음 | compatibility test, deprecated 기간 |

## 장점

| 장점 | 설명 |
|------|------|
| 장애 복구 쉬움 | DLQ, replay, 멱등성으로 복구 가능 |
| 유실 위험 감소 | producer와 broker 설정 기준 명확 |
| 운영 예측 가능 | topic 기준과 알림 기준이 있음 |
| 팀 간 협업 쉬움 | event schema와 owner가 명확 |

## 단점

| 단점 | 설명 |
|------|------|
| 초기 설계 비용 | event envelope, topic 기준, DLQ 준비 필요 |
| 코드 복잡도 증가 | 멱등성, 보정 로직, 상태 전이 검증 필요 |
| 운영 비용 증가 | 모니터링, ACL, retention 관리 필요 |
| 성능 trade-off | 안전한 설정은 지연이나 비용을 늘릴 수 있음 |

## 특징

| 특징 | 설명 |
|------|------|
| 실패를 기본 시나리오로 둠 | 중복, 지연, 재처리를 정상 흐름으로 고려 |
| 업무 기준 설계 | topic, key, retention을 기술이 아니라 업무 기준으로 정함 |
| 운영과 개발 연결 | eventId, DLQ, metrics가 코드와 운영을 이어줌 |
| 작은 변경도 계약 | event field 변경은 consumer 계약 변경 |

## 주의할 점

| 주의 | 설명 |
|------|------|
| "exactly once"를 과신하지 않기 | 외부 DB/API는 별도 멱등성 필요 |
| DLQ를 쓰레기통으로 방치하지 않기 | 재처리 owner와 절차 필요 |
| partition 증설을 가볍게 보지 않기 | key 배치와 순서 영향 |
| offset reset을 운영 중 즉흥 실행 금지 | dry-run과 승인 필요 |
| 모든 이벤트를 하나의 topic에 넣지 않기 | 권한, schema, retention 관리 어려움 |
| consumer group id 변경 주의 | 새 group처럼 다시 읽을 수 있음 |

## 실무 체크리스트

| 체크 | 질문 |
|------|------|
| Topic | owner, retention, partition, RF가 정해졌는가 |
| Producer | 실패 시 재시도와 유실 대응이 있는가 |
| Event | `eventId`, `schemaVersion`, key 기준이 있는가 |
| Consumer | 중복 처리와 재처리에 안전한가 |
| Error | DLQ와 재처리 owner가 있는가 |
| Schema | 호환성 규칙과 테스트가 있는가 |
| Monitoring | lag, DLQ, ISR, disk 알림이 있는가 |
| Security | ACL과 민감정보 기준이 있는가 |

## 정리

| 항목 | 설명 |
|------|------|
| 핵심 원칙 | 유실보다 중복을 견디게 설계 |
| producer 기준 | 안전한 발행과 추적 가능성 |
| consumer 기준 | 처리 후 commit과 멱등성 |
| 운영 기준 | lag, DLQ, ISR, disk를 관찰 |
| 실무 기준 | Kafka 도입은 코드와 운영 절차를 함께 설계하는 일 |
