# Kafka 비교와 선택 기준

Kafka는 강력하지만 모든 비동기 문제의 답은 아닙니다. 선택 기준은 **이벤트를 얼마나 오래 보관해야 하는지, 여러 consumer가 다시 읽어야 하는지, 처리량이 얼마나 큰지, 운영 복잡도를 감당할 수 있는지**입니다.

## Kafka vs RabbitMQ

| 기준 | Kafka | RabbitMQ |
|------|-------|----------|
| 중심 모델 | append-only event log | exchange와 queue 기반 broker |
| 메시지 보관 | retention 기반 장기 보관 가능 | 소비 후 제거되는 queue 모델 중심 |
| 재처리 | offset 조정으로 유리 | 별도 저장/재발행 필요 |
| 라우팅 | topic과 consumer group 중심 | exchange routing이 강점 |
| 처리량 | 대량 스트림에 강함 | 작업 큐와 라우팅에 강함 |
| 적합 | 이벤트 로그, 로그 수집, replay | 명령성 작업 큐, 복잡한 라우팅 |

## Kafka vs Redis Stream

| 기준 | Kafka | Redis Stream |
|------|-------|--------------|
| 저장 기반 | disk 기반 분산 log | Redis 메모리 중심 stream |
| 장기 보관 | 상대적으로 강함 | 메모리 비용과 Redis 운영 영향 |
| 처리량 확장 | partition과 broker 확장 | Redis cluster 설계 영향 |
| 운영 목적 | 서비스 간 핵심 이벤트 파이프라인 | Redis 내부 경량 작업 큐 |
| 재처리 | retention과 offset 기반 | stream trim 정책 영향 |
| 적합 | 대규모 이벤트 스트리밍 | 작은 규모의 빠른 내부 이벤트 |

## Kafka vs Redis Pub/Sub

| 기준 | Kafka | Redis Pub/Sub |
|------|-------|---------------|
| 메시지 보관 | 보관함 | 보관하지 않음 |
| consumer 장애 | 나중에 다시 읽기 가능 | 구독 중이 아니면 놓침 |
| 재처리 | 가능 | 어려움 |
| 지연 | 낮지만 log 저장 비용 있음 | 매우 단순하고 빠름 |
| 적합 | 유실되면 안 되는 이벤트 | 실시간 신호, 캐시 무효화 알림 |

## Kafka vs SQS

| 기준 | Kafka | SQS |
|------|-------|-----|
| 운영 방식 | 직접 cluster 또는 managed Kafka 운영 | AWS managed queue |
| 소비 모델 | log와 consumer group | queue message 처리 |
| 재처리 | offset replay | visibility timeout, DLQ 중심 |
| 처리량 | 대량 스트림과 다중 소비에 강함 | 단순 queue 처리에 편함 |
| 운영 부담 | 상대적으로 큼 | 낮음 |
| 적합 | event streaming platform | 단순 비동기 작업 큐 |

## Kafka vs DB Polling

| 기준 | Kafka | DB Polling |
|------|-------|------------|
| 변경 전파 | 이벤트 기반 | 주기적 조회 |
| 지연 | 낮게 유지 가능 | polling 주기에 영향 |
| DB 부하 | producer/consumer 분리 | polling query 부하 |
| 재처리 | retention 기반 | DB 이력 설계 필요 |
| 구현 난도 | Kafka 운영 필요 | 초기 구현 단순 |
| 적합 | 여러 consumer, 대량 이벤트 | 작은 시스템, 단순 배치 |

## Kafka를 쓰면 좋은 경우

| 상황 | 이유 |
|------|------|
| 여러 consumer group이 같은 이벤트를 각자 처리 | Kafka의 독립 offset이 강점 |
| 장애 후 재처리와 replay가 중요 | retention 기반 재처리 |
| 초당 많은 이벤트를 안정적으로 수집 | partition과 batch 기반 처리량 |
| producer와 consumer 장애를 분리해야 함 | 중간 log가 buffer 역할 |
| 이벤트 기반 아키텍처를 구축 | 서비스 간 결합 완화 |

## Kafka를 쓰지 않는 편이 좋은 경우

| 상황 | 대안 |
|------|------|
| 단순 background job 몇 개 | SQS, RabbitMQ, scheduler |
| 짧은 캐시 무효화 신호 | Redis Pub/Sub |
| 임시 상태 저장 | Redis |
| 즉시 조회가 필요한 데이터 | RDBMS, Redis, 검색 엔진 |
| 운영 인력이 부족한 작은 서비스 | managed queue 또는 managed Kafka 검토 |
| 강한 동기 transaction이 핵심 | DB transaction 중심 설계 |

## 장점

| 장점 | 설명 |
|------|------|
| replay 가능 | offset으로 과거 이벤트 재처리 |
| 다중 소비 | 같은 이벤트를 여러 group이 독립 사용 |
| 높은 처리량 | batch와 partition 기반 |
| 장애 흡수 | consumer 지연을 log가 완충 |
| event sourcing/streaming 친화 | 이벤트 중심 아키텍처에 적합 |

## 단점

| 단점 | 설명 |
|------|------|
| 운영 복잡도 | cluster, lag, schema, 보안 관리 |
| 작은 문제에는 과함 | 단순 큐보다 학습과 비용이 큼 |
| 최종 일관성 | 동기 transaction처럼 즉시 일치하지 않음 |
| consumer 책임 증가 | 멱등성, DLQ, 재처리 필요 |

## 특징

| 특징 | 설명 |
|------|------|
| log 중심 | 메시지를 보관하고 offset으로 읽음 |
| consumer group | group별로 독립 소비 |
| partition | 처리량과 순서 보장의 핵심 단위 |
| retention | 소비 여부와 삭제 시점이 분리 |
| 운영형 플랫폼 | 단순 라이브러리보다 인프라 성격이 강함 |

## 주의할 점

| 주의 | 설명 |
|------|------|
| "비동기니까 Kafka"로 결정하지 않기 | 보관, replay, 다중 소비가 필요한지 확인 |
| RabbitMQ/SQS와 queue 요구를 비교 | 작업 분배만 필요하면 queue가 더 단순 |
| Redis Stream과 규모를 비교 | Redis 내부 경량 이벤트면 Stream도 후보 |
| managed 서비스 여부 검토 | 운영 역량과 비용을 같이 봐야 함 |
| 장애 책임을 consumer까지 포함 | Kafka 도입이 멱등성과 DLQ를 대신하지 않음 |

## 베스트 프랙티스

| 권장 방식 | 이유 |
|-----------|------|
| replay 필요 여부를 먼저 묻기 | Kafka 선택의 핵심 기준 |
| consumer group이 여러 개인지 확인 | 다중 소비 장점 활용 여부 |
| 처리량과 retention을 수치로 추정 | partition과 비용 산정 |
| 운영 담당과 알림 기준을 정함 | Kafka는 운영형 인프라 |
| 작은 문제에는 작은 도구 선택 | 복잡도 비용 절감 |

## 실무 선택표

| 요구사항 | 추천 |
|----------|------|
| 주문 이벤트를 재고/알림/검색/통계가 각자 소비 | Kafka |
| 이메일 발송 작업을 queue로 분배 | RabbitMQ, SQS, Kafka 모두 후보지만 단순 큐 우선 검토 |
| 캐시 무효화 신호를 실시간 broadcast | Redis Pub/Sub |
| Redis 내부에서 작은 작업 stream 처리 | Redis Stream |
| 클릭 로그 대량 수집 후 분석 저장소 적재 | Kafka |
| DB 변경을 여러 시스템에 전달 | CDC + Kafka 검토 |

## 정리

| 항목 | 설명 |
|------|------|
| Kafka 선택 기준 | 보관, replay, 다중 소비, 대량 처리 |
| Kafka가 강한 곳 | 이벤트 스트림과 재처리 |
| Kafka가 과한 곳 | 단순 작업 큐, 짧은 신호, 임시 상태 |
| 실무 기준 | 문제 크기와 운영 비용을 함께 비교 |
