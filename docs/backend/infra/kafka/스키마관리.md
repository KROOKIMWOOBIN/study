# Kafka Schema 관리와 이벤트 버전

Kafka 이벤트는 여러 consumer가 각자 다른 시점에 읽습니다. producer가 필드를 바꾸면 당장 배포되지 않은 consumer가 깨질 수 있으므로, 이벤트 schema와 version 관리가 필요합니다.

<div class="concept-box" markdown="1">

**Schema 관리**는 이벤트 payload의 필드, 타입, 필수 여부, 호환성 규칙을 정하고 변경 이력을 관리하는 일입니다. Kafka에서는 producer보다 consumer 호환성을 먼저 생각해야 합니다.

</div>

## 왜 쓰는지

Kafka 이벤트는 한 번 발행되면 retention 기간 동안 남고, 여러 consumer group이 독립적으로 읽습니다. 즉시 같이 배포되는 API 요청보다 변경 영향이 넓습니다.

```text
Producer는 오늘 새 필드를 보낸다.
Consumer A는 오늘 배포됐다.
Consumer B는 다음 주에 배포된다.
Consumer C는 장애 복구 중 과거 이벤트를 다시 읽는다.
```

이 상황에서 schema 규칙이 없으면 특정 consumer만 실패하거나, 과거 이벤트 재처리가 깨질 수 있습니다.

## 어떻게 쓰는지

### 이벤트 envelope 만들기

업무 payload만 보내지 말고, 추적과 호환성 관리를 위한 공통 필드를 둡니다.

```json
{
  "eventId": "evt-20260425-000001",
  "eventType": "ORDER_CREATED",
  "schemaVersion": 2,
  "occurredAt": "2026-04-25T10:15:00Z",
  "aggregateType": "ORDER",
  "aggregateId": "order-1001",
  "producer": "order-service",
  "payload": {
    "orderId": 1001,
    "userId": 10,
    "amount": 39000,
    "currency": "KRW"
  }
}
```

| 필드 | 이유 |
|------|------|
| `eventId` | 중복 처리 방지와 추적 |
| `eventType` | consumer가 이벤트 의미를 판단 |
| `schemaVersion` | payload 구조 변경 대응 |
| `occurredAt` | 이벤트 발생 시각 |
| `aggregateId` | 순서 보장 key와 추적 기준 |
| `producer` | 장애 원인 추적 |

### 호환성 규칙 정하기

| 변경 | 안전성 | 설명 |
|------|--------|------|
| optional 필드 추가 | 비교적 안전 | 기존 consumer가 무시 가능 |
| 필수 필드 추가 | 위험 | 과거 이벤트와 기존 consumer가 값 없음 |
| 필드 삭제 | 위험 | 기존 consumer가 읽던 필드가 사라짐 |
| 필드 이름 변경 | 위험 | 삭제 + 추가와 같음 |
| 타입 변경 | 위험 | 역직렬화나 비즈니스 로직 실패 |
| enum 값 추가 | 주의 | consumer의 default 처리 필요 |
| 의미 변경 | 매우 위험 | schema는 같아도 업무 결과가 달라짐 |

## 형식 선택

| 형식 | 장점 | 단점 | 적합한 경우 |
|------|------|------|-------------|
| JSON | 읽기 쉽고 도입이 쉬움 | 타입 안정성과 schema 검증이 약함 | 초기 학습, 단순 이벤트 |
| Avro | schema evolution과 압축 효율이 좋음 | Schema Registry 등 운영 요소 필요 | 대규모 이벤트 파이프라인 |
| Protobuf | 타입 안정성과 다국어 지원 | 필드 번호 관리 필요 | 여러 언어 서비스 연동 |
| String | 가장 단순 | 구조화와 호환성 관리가 어려움 | 로그성 단순 메시지 |

Schema Registry를 쓰면 producer와 consumer가 정해진 호환성 규칙을 기준으로 schema 변경을 검증할 수 있습니다. 다만 registry도 운영 구성요소이므로 장애와 권한을 함께 설계해야 합니다.

## 언제 version을 올리는지

| 상황 | 처리 |
|------|------|
| optional 필드 추가 | minor 수준 변경, 기존 version 유지 가능 |
| 필수 필드 추가 | 새 version 또는 새 event type 검토 |
| payload 의미 변경 | 새 event type 검토 |
| 기존 consumer가 모두 대응 완료 | deprecated 필드 제거 가능 |
| 과거 이벤트 재처리가 필요 | old version reader 유지 |

## 장점

| 장점 | 설명 |
|------|------|
| consumer 장애 예방 | 변경 전에 호환성 위험을 확인 |
| 재처리 안정성 | 과거 이벤트를 다시 읽어도 처리 가능 |
| 추적성 증가 | event type과 version으로 장애 범위 파악 |
| 팀 간 계약 명확화 | producer와 consumer가 같은 규칙을 공유 |

## 단점

| 단점 | 설명 |
|------|------|
| 초기 설계 비용 | envelope, version, 호환성 규칙이 필요 |
| 운영 구성 증가 | Schema Registry를 쓰면 관리 대상이 늘어남 |
| 변경 속도 저하 | consumer 호환성을 확인해야 함 |
| old version 유지 부담 | retention과 재처리 기간 동안 reader가 필요 |

## 특징

| 특징 | 설명 |
|------|------|
| 이벤트는 오래 남음 | retention 기간 동안 old schema가 살아 있음 |
| consumer는 독립 배포 | producer 변경 시점과 consumer 반영 시점이 다름 |
| 호환성은 양방향으로 봄 | 새 consumer가 old event를, old consumer가 new event를 읽을 수 있는지 확인 |
| 의미 변경도 schema 변경 | 타입이 같아도 의미가 바뀌면 장애가 될 수 있음 |

## 주의할 점

| 주의 | 설명 |
|------|------|
| 필드 삭제부터 하지 않기 | deprecated 기간을 둔 뒤 제거 |
| null 처리 방치 금지 | optional 필드는 default와 null 정책 필요 |
| enum default 처리 필요 | 모르는 값이 와도 consumer가 죽지 않게 함 |
| event type을 재사용하지 않기 | 의미가 바뀌면 새 event type이 안전 |
| schema version만 믿지 않기 | consumer의 실제 처리 로직 호환성도 테스트 |
| PII 포함 주의 | topic 권한과 retention에 개인정보가 묶임 |

## 베스트 프랙티스

| 권장 방식 | 이유 |
|-----------|------|
| 공통 envelope 사용 | 추적, 멱등성, 버전 관리가 쉬움 |
| optional 추가를 기본 변경 방식으로 둠 | 기존 consumer 영향 최소화 |
| 호환성 테스트를 배포 파이프라인에 포함 | 변경 사고 방지 |
| deprecated 필드를 일정 기간 유지 | 점진적 consumer 전환 |
| schema owner를 명확히 함 | 임의 변경 방지 |
| DLQ에 schema version을 남김 | 장애 분석과 재처리 용이 |

## 실무에서는?

| 상황 | 설계 기준 |
|------|-----------|
| 주문 이벤트에 쿠폰 정보 추가 | optional 필드 추가, consumer 무시 가능 확인 |
| 금액 타입을 number에서 object로 변경 | 새 필드 추가 후 점진 전환 |
| enum에 새 주문 상태 추가 | consumer default branch 확인 |
| 개인정보 필드 추가 | retention, ACL, 마스킹 정책 먼저 검토 |
| 오래된 이벤트 재처리 | version별 reader 유지 |

## 정리

| 항목 | 설명 |
|------|------|
| schema 관리 목적 | 여러 consumer와 과거 이벤트를 깨뜨리지 않기 |
| 핵심 필드 | `eventId`, `eventType`, `schemaVersion`, `aggregateId` |
| 안전한 변경 | optional 필드 추가 |
| 위험한 변경 | 필수 필드 추가, 삭제, 타입 변경, 의미 변경 |
| 실무 기준 | producer 편의보다 consumer 호환성을 먼저 본다 |
