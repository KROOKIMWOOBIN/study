    객체지향 프로그래밍 (OOP)

    왜 쓰는지

    프로그램이 커질수록 변수와 함수만으로는 "어떤 데이터가 어떤 규칙으로 변경되는지"를 추적하기 어려워집니다.

    • 주문은 주문 상태와 결제 규칙을 함께 가져야 함
    • 회원은 비밀번호 변경 규칙을 스스로 지켜야 함
    • 결제 수단이 늘어나도 주문 로직은 크게 바뀌지 않아야 함

    핵심: 객체지향 프로그래밍은 데이터와 그 데이터를 다루는 책임을 객체 안에 묶고, 객체들이 메시지를 주고받으며 문제를 해결하는 방식입니다.

    핵심 원칙

    원칙 한 줄 설명 먼저 볼 문서
    캡슐화 상태를 숨기고 메서드로만 안전하게 변경한다 캡슐화
    상속 is-a 관계로 공통 개념을 표현한다 상속
    추상화 구현보다 역할과 계약에 집중한다 추상화
    다형성 같은 호출이 실제 객체에 따라 다르게 동작한다 다형성

    어떻게 쓰는지

    객체지향 설계는 보통 상태를 가진 대상을 찾는 것보다, 먼저 책임을 찾는 식으로 접근합니다.

    public class Order {
    private OrderStatus status = OrderStatus.CREATED;
    private int totalPrice;

    public Order(int totalPrice) {
        if (totalPrice <= 0) {
            throw new IllegalArgumentException("주문 금액은 0보다 커야 합니다");
        }
        this.totalPrice = totalPrice;
    }

    public void pay(PaymentMethod paymentMethod) {
        if (status != OrderStatus.CREATED) {
            throw new IllegalStateException("이미 처리된 주문입니다");
        }

        paymentMethod.pay(totalPrice);
        status = OrderStatus.PAID;
    }
}

public interface PaymentMethod {
    void pay(int amount);
}

enum OrderStatus {
    CREATED, PAID
}

    이 예시에서 Order는 자신의 상태 변경 규칙을 직접 지키고, PaymentMethod라는 추상화에 의존합니다. 그래서 카드 결제, 계좌이체, 간편결제가 늘어나도 주문의 핵심 흐름은 크게 흔들리지 않습니다.

    언제 쓰는지

    상황 OOP가 잘 맞는 이유
    상태와 규칙이 함께 움직임 상태 변경을 객체 안에서 검증할 수 있음
    동일한 역할의 구현이 여러 개 있음 인터페이스와 다형성으로 교체 가능
    도메인 규칙이 중요함 객체가 자신의 책임을 가지게 만들 수 있음
    장기 유지보수가 필요함 역할별로 변경 범위를 줄일 수 있음
    단순 스크립트성 로직 OOP가 오히려 과한 구조가 될 수 있음

    장점

    장점 설명
    변경 범위 축소 책임이 분리되어 수정 위치가 명확해짐
    재사용성 역할과 구현을 분리하면 구현체 교체가 쉬움
    테스트 용이성 객체 단위로 규칙을 검증하기 좋음
    도메인 표현력 코드가 업무 용어와 가까워짐

    단점

    단점 설명
    과설계 위험 단순한 기능에도 클래스와 인터페이스가 과하게 늘 수 있음
    추적 비용 추상화가 많으면 실제 실행 구현을 따라가기 어려움
    잘못된 책임 분리 객체가 데이터만 들고 로직은 흩어지는 구조가 되기 쉬움
    상속 오남용 부모-자식 결합이 강해져 수정이 어려워질 수 있음

    특징

    클래스와 객체

    // 클래스: 객체를 만들기 위한 설계도
public class Member {
    private String name;

    public Member(String name) {
        this.name = name;
    }
}

// 객체: 클래스로부터 생성된 실제 인스턴스
Member member = new Member("kim");

    책임 중심 사고

    // 좋지 않은 흐름: 외부에서 객체 상태를 꺼내 규칙을 처리
if (order.getStatus() == OrderStatus.CREATED) {
    order.setStatus(OrderStatus.PAID);
}

// 권장 흐름: 객체에게 행동을 요청
order.pay(paymentMethod);

    객체지향에서 중요한 질문은 "어떤 데이터를 갖는가?"보다 "어떤 책임을 가져야 하는가?"입니다.

    주의할 점

    OOP는 클래스 많이 만들기가 아닙니다.

    클래스가 많아도 객체가 자기 책임을 갖지 못하면 절차지향 코드가 여러 파일로 흩어진 형태가 됩니다.

    Getter/Setter만 있는 객체는 도메인 객체가 아닐 수 있습니다.

    상태 변경 규칙이 외부 서비스에 흩어져 있으면 객체 내부의 상태를 안전하게 지키기 어렵습니다.

    베스트 프랙티스

    권장 방식 이유
    필드는 private으로 시작 객체 상태를 외부에서 마음대로 바꾸지 못하게 함
    Setter보다 의미 있는 메서드 사용 setStatus()보다 pay(), cancel()이 의도를 잘 표현함
    역할이 먼저, 구현은 나중 인터페이스와 다형성을 자연스럽게 적용할 수 있음
    상속보다 컴포지션 우선 검토 강한 결합을 줄이고 조합을 유연하게 만들 수 있음
    작은 객체부터 테스트 도메인 규칙이 깨지는지 빠르게 확인 가능

    실무에서는?

    위치 OOP 관점
    Domain/Entity 상태와 핵심 비즈니스 규칙을 함께 둔다
    Service 여러 객체의 흐름을 조율하되, 모든 규칙을 독점하지 않는다
    DTO 계층 간 데이터 전달 목적이므로 단순 구조여도 괜찮다
    Repository 저장소 접근 역할을 인터페이스로 숨긴다
    외부 API 연동 결제, 알림, 파일 저장소 등을 역할 인터페이스 뒤에 둔다
    테스트 코드 인터페이스를 Fake/Mock으로 대체해 흐름을 검증한다

    정리

    항목 설명
    OOP 핵심 객체가 상태와 책임을 함께 가진다
    좋은 객체 자기 상태를 스스로 보호하고 의미 있는 행동을 제공한다
    설계 기준 변경될 가능성이 큰 구현은 추상화 뒤로 숨긴다
    주의 상속, 인터페이스, 패턴을 무조건 늘리는 것은 과설계다
