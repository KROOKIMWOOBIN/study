    Kafka 모니터링과 보안

    Kafka 운영에서는 lag, broker health, producer error, consumer 처리 시간, 인증/권한을 함께 봐야 합니다. lag만 알림으로 걸어두면 원인이 producer인지 broker인지 consumer인지 늦게 알게 됩니다.

    왜 쓰는지

    Kafka는 여러 서비스가 공유하는 이벤트 인프라입니다. 장애가 나면 producer 발행 지연, consumer lag, DB 반영 지연, 알림 누락처럼 여러 증상으로 나타납니다. 모니터링은 증상을 빠르게 나누기 위해 필요하고, 보안은 topic에 쌓이는 민감 이벤트를 보호하기 위해 필요합니다.

    모니터링 지표

    Broker

    지표 의미 위험 신호
    UnderReplicatedPartitions 복제가 부족한 partition broker 장애, follower 지연
    OfflinePartitionsCount leader가 없는 partition 읽기/쓰기 불가
    request latency produce/fetch 요청 지연 broker 병목
    request handler idle 요청 처리 thread 여유 낮으면 처리 여유 부족
    network processor idle network 처리 여유 낮으면 network 병목
    disk usage log 저장 공간 full 근접 시 장애 위험

    Producer

    지표 의미
    record send rate 발행량
    record error rate 발행 실패
    record retry rate 재시도
    request latency p95/p99 broker 응답 지연
    buffer available bytes producer buffer 여유
    batch size avg batch 효율

    Consumer

    지표 의미
    records consumed rate 소비량
    records lag max 최대 lag
    commit latency offset commit 지연
    rebalance count group 안정성
    poll duration poll loop 처리 시간
    processing time 업무 로직 처리 시간

    Lag는 원인이 아니라 결과입니다.

    lag = LOG-END-OFFSET - CURRENT-OFFSET

    한 partition만 lag가 높으면 hot partition이나 poison pill을 의심하고, 전체 partition lag가 높으면 consumer 처리량이나 sink DB 병목을 봅니다.

    알림 기준

    알림 기준
    broker offline 즉시 대응
    offline partition > 0 즉시 대응
    under replicated partition 증가 장애 전조
    disk 사용률 높음 retention, 증설 검토
    producer error rate 증가 발행 실패 확인
    consumer lag 증가 업무 지연 기준으로 판단
    rebalance 반복 consumer 안정성 확인
    DLQ 증가 schema, poison pill, 외부 API 장애 확인

    보안 요소

    요소 설명
    인증 client가 Kafka에 접속할 때 신원 확인
    권한 topic별 read/write/create 권한 제어
    암호화 broker-client, broker-broker 통신 보호
    ACL principal별 topic 접근 제어
    Secret 관리 username/password, keytab, certificate 관리
    Audit 누가 어떤 topic을 읽고 썼는지 추적 
    producer는 자기 topic에 write만 가능
consumer는 필요한 topic에 read만 가능
운영 명령은 별도 admin principal만 가능

    개인정보와 민감정보

    Kafka topic은 여러 consumer가 읽을 수 있고 retention 기간 동안 남습니다. 개인정보나 결제 정보가 들어가면 권한, 마스킹, 보관 기간, 삭제 요구 대응을 함께 설계해야 합니다.

    데이터 기준
    이메일, 전화번호 필요 최소화, 마스킹 검토
    결제 정보 직접 저장 지양, token/reference 사용
    사용자 식별자 권한과 retention 검토
    로그 payload request/response 전체 저장 주의
    DLQ 실패 메시지에도 민감정보가 남을 수 있음

    언제 무엇을 보는지

    증상 먼저 볼 지표
    이벤트 발행 지연 producer latency, error rate, broker produce latency
    consumer 반영 지연 partition별 lag, processing time, DB latency
    특정 consumer만 실패 deserialization error, DLQ, schema version
    broker 불안정 under replicated, offline partition, disk, controller
    보안 접속 실패 authentication error, ACL deny log
    특정 팀 topic 접근 이슈 principal, ACL, topic naming

    장점

    장점 설명
    장애 위치 분리 producer, broker, consumer, sink 중 어디가 문제인지 빠르게 확인
    장애 전조 탐지 ISR 감소, disk 증가, retry 증가를 조기 감지
    보안 사고 예방 topic 접근과 민감정보 노출 제한
    운영 신뢰성 증가 알림과 runbook으로 대응 속도 향상

    단점

    단점 설명
    지표 많음 핵심 지표를 정하지 않으면 잡음 증가
    lag 기준 어려움 업무별 허용 지연이 다름
    ACL 운영 비용 topic과 principal이 늘면 관리 복잡
    민감정보 관리 부담 payload와 DLQ까지 같이 봐야 함

    특징

    특징 설명
    group별 지표 필요 같은 topic도 consumer group마다 lag가 다름
    partition별 확인 중요 hot partition은 평균 lag로 숨을 수 있음
    DLQ도 운영 대상 실패 메시지 저장소도 보안과 retention 필요
    보안은 topic 설계와 연결 topic 경계가 권한 경계가 됨

    주의할 점

    주의 설명
    전체 lag 평균만 보지 않기 특정 partition 병목을 놓칠 수 있음
    topic 권한을 넓게 주지 않기 민감 이벤트 노출 위험
    DLQ retention 방치 금지 실패 메시지와 개인정보가 오래 남을 수 있음
    운영 계정 공유 금지 누가 offset reset을 했는지 추적 불가
    request/response 로그 전체 저장 주의 토큰과 개인정보가 섞일 수 있음
    알림 임계값을 업무와 맞추기 무의미한 알림 폭주 방지

    베스트 프랙티스

    권장 방식 이유
    producer/broker/consumer dashboard 분리 병목 위치 파악
    partition별 lag 알림 hot partition 탐지
    DLQ 증가 알림 poison pill 조기 대응
    topic owner와 ACL owner 지정 권한 관리 명확화
    최소 권한 원칙 적용 불필요한 read/write 차단
    민감정보는 reference 중심으로 전달 Kafka 보관 데이터 위험 감소
    운영 명령 audit offset reset, topic 변경 사고 추적

    실무에서는?

    운영 작업 체크
    신규 topic 생성 owner, ACL, retention, 민감정보 여부
    신규 consumer 배포 group id, lag 알림, DLQ
    장애 대응 broker 지표와 consumer 지표 분리
    보안 점검 unused principal 제거, ACL 검토
    개인정보 점검 payload, headers, DLQ 포함 확인
    용량 계획 topic별 log size와 retention 추세

    정리

    항목 설명
    모니터링 핵심 broker health, producer error, consumer lag, DLQ
    보안 핵심 인증, ACL, 암호화, 최소 권한
    가장 큰 주의점 lag만 보고 원인을 단정하지 않기
    실무 기준 topic 설계와 보안/모니터링을 같이 설계

    관련 파일: - 성능 최적화 - 장애 대응과 트러블슈팅 - 운영 구조와 고가용성
