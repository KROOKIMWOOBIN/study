# AGENTS.md

이 파일은 이 저장소에서 작업하는 모든 에이전트가 먼저 읽어야 하는 공통 작업 기준이다.

## 저장소 개요

- 이 저장소는 MkDocs 기반 학습 노트 저장소다.
- 문서는 `docs/` 아래에 주제별로 정리한다.
- 기존 문서의 구성, 표현 방식, 링크 구조를 우선 따른다.

## 학습 노트 정리 필수 항목

개념을 정리할 때 아래 항목을 반드시 고려한다.

| 항목 | 설명 |
|------|------|
| 왜 쓰는지 | 이 개념/기능이 존재하는 이유, 해결하는 문제 |
| 어떻게 쓰는지 | 실제 사용 방법, 코드 예시, 설정 방법 |
| 언제 쓰는지 | 적합한 사용 상황, 조건 |
| 장점 | 사용했을 때의 이점 |
| 단점 | 한계, 트레이드오프 |
| 특징 | 다른 것과 구별되는 핵심 성질 |
| 주의할 점 | 실수하기 쉬운 부분, 안티패턴 |
| 베스트 프랙티스 | 권장 사용 패턴, 잘 정착된 관례 |
| 실무에서는? | 실제 프로젝트에서 어떻게 활용되는지 |

모든 항목이 항상 별도 섹션일 필요는 없다. 테이블, 인라인 설명 등 형식은 자유롭게 사용하되 내용은 빠짐없이 담는다.

## 학습 노트 간결성 기준

- 카테고리 분리 이유, 다른 카테고리와의 관계 설명, "이 문서는 무엇을 다루고 저 문서는 무엇을 다룬다" 같은 메타 설명은 사용자가 요청한 경우에만 작성한다.
- 개요 문서는 장황한 설명보다 목차, 상황별 선택 기준, 핵심 링크 중심으로 짧게 작성한다.
- 학습 내용 자체가 아닌 운영 안내성 문단은 꼭 필요할 때만 남긴다.

## 기존 문서와 겹칠 때

새 주제가 기존 파일에서 이미 다루고 있는 내용과 겹치더라도, 별도 파일로 더 깊게 작성할 수 있다.

- 기존 파일: 개요 수준으로 언급
- 새 파일: 해당 주제만 집중해서 심화 정리

단, 사용자가 요청한 내용이 기존 파일에서 이미 다루고 있는 내용과 동일하거나 거의 같다면 새 파일을 만들지 말고 기존 파일을 고도화한다.

- 작성 전 반드시 관련 파일을 먼저 확인한다.
- 기존 내용이 얕거나 누락된 항목이 있으면 보강한다.
- 완전히 새로운 관점이나 심화 내용이면 기존 파일에 섹션을 추가한다.
- 불필요한 중복 파일은 만들지 않는다.

## DB 노트 작성 기준

`docs/db/` 경로의 모든 DB 관련 노트는 MySQL 기준으로 작성한다.

- SQL 문법, 함수, 동작 방식이 DBMS마다 다를 경우 MySQL 기준으로 먼저 설명한다.
- PostgreSQL, Oracle 등 다른 DBMS와 차이가 있는 경우 별도 언급할 수 있다.
- 예시 코드도 MySQL 방언 기준으로 작성한다. 예: `LIMIT`, `IFNULL`, `AUTO_INCREMENT`.
- 신입 개발자도 따라올 수 있도록 용어 정의, 사용 상황, 주의점을 함께 적는다.

## 스타일 표현 가이드

노트 작성 시 `docs/stylesheets/extra.css`와 `mkdocs.yml`의 Markdown 확장 기능을 활용할 수 있다.

### 인라인 텍스트 색상

```html
<span class="text-red">위험·금지·오류</span>
<span class="text-blue">핵심 개념·용어</span>
<span class="text-green">권장·정답</span>
<span class="text-orange">주의·경고</span>
<span class="text-purple">추가 정보·팁</span>
<span class="text-gray">보조 설명</span>
```

### 하이라이트

```markdown
==중요한 단어== 또는 ==핵심 문장==
```

### 개념·정보 박스

박스 안에서 Markdown 문법이 동작해야 하므로 `markdown="1"` 속성을 붙인다.

```markdown
<div class="concept-box" markdown="1">
**핵심**: 내용
</div>

<div class="warning-box" markdown="1">
**주의**: 내용
</div>

<div class="danger-box" markdown="1">
**위험**: 내용
</div>

<div class="success-box" markdown="1">
**권장**: 내용
</div>

<div class="tip-box" markdown="1">
**팁**: 내용
</div>
```

| 클래스 | 용도 |
|---|---|
| `concept-box` | 핵심 개념 정의 |
| `warning-box` | 주의해야 할 점 |
| `danger-box` | 위험, 금지, 안티패턴 |
| `success-box` | 권장 방법, 정답 |
| `tip-box` | 추가 팁, 참고 정보 |

### Before / After 비교 그리드

````markdown
<div class="compare-grid" markdown="1">
<div class="before" markdown="1">
**Bad - 나쁜 방법**

```java
// 코드 예시
```
</div>
<div class="after" markdown="1">
**Good - 좋은 방법**

```java
// 코드 예시
```
</div>
</div>
````

- `before`: 잘못된 방법
- `after`: 올바른 방법
- 모바일에서는 자동으로 세로 전환된다.

### MkDocs Admonition

```markdown
!!! note "참고"
    내용

!!! warning "주의"
    내용

!!! danger "위험"
    내용

!!! tip "팁"
    내용

!!! info "정보"
    내용
```

접을 수 있는 박스:

```markdown
??? note "접을 수 있는 참고"
    내용

???+ note "기본으로 열린 접기 박스"
    내용
```

## 작업 완료 후 Git 처리

- 사용자가 명시적으로 작업 완료를 요청했거나 문서/코드 변경 작업을 맡긴 경우, 작업이 끝나면 커밋 후 `git push`까지 수행한다.
- 단, 사용자 작업으로 보이는 변경사항은 되돌리지 않는다.
- 커밋 전 `git status`로 변경 범위를 확인한다.
