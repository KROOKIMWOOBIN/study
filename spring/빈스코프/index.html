    빈 스코프 (Bean Scope)

    스코프란? 왜 필요한가

    스프링 빈이 존재할 수 있는 범위를 스코프라고 한다.

    기본적으로 스프링 빈은 싱글톤으로 관리되지만, 경우에 따라 요청마다 새 객체, HTTP 요청마다 새 객체 등이 필요한 경우가 있다.

    스코프 종류

    스코프 설명 사용 환경
    singleton 기본값. 컨테이너 당 하나의 인스턴스 모든 환경
    prototype 요청할 때마다 새 인스턴스 생성 모든 환경
    request HTTP 요청 하나당 하나의 인스턴스 웹
    session HTTP 세션 하나당 하나의 인스턴스 웹
    application 서블릿 컨텍스트 하나당 하나 (= 싱글톤과 유사) 웹
    websocket 웹 소켓 하나당 하나 웹 소켓

    싱글톤 vs 프로토타입

    싱글톤 스코프

    @Component
// @Scope("singleton") — 기본값이므로 생략 가능
public class SingletonBean {
    private int count = 0;
    public void addCount() { count++; }
    public int getCount() { return count; }
}

    [클라이언트 A 요청] → getBean() → SingletonBean@x01 반환
[클라이언트 B 요청] → getBean() → SingletonBean@x01 반환 (같은 인스턴스!)

[스프링 컨테이너] ── 생성 ──> 빈 관리 ── 종료 ──> 소멸 콜백

    프로토타입 스코프

    @Component
@Scope("prototype")
public class PrototypeBean {
    private int count = 0;
    public void addCount() { count++; }
    public int getCount() { return count; }
}

    [클라이언트 A 요청] → getBean() → 새 인스턴스 생성 → PrototypeBean@x01 반환
[클라이언트 B 요청] → getBean() → 새 인스턴스 생성 → PrototypeBean@x02 반환 (다른 인스턴스!)

[스프링 컨테이너] ── 생성 ──> 초기화까지만 처리
                   이후 관리는 클라이언트 책임! (소멸 콜백 호출 안 함)

    비교

    구분 싱글톤 프로토타입
    인스턴스 수 1개 요청마다 새로 생성
    관리 주체 스프링 컨테이너 클라이언트
    초기화 콜백 O O
    소멸 콜백 O X (컨테이너가 관리 안 함)
    사용 시기 대부분의 빈 매번 다른 상태가 필요한 빈

    프로토타입 테스트

    @Test
void prototypeBeanFind() {
    AnnotationConfigApplicationContext ac =
        new AnnotationConfigApplicationContext(PrototypeBean.class);

    System.out.println("find prototypeBean1");
    PrototypeBean bean1 = ac.getBean(PrototypeBean.class);

    System.out.println("find prototypeBean2");
    PrototypeBean bean2 = ac.getBean(PrototypeBean.class);

    System.out.println("bean1=" + bean1);
    System.out.println("bean2=" + bean2);

    assertThat(bean1).isNotSameAs(bean2); // 서로 다른 인스턴스!

    ac.close();
    // 출력: bean1.destroy() 호출 안 됨!
}

    싱글톤 빈에서 프로토타입 빈 사용 문제

    문제 상황

    @Scope("singleton")
@Component
public class ClientBean {

    @Autowired
    private PrototypeBean prototypeBean; // 주입 시점에 한 번만 생성됨!

    public int logic() {
        prototypeBean.addCount();
        return prototypeBean.getCount();
    }
}

@Scope("prototype")
@Component
public class PrototypeBean {
    private int count = 0;
    public void addCount() { count++; }
    public int getCount() { return count; }
}

    기대:
  clientBean.logic() 첫 번째 호출 → 새 PrototypeBean 생성 → count: 1
  clientBean.logic() 두 번째 호출 → 새 PrototypeBean 생성 → count: 1

실제:
  ClientBean 생성 시 PrototypeBean 한 번만 주입됨
  clientBean.logic() 첫 번째 호출 → 기존 PrototypeBean 재사용 → count: 1
  clientBean.logic() 두 번째 호출 → 기존 PrototypeBean 재사용 → count: 2 ← 문제!

    해결 방법 1: ObjectProvider

    스프링이 제공하는 방법. 빈을 직접 주입받는 대신 빈 조회 기능(Provider)을 주입받는다.

    @Scope("singleton")
@Component
public class ClientBean {

    @Autowired
    private ObjectProvider<PrototypeBean> prototypeBeanProvider;

    public int logic() {
        PrototypeBean prototypeBean = prototypeBeanProvider.getObject(); // 매번 새로 생성
        prototypeBean.addCount();
        return prototypeBean.getCount();
    }
}

    clientBean.logic() 첫 번째 → getObject() → 새 PrototypeBean@x01 생성 → count: 1
clientBean.logic() 두 번째 → getObject() → 새 PrototypeBean@x02 생성 → count: 1 ✓

    ObjectProvider의 다른 기능

    // 빈이 없어도 됨
ObjectProvider<PrototypeBean> provider;
PrototypeBean bean = provider.getIfAvailable(); // null 가능

// 스트림
List<PrototypeBean> beans = provider.stream().collect(Collectors.toList());

    해결 방법 2: JSR-330 Provider

    javax.inject.Provider 표준 인터페이스 사용. 스프링에 의존하지 않는다.

    // build.gradle에 의존성 추가
implementation 'jakarta.inject:jakarta.inject-api:2.0.1'

    import jakarta.inject.Provider;

@Scope("singleton")
@Component
public class ClientBean {

    @Autowired
    private Provider<PrototypeBean> provider;

    public int logic() {
        PrototypeBean prototypeBean = provider.get(); // 매번 새로 생성
        prototypeBean.addCount();
        return prototypeBean.getCount();
    }
}

    ObjectProvider vs Provider

    구분 ObjectProvider Provider (JSR-330)
    표준 여부 스프링 전용 자바 표준
    기능 다양 (getIfAvailable, stream 등) 단순 (get() 하나)
    의존성 추가 불필요 (스프링에 포함) 필요 (별도 라이브러리)
    권장 스프링 프레임워크 사용 시 스프링 외 환경 고려 시

    해결 방법 3: 스코프 프록시 (가장 편리)

    @Scope(value = "prototype", proxyMode = ScopedProxyMode.TARGET_CLASS)
@Component
public class PrototypeBean {
    private int count = 0;
    public void addCount() { count++; }
    public int getCount() { return count; }
}

    @Scope("singleton")
@Component
public class ClientBean {

    @Autowired
    private PrototypeBean prototypeBean; // 프록시 객체가 주입됨

    public int logic() {
        // 프록시의 메서드 호출 → 내부에서 실제 빈 조회 후 호출
        prototypeBean.addCount();
        return prototypeBean.getCount();
    }
}

    [ClientBean]
    ↓ 주입된 것
[PrototypeBean CGLIB 프록시]
    ↓ addCount() 호출 시마다
[실제 PrototypeBean 조회 후 호출] → 매번 새 인스턴스

    프록시 동작 원리

    // 프록시가 하는 일 (개념적)
class PrototypeBeanCGLIB extends PrototypeBean {
    @Override
    public void addCount() {
        // 실제 빈을 스프링 컨테이너에서 꺼내서 호출
        PrototypeBean realBean = getBean(PrototypeBean.class);
        realBean.addCount();
    }
}

    ScopedProxyMode 옵션

    옵션 설명
    TARGET_CLASS CGLIB 바이트코드 조작 (클래스 기반)
    INTERFACES JDK 동적 프록시 (인터페이스 기반)
    NO 프록시 사용 안 함 (기본값)
    DEFAULT NO와 동일

    웹 스코프

    웹 환경에서만 동작. 스프링이 해당 스코프의 종료 시점까지 관리 (소멸 콜백 O).

    request 스코프

    HTTP 요청 하나당 하나의 빈. 같은 요청 안에서는 같은 인스턴스 공유.

    @Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class MyLogger {
    private String uuid;
    private String requestURL;

    public void setRequestURL(String requestURL) {
        this.requestURL = requestURL;
    }

    public void log(String message) {
        System.out.println("[" + uuid + "]" + "[" + requestURL + "] " + message);
    }

    @PostConstruct
    public void init() {
        uuid = UUID.randomUUID().toString();
        System.out.println("[" + uuid + "] request scope bean create: " + this);
    }

    @PreDestroy
    public void close() {
        System.out.println("[" + uuid + "] request scope bean close: " + this);
    }
}

    @Service
@RequiredArgsConstructor
public class LogDemoService {
    private final MyLogger myLogger;

    public void logic(String id) {
        myLogger.log("service id=" + id); // 요청마다 다른 MyLogger 인스턴스
    }
}

@Controller
@RequiredArgsConstructor
public class LogDemoController {
    private final LogDemoService logDemoService;
    private final MyLogger myLogger;

    @RequestMapping("/log-demo")
    @ResponseBody
    public String logDemo(HttpServletRequest request) {
        String requestURL = request.getRequestURL().toString();
        myLogger.setRequestURL(requestURL);
        myLogger.log("controller test");
        logDemoService.logic("testId");
        return "OK";
    }
}

    request 스코프 없이 구현하면?

    // request 스코프 없이 동일 기능을 구현하려면
// 모든 메서드 파라미터에 uuid, requestURL을 넘겨야 함 → 코드 지저분
public void logic(String id, String uuid, String requestURL) { ... }

    요청 처리 흐름

    HTTP 요청 A
    ↓
[LogDemoController]
    → myLogger.setRequestURL("http://localhost:8080/log-demo") [A 전용 MyLogger]
    → myLogger.log("controller test")
    ↓
[LogDemoService]
    → myLogger.log("service id=testId")     [같은 A 전용 MyLogger]
    ↓
HTTP 응답 A → MyLogger A 소멸 (@PreDestroy 호출)

HTTP 요청 B (동시)
    → 완전히 다른 MyLogger B 인스턴스 사용

    스코프 선택 기준 요약

    상황 스코프
    대부분의 경우 (무상태 서비스, 레포지토리) singleton
    매 요청마다 다른 상태가 필요한 경우 prototype
    HTTP 요청 단위 상태 관리 (로그 추적, 요청 컨텍스트) request
    사용자 세션 단위 상태 관리 (로그인 정보 등) session 
    // request 스코프 비교 예 (스코프 프록시 없이)
// → 순수 JDBC로 커넥션을 파라미터로 직접 넘기는 것처럼 번거로움

// request 스코프 + 프록시 사용 시
// → 스프링이 TransactionSynchronizationManager처럼 ThreadLocal로 관리해줌
// → 코드가 깔끔해짐
