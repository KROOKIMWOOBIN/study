    의존관계 자동 주입 (@Autowired)

    @Autowired 동작 방식

    기본 동작: 타입으로 조회

    @Autowired
private MemberRepository memberRepository;
// = ac.getBean(MemberRepository.class)와 동일

    스프링 컨테이너에서 타입이 일치하는 빈을 찾아서 주입한다.

    주입 과정

    @Autowired 발견
    ↓
타입으로 스프링 컨테이너 조회
    ↓
빈 1개 → 주입 성공
빈 0개 → 기본적으로 NoSuchBeanDefinitionException
         (@Autowired(required=false) 설정 시 주입 생략)
빈 2개 이상 → NoUniqueBeanDefinitionException

    @Autowired 옵션 처리

    required 옵션

    // 빈이 없어도 됨 (주입 생략)
@Autowired(required = false)
public void setMailSender(MailSender mailSender) {
    this.mailSender = mailSender; // 빈 없으면 호출 자체가 안 됨
}

// @Nullable 사용 (빈 없으면 null 주입)
@Autowired
public void setMailSender(@Nullable MailSender mailSender) {
    this.mailSender = mailSender; // null 허용
}

// Optional 사용
@Autowired
public void setMailSender(Optional<MailSender> mailSender) {
    this.mailSender = mailSender.orElse(null);
}
    방법 빈 없을 때 빈 있을 때
    @Autowired(required=false) 메서드 호출 자체 안 함 주입
    @Nullable null 주입 주입
    Optional<T> Optional.empty() Optional.of(빈)

    조회 대상 빈이 2개 이상일 때 문제

    문제 발생 상황

    public interface DiscountPolicy { int discount(Member member, int price); }

@Component
public class FixDiscountPolicy implements DiscountPolicy { ... }

@Component
public class RateDiscountPolicy implements DiscountPolicy { ... }

    @Autowired
private DiscountPolicy discountPolicy;
// NoUniqueBeanDefinitionException:
// expected single matching bean but found 2: fixDiscountPolicy, rateDiscountPolicy

    해결 방법 1: @Autowired + 필드명/파라미터명

    @Autowired는 타입 매칭 실패 시 필드명(또는 파라미터명)으로 빈 이름을 추가 매칭한다.

    // 필드명을 빈 이름과 일치시킴
@Autowired
private DiscountPolicy rateDiscountPolicy; // rateDiscountPolicy 빈 주입

// 생성자 파라미터명 이용
@Autowired
public OrderServiceImpl(MemberRepository memberRepository,
                        DiscountPolicy rateDiscountPolicy) { // 파라미터명으로 매칭
    this.memberRepository = memberRepository;
    this.discountPolicy = rateDiscountPolicy;
}

    단점: 필드명/파라미터명이 특정 구현체 이름에 종속되어 가독성이 좋지 않을 수 있음.

    해결 방법 2: @Qualifier

    추가 구별자를 붙여서 매칭한다.

    적용 방법

    // 빈에 @Qualifier 적용
@Component
@Qualifier("mainDiscountPolicy") // 구별자 지정
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Component
@Qualifier("fixDiscountPolicy")
public class FixDiscountPolicy implements DiscountPolicy { ... }

    // 주입 시 @Qualifier 사용
@Autowired
public OrderServiceImpl(MemberRepository memberRepository,
                        @Qualifier("mainDiscountPolicy") DiscountPolicy discountPolicy) {
    this.memberRepository = memberRepository;
    this.discountPolicy = discountPolicy;
}

    @Qualifier 매칭 과정

    @Qualifier("mainDiscountPolicy") 발견
    ↓
@Qualifier("mainDiscountPolicy") 찾음 → 주입 성공
찾지 못하면 → 빈 이름 "mainDiscountPolicy"로 추가 탐색
또 없으면 → NoSuchBeanDefinitionException

    커스텀 어노테이션으로 개선 (타입 안전)

    // 문자열은 컴파일 타임에 오류를 잡을 수 없음 (오타 위험)
// @Qualifier("mainDiscountPolicy") 대신 커스텀 어노테이션 사용

@Target({ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER, ElementType.TYPE, ElementType.ANNOTATION_TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Qualifier("mainDiscountPolicy")
public @interface MainDiscountPolicy { // 커스텀 어노테이션
}

    @Component
@MainDiscountPolicy // 타입 안전 (오타 시 컴파일 오류)
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Autowired
public OrderServiceImpl(MemberRepository memberRepository,
                        @MainDiscountPolicy DiscountPolicy discountPolicy) {
    this.discountPolicy = discountPolicy;
}

    해결 방법 3: @Primary

    우선순위를 지정한다. 여러 빈 중 우선 주입될 빈에 @Primary를 붙인다.

    @Component
@Primary // 우선순위 최고
public class RateDiscountPolicy implements DiscountPolicy { ... }

@Component
public class FixDiscountPolicy implements DiscountPolicy { ... }

    // @Primary 설정 후에는 그냥 타입으로 주입 가능
@Autowired
private DiscountPolicy discountPolicy; // RateDiscountPolicy 주입됨

    @Primary vs @Qualifier 우선순위

    @Qualifier > @Primary

@Qualifier를 명시하면 @Primary보다 우선 적용

    실무 활용 패턴

    // 기본 DB (메인 DB) - @Primary
@Bean
@Primary
public DataSource dataSource() {
    return new HikariDataSource(); // 주 DB
}

// 서브 DB (통계용) - @Qualifier
@Bean
@Qualifier("subDataSource")
public DataSource subDataSource() {
    return new HikariDataSource(); // 서브 DB
}

    조회한 빈이 모두 필요할 때: List, Map

    @Service
public class DiscountService {

    // 모든 DiscountPolicy 구현체를 Map으로 주입
    private final Map<String, DiscountPolicy> policyMap;
    private final List<DiscountPolicy> policies;

    @Autowired
    public DiscountService(Map<String, DiscountPolicy> policyMap,
                           List<DiscountPolicy> policies) {
        this.policyMap = policyMap;
        this.policies = policies;
        // policyMap: {"fixDiscountPolicy": FixDiscountPolicy, "rateDiscountPolicy": RateDiscountPolicy}
        // policies: [FixDiscountPolicy, RateDiscountPolicy]
    }

    public int discount(Member member, int price, String discountCode) {
        // 클라이언트가 선택한 할인 정책을 동적으로 사용
        DiscountPolicy discountPolicy = policyMap.get(discountCode);
        return discountPolicy.discount(member, price);
    }
}

    // 테스트
@Test
void findAllBean() {
    ApplicationContext ac = new AnnotationConfigApplicationContext(
        AutoAppConfig.class, DiscountService.class);
    DiscountService discountService = ac.getBean(DiscountService.class);

    int fixDiscountPrice = discountService.discount(member, 10000, "fixDiscountPolicy");
    int rateDiscountPrice = discountService.discount(member, 10000, "rateDiscountPolicy");

    assertThat(fixDiscountPrice).isEqualTo(1000);
    assertThat(rateDiscountPrice).isEqualTo(1000);
}

    자동 등록 vs 수동 등록 선택 기준

    상황 권장 방식 이유
    일반 비즈니스 로직 자동 (@ComponentScan) 수가 많고 패턴이 명확
    기술 지원 빈 (AOP, DataSource 등) 수동 (@Bean) 애플리케이션 전반에 영향, 명확히 드러내야
    다형성 활용 (여러 구현체 존재) 수동 (@Bean) 또는 @Qualifier 어떤 빈이 주입될지 한 눈에 보여야 함
    외부 라이브러리 빈 수동 (@Bean) 소스 코드에 어노테이션 추가 불가

    다형성 빈 관리 - 수동 등록 예시

    @Configuration
public class DiscountPolicyConfig {
    // 어떤 할인 정책 빈이 있는지 한 눈에 파악 가능
    @Bean
    public DiscountPolicy rateDiscountPolicy() {
        return new RateDiscountPolicy();
    }

    @Bean
    public DiscountPolicy fixDiscountPolicy() {
        return new FixDiscountPolicy();
    }
}

    @Autowired 매칭 정리

    @Autowired 주입 순서:
1. 타입 매칭 시도
2. 타입 매칭 결과가 2개 이상이면:
   → 2-1. @Qualifier 확인 → 있으면 매칭
   → 2-2. @Primary 확인 → 있으면 매칭
   → 2-3. 필드명/파라미터명으로 빈 이름 매칭
3. 모두 실패하면 예외
