    데드락 (Deadlock)

    왜 알아야 하는가?

    멀티스레드 환경과 데이터베이스 트랜잭션에서 데드락은 언제든지 발생할 수 있다. 데드락이 발생하면 관련 트랜잭션이 무한 대기 상태에 빠지거나 DB가 강제로 하나를 롤백시킨다. 원인을 모르면 간헐적으로 발생하는 에러를 재현하기도 어렵고, 로그도 알아보기 어렵다.

    정의

    두 개 이상의 프로세스/트랜잭션이 서로가 가진 자원을 기다리면서 영원히 진행되지 못하는 상태.

    트랜잭션 A ──잠금 보유──→ 자원 1
트랜잭션 A ──대기 중──→  자원 2 (B가 잠금 보유)

트랜잭션 B ──잠금 보유──→ 자원 2
트랜잭션 B ──대기 중──→  자원 1 (A가 잠금 보유)

결과: A는 B를 기다리고, B는 A를 기다린다 → 영원히 대기

    발생 조건 4가지 (Coffman Conditions)

    데드락이 발생하려면 다음 4가지 조건이 동시에 성립해야 한다.

    조건 설명 예시
    상호 배제 (Mutual Exclusion) 자원을 한 번에 하나의 프로세스만 사용 행 잠금(Row Lock)
    점유와 대기 (Hold and Wait) 자원을 보유한 채 다른 자원을 기다림 A가 row1 잠금 후 row2 대기
    비선점 (No Preemption) 다른 프로세스의 자원을 강제로 빼앗을 수 없음 트랜잭션이 완료 전까지 잠금 유지
    순환 대기 (Circular Wait) 프로세스들이 원형으로 서로 대기 A→B→A

    이 중 하나라도 제거하면 데드락을 예방할 수 있다.

    DB 데드락 발생 예시

    시나리오: 계좌 이체

    [계좌 A: id=1, 잔액=10000]
[계좌 B: id=2, 잔액=5000]

트랜잭션 1: A에서 B로 이체
트랜잭션 2: B에서 A로 이체

    타임라인

    시간  트랜잭션 1 (A→B)              트랜잭션 2 (B→A)
────────────────────────────────────────────────────
t1:  BEGIN;
t2:                                  BEGIN;
t3:  SELECT * FROM account
     WHERE id=1 FOR UPDATE;  ←── 계좌 A 잠금 획득
t4:                                  SELECT * FROM account
                                     WHERE id=2 FOR UPDATE; ←── 계좌 B 잠금 획득
t5:  SELECT * FROM account
     WHERE id=2 FOR UPDATE;  ←── 계좌 B 잠금 대기... (B가 보유 중)
t6:                                  SELECT * FROM account
                                     WHERE id=1 FOR UPDATE; ←── 계좌 A 잠금 대기... (A가 보유 중)

     ★ 데드락 발생! ★
     DB가 감지 → 하나를 Victim으로 선택 → 롤백

    텍스트 다이어그램

         트랜잭션 1                    트랜잭션 2
     ──────────                    ──────────
     잠금 보유: [계좌 A]           잠금 보유: [계좌 B]
          │                              │
          │ 대기 중                       │ 대기 중
          ▼                              ▼
     [계좌 B] ←───── 서로 대기 ──────→ [계좌 A]

     A는 B를 기다리고, B는 A를 기다린다
     → 순환 대기 → 데드락!

    SQL로 재현

    -- 세션 1
BEGIN;
SELECT * FROM account WHERE id = 1 FOR UPDATE;
-- 잠깐 대기 (세션 2가 id=2 잠금 획득하도록)
SELECT * FROM account WHERE id = 2 FOR UPDATE; -- 여기서 대기
COMMIT;

-- 세션 2 (동시에 실행)
BEGIN;
SELECT * FROM account WHERE id = 2 FOR UPDATE;
SELECT * FROM account WHERE id = 1 FOR UPDATE; -- 데드락 발생 지점
COMMIT;

    감지 및 해소 방법

    1. DB 타임아웃 설정

    -- MySQL: 잠금 대기 타임아웃 (초)
SET innodb_lock_wait_timeout = 5;

-- PostgreSQL: 잠금 대기 타임아웃 (밀리초)
SET lock_timeout = '5s';

-- 세션 레벨
SET LOCAL lock_timeout = '3s';

    # Spring application.yml
spring:
  datasource:
    hikari:
      connection-timeout: 30000
  jpa:
    properties:
      javax.persistence.lock.timeout: 3000  # 3초

    2. 데드락 감지 (DB 자동)

    MySQL InnoDB는 데드락을 자동으로 감지하고, 롤백 비용이 적은 트랜잭션을 Victim으로 선택하여 롤백한다.

    -- MySQL 데드락 로그 확인
SHOW ENGINE INNODB STATUS;
-- 'LATEST DETECTED DEADLOCK' 섹션 확인

-- PostgreSQL 데드락 로그 활성화
-- postgresql.conf
deadlock_timeout = 1s  -- 데드락 체크 간격
log_lock_waits = on    -- 잠금 대기 로깅

    예방 방법

    1. 잠금 순서 통일 (가장 효과적)

    모든 트랜잭션이 자원을 항상 같은 순서로 접근하도록 강제한다. 순환 대기 조건을 제거한다.

    // 나쁜 예: 순서가 일관되지 않음
// 트랜잭션 1: from=1 → to=2 (id 순서대로 잠금)
// 트랜잭션 2: from=2 → to=1 (역순으로 잠금) → 데드락 위험

// 좋은 예: 항상 작은 id 먼저 잠금
@Service
public class AccountService {

    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        // 항상 작은 id를 먼저 잠금 (순서 통일)
        Long firstId = Math.min(fromId, toId);
        Long secondId = Math.max(fromId, toId);

        Account first = accountRepository.findByIdWithLock(firstId);
        Account second = accountRepository.findByIdWithLock(secondId);

        Account from = fromId.equals(firstId) ? first : second;
        Account to = toId.equals(firstId) ? first : second;

        from.withdraw(amount);
        to.deposit(amount);
    }
}

    2. 트랜잭션을 짧게 유지

    // 나쁜 예: 트랜잭션 안에서 외부 API 호출
@Transactional
public void processOrder(Long orderId) {
    Order order = orderRepository.findByIdWithLock(orderId); // 잠금 획득

    // 외부 결제 API 호출 (수 초 소요 가능) → 잠금을 오래 유지
    paymentApiClient.charge(order.getAmount());

    order.setStatus(OrderStatus.PAID);
}

// 좋은 예: 잠금이 필요한 부분만 트랜잭션으로 분리
public void processOrder(Long orderId) {
    // 1. 조회 (잠금 없음)
    Order order = orderRepository.findById(orderId).orElseThrow();

    // 2. 외부 API 호출 (트랜잭션 밖)
    String paymentId = paymentApiClient.charge(order.getAmount());

    // 3. 상태 업데이트 (짧은 트랜잭션)
    orderService.updatePaymentStatus(orderId, paymentId);
}

    3. 낙관적 락 사용

    비관적 락(SELECT FOR UPDATE) 대신 낙관적 락(@Version)을 사용하면 물리적 잠금이 없으므로 데드락이 발생하지 않는다.

    @Entity
public class Product {
    @Id @GeneratedValue
    private Long id;

    private int stock;

    @Version  // 낙관적 락
    private Long version;
}

@Service
public class ProductService {

    @Transactional
    @Retryable(value = OptimisticLockingFailureException.class, maxAttempts = 3)
    public void decreaseStock(Long productId, int quantity) {
        Product product = productRepository.findById(productId).orElseThrow();
        product.setStock(product.getStock() - quantity);
        // version이 맞지 않으면 OptimisticLockingFailureException
        // → @Retryable로 재시도
    }
}

    4. 인덱스 활용 (Lock 범위 최소화)

    -- 인덱스 없음: Table Lock 또는 많은 행에 Lock
UPDATE orders SET status = 'SHIPPED' WHERE user_id = 100;
-- user_id에 인덱스가 없으면 전체 테이블 스캔 + 많은 행 잠금

-- 인덱스 있음: 해당 행만 Lock
CREATE INDEX idx_orders_user_id ON orders(user_id);
-- 이제 user_id=100인 행만 잠금 → 다른 트랜잭션과 충돌 최소화

    Java/Spring 데드락

    DB뿐 아니라 Java 멀티스레드 환경에서도 발생한다.

    synchronized 데드락

    public class DeadlockExample {
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    // 스레드 1이 실행하는 메서드
    public void methodA() {
        synchronized (lockA) {
            System.out.println("Thread 1: lockA 획득");
            try { Thread.sleep(100); } catch (InterruptedException e) {}

            synchronized (lockB) { // lockB 대기 (Thread 2가 보유)
                System.out.println("Thread 1: lockB 획득");
            }
        }
    }

    // 스레드 2가 실행하는 메서드
    public void methodB() {
        synchronized (lockB) {
            System.out.println("Thread 2: lockB 획득");
            try { Thread.sleep(100); } catch (InterruptedException e) {}

            synchronized (lockA) { // lockA 대기 (Thread 1이 보유) → 데드락!
                System.out.println("Thread 2: lockA 획득");
            }
        }
    }
}

    ReentrantLock으로 타임아웃 설정

    import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class SafeLockExample {
    private final ReentrantLock lockA = new ReentrantLock();
    private final ReentrantLock lockB = new ReentrantLock();

    public void safeMethod() throws InterruptedException {
        boolean acquiredA = false;
        boolean acquiredB = false;

        try {
            acquiredA = lockA.tryLock(1, TimeUnit.SECONDS); // 1초 내 획득 시도
            if (!acquiredA) {
                System.out.println("lockA 획득 실패, 재시도 필요");
                return;
            }

            acquiredB = lockB.tryLock(1, TimeUnit.SECONDS);
            if (!acquiredB) {
                System.out.println("lockB 획득 실패, 재시도 필요");
                return;
            }

            // 두 락 모두 획득한 경우에만 작업 수행
            doWork();

        } finally {
            // 반드시 역순으로 해제
            if (acquiredB) lockB.unlock();
            if (acquiredA) lockA.unlock();
        }
    }

    private void doWork() {
        System.out.println("작업 수행");
    }
}

    Java 데드락 감지 (JVM 레벨)

    // ThreadMXBean으로 데드락 스레드 감지
import java.lang.management.ManagementFactory;
import java.lang.management.ThreadMXBean;

public class DeadlockDetector {

    public static void detectDeadlock() {
        ThreadMXBean threadBean = ManagementFactory.getThreadMXBean();
        long[] deadlockedThreads = threadBean.findDeadlockedThreads();

        if (deadlockedThreads != null) {
            System.err.println("데드락 감지! 스레드 수: " + deadlockedThreads.length);
            ThreadInfo[] threadInfos = threadBean.getThreadInfo(deadlockedThreads, true, true);
            for (ThreadInfo info : threadInfos) {
                System.err.println("데드락 스레드: " + info.getThreadName());
                System.err.println("대기 중인 락: " + info.getLockName());
                System.err.println("락 보유자: " + info.getLockOwnerName());
            }
        }
    }
}

    Spring 트랜잭션에서 자주 발생하는 데드락 패턴

    // 패턴 1: 같은 테이블에서 INSERT + UPDATE 순서 교차
@Transactional
public void methodA(Long userId) {
    userRepository.save(new User(userId));          // INSERT user(1)
    orderRepository.updateUserOrders(userId);       // UPDATE order WHERE user_id=1
}

@Transactional
public void methodB(Long userId) {
    orderRepository.updateUserOrders(userId);       // UPDATE order WHERE user_id=1
    userRepository.save(new User(userId));          // INSERT user(1) → 데드락 가능
}

    // 패턴 2: 부모-자식 테이블 동시 접근
// 외래키 제약 때문에 INSERT 시 부모 테이블에 S-Lock 발생
// 동시에 부모 테이블 UPDATE 시 X-Lock 충돌 → 데드락

// 해결: CASCADE 설정 재검토 또는 잠금 순서 통일

   - 잠금 순서 통일

   - 낙관적 락으로 전환 검토
   - 인덱스 추가로 락 범위 축소

    @Retryable을 이용한 재시도 처리

    // build.gradle
implementation 'org.springframework.retry:spring-retry'

// @EnableRetry 추가
@SpringBootApplication
@EnableRetry
public class Application { ... }

// 서비스
@Service
public class OrderService {

    @Transactional
    @Retryable(
        value = { CannotAcquireLockException.class, DeadlockLoserDataAccessException.class },
        maxAttempts = 3,
        backoff = @Backoff(delay = 100, multiplier = 2) // 100ms, 200ms, 400ms
    )
    public void processOrder(Long orderId) {
        Order order = orderRepository.findByIdWithLock(orderId);
        order.process();
    }

    @Recover
    public void recoverFromDeadlock(CannotAcquireLockException e, Long orderId) {
        log.error("데드락으로 인한 주문 처리 실패. orderId: {}", orderId, e);
        // 알림 발송, 수동 처리 큐 등록 등
    }
}

    데드락 vs 라이브락 vs 기아 상태

    구분 설명 특징
    데드락 (Deadlock) 모두 대기 상태, 진행 불가 CPU 사용 없음, 완전 멈춤
    라이브락 (Livelock) 계속 상태를 바꾸지만 진행 불가 CPU 사용 있음, 움직이는 데드락
    기아 (Starvation) 특정 프로세스가 자원을 영원히 못 얻음 다른 프로세스는 정상 동작

    요약 정리

    데드락 = 두 트랜잭션이 서로가 가진 자원을 기다리며 영원히 대기

발생 조건 (Coffman):
1. 상호 배제
2. 점유와 대기
3. 비선점
4. 순환 대기 ← 이것을 없애는 게 핵심

예방:
├── 잠금 순서 통일 (작은 id 먼저)
├── 트랜잭션 짧게
├── 낙관적 락 (@Version)
└── 인덱스로 락 범위 축소
