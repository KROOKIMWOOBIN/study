    데이터베이스 인덱스 (Database Index)

    왜 사용하는가?

    인덱스는 데이터베이스 테이블의 검색 속도를 향상시키기 위한 자료구조다. 책의 목차와 같은 역할을 한다.

    Full Table Scan vs Index Scan

    Full Table Scan (인덱스 없음):
┌─────────────────────────────────────────┐
│ 전체 테이블 1,000,000 행을 순서대로 탐색  │
│ [행1] → [행2] → [행3] → ... → [행1M]    │
│                                          │
│ WHERE name = 'Kim' 인 행을 찾으려면      │
│ 최악의 경우 100만 행 모두 탐색           │
│ 시간복잡도: O(n)                         │
└─────────────────────────────────────────┘

Index Scan (인덱스 있음):
┌─────────────────────────────────────────┐
│ B-Tree 인덱스로 'Kim' 위치 즉시 파악     │
│ → 해당 행만 직접 접근                    │
│ 시간복잡도: O(log n)                     │
└─────────────────────────────────────────┘

    성능 비교

    구분 Full Table Scan Index Scan
    시간복잡도 O(n) O(log n)
    100만 행 기준 ~1,000,000 비교 ~20 비교
    디스크 I/O 전체 페이지 읽기 필요한 페이지만
    적합한 경우 소량 데이터, 대부분 행 반환 선택도 높은 조건

    B-Tree 인덱스 구조

    MySQL InnoDB의 기본 인덱스 자료구조는 B+ Tree다.

    B+ Tree 인덱스 구조:

                    [루트 노드]
                    [30 | 60]
                   /    |    \
          [리프]      [리프]     [리프]
        [10|20|30]  [40|50|60]  [70|80|90]
           ↓↓↓         ↓↓↓        ↓↓↓
        실제 데이터   실제 데이터  실제 데이터
        (또는 포인터) (또는 포인터)(또는 포인터)

리프 노드끼리 연결 (Linked List):
[10|20|30] ←→ [40|50|60] ←→ [70|80|90]
    ↑ 범위 검색(BETWEEN, >, <)이 빠른 이유!

    B+ Tree 특징

    • 모든 실제 데이터(또는 포인터)는 리프 노드에 저장
    • 리프 노드는 연결 리스트로 연결되어 범위 검색에 유리
    • 트리 높이가 낮아 탐색 효율이 높음 (균형 트리)
    • 삽입/삭제 시 트리 재구성 비용 발생

    인덱스 종류

    1. Clustered Index vs Non-Clustered Index

    Clustered Index:
┌──────────────────────────────────┐
│ 실제 데이터가 인덱스 순서로 정렬   │
│ 인덱스 = 데이터 자체              │
│                                  │
│ B-Tree 리프 노드 = 실제 행 데이터  │
│ MySQL InnoDB: PK가 Clustered     │
└──────────────────────────────────┘

Non-Clustered Index (Secondary Index):
┌──────────────────────────────────┐
│ 인덱스는 별도 구조                │
│ 리프 노드 = PK 값 (포인터)        │
│                                  │
│ name 인덱스 탐색                  │
│   → PK(id) 조회                  │
│   → Clustered Index에서 실제 행  │
│ (이중 탐색 = "Double Lookup")    │
└──────────────────────────────────┘

    -- Clustered Index (PK 자동 생성)
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,  -- Clustered Index
    name VARCHAR(100),
    email VARCHAR(200)
);

-- Non-Clustered Index
CREATE INDEX idx_users_name ON users(name);  -- Secondary Index
    구분 Clustered Index Non-Clustered Index
    테이블당 개수 1개 여러 개
    데이터 정렬 인덱스 순서로 정렬됨 별도 정렬 없음
    저장 방식 리프 노드 = 실제 데이터 리프 노드 = PK 포인터
    조회 속도 빠름 (단일 탐색) 상대적으로 느림 (이중 탐색)
    쓰기 비용 높음 (데이터 재정렬) 낮음

    2. 복합 인덱스 (Composite Index)

    두 개 이상의 컬럼을 묶어 만드는 인덱스.

    -- 복합 인덱스 생성
CREATE INDEX idx_orders_user_status ON orders(user_id, status, created_at);

-- 이 인덱스를 사용하는 쿼리
SELECT * FROM orders WHERE user_id = 1;                          -- O (선두 컬럼)
SELECT * FROM orders WHERE user_id = 1 AND status = 'PAID';     -- O
SELECT * FROM orders WHERE user_id = 1 AND status = 'PAID'
                       AND created_at > '2024-01-01';           -- O

-- 이 인덱스를 사용하지 못하는 쿼리
SELECT * FROM orders WHERE status = 'PAID';                      -- X (선두 컬럼 없음)
SELECT * FROM orders WHERE created_at > '2024-01-01';           -- X

    선두 컬럼 원칙 (Leftmost Prefix Rule)

    인덱스: (A, B, C)

사용 가능:
  WHERE A = ?               → A 사용
  WHERE A = ? AND B = ?     → A, B 사용
  WHERE A = ? AND B = ? AND C = ?  → A, B, C 사용
  WHERE A = ? AND C = ?     → A만 사용 (B 건너뜀)

사용 불가:
  WHERE B = ?               → 인덱스 미사용
  WHERE C = ?               → 인덱스 미사용
  WHERE B = ? AND C = ?     → 인덱스 미사용

    복합 인덱스 컬럼 순서 결정 기준

    1. 선택도(Cardinality)가 높은 컬럼을 앞에 (= 구분되는 값이 많은 컬럼)
    2. WHERE 조건에 자주 등장하는 컬럼을 앞에
    3. 범위 조건(>, <, BETWEEN)은 뒤에 (범위 조건 이후 컬럼은 인덱스 효율 저하)
    -- 잘못된 순서 (status 카디널리티 낮음)
CREATE INDEX idx_bad ON orders(status, user_id);

-- 올바른 순서
CREATE INDEX idx_good ON orders(user_id, status);

    3. Covering Index

    쿼리에 필요한 모든 컬럼이 인덱스에 포함된 경우. 실제 테이블 접근 없이 인덱스만으로 결과 반환.

    -- 인덱스 정의
CREATE INDEX idx_covering ON users(name, email, age);

-- Covering Index 활용 쿼리 (SELECT 컬럼이 모두 인덱스에 포함)
SELECT name, email FROM users WHERE name = 'Kim';
-- EXPLAIN Extra: "Using index" → 테이블 접근 없음!

-- Covering Index 미활용 (phone은 인덱스에 없음)
SELECT name, phone FROM users WHERE name = 'Kim';
-- EXPLAIN Extra: (없음) → 테이블 접근 필요

    Non-Covering:
  인덱스 탐색 → PK 획득 → 테이블 접근 → 데이터 반환
      (느림: 두 번의 I/O)

Covering:
  인덱스 탐색 → 데이터 반환 (인덱스 내에 있음)
      (빠름: 한 번의 I/O)

    4. Unique Index

    중복값을 허용하지 않는 인덱스. 제약조건 + 검색 최적화 역할.

    -- Unique Index 생성
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- PK는 자동으로 Unique Index
-- UNIQUE 제약조건도 내부적으로 Unique Index 생성
ALTER TABLE users ADD CONSTRAINT uq_email UNIQUE (email);

    인덱스를 타지 않는 경우

    1. 함수 사용

    -- 인덱스 미사용 (함수로 컬럼 변형)
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- 인덱스 사용 (범위 조건으로 변환)
SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

    2. 묵시적 형 변환 (Implicit Type Conversion)

    -- 테이블: id는 BIGINT 타입
-- 인덱스 미사용 (문자열 → 숫자 변환 발생)
SELECT * FROM users WHERE id = '123';

-- 인덱스 사용
SELECT * FROM users WHERE id = 123;

-- 반대도 주의: phone이 VARCHAR인데 숫자로 비교
SELECT * FROM users WHERE phone = 01012345678;  -- 형변환 발생

    3. LIKE 앞 와일드카드

    -- 인덱스 미사용 (앞 와일드카드)
SELECT * FROM users WHERE name LIKE '%Kim';
SELECT * FROM users WHERE name LIKE '%Kim%';

-- 인덱스 사용 (뒤 와일드카드만)
SELECT * FROM users WHERE name LIKE 'Kim%';

    이유:
B-Tree는 왼쪽부터 정렬됨
'%Kim' 패턴은 시작점을 알 수 없으므로 전체 스캔 필요
'Kim%' 패턴은 'Kim'에서 시작하는 범위를 특정 가능

    4. OR 조건

    -- 인덱스 미사용 (컬럼이 다른 OR)
SELECT * FROM users WHERE name = 'Kim' OR email = 'kim@test.com';

-- 해결: UNION ALL 사용
SELECT * FROM users WHERE name = 'Kim'
UNION ALL
SELECT * FROM users WHERE email = 'kim@test.com' AND name != 'Kim';

-- 같은 컬럼 OR은 IN으로 최적화 가능
SELECT * FROM users WHERE status = 'ACTIVE' OR status = 'PENDING';
-- →
SELECT * FROM users WHERE status IN ('ACTIVE', 'PENDING');

    5. NOT, !=, <>

    -- 인덱스 미사용 경향
SELECT * FROM users WHERE status != 'DELETED';
SELECT * FROM users WHERE status <> 'DELETED';
SELECT * FROM users WHERE NOT (status = 'DELETED');

-- 해결: 긍정 조건으로 변환
SELECT * FROM users WHERE status IN ('ACTIVE', 'PENDING', 'SUSPENDED');

    6. 카디널리티가 낮은 컬럼

    -- gender 컬럼: 'M', 'F' 두 가지 값 (카디널리티 = 2)
-- 전체 행의 50%를 반환하므로 Full Scan이 더 효율적
-- MySQL 옵티마이저가 인덱스를 사용하지 않을 수 있음
SELECT * FROM users WHERE gender = 'M';

-- 복합 인덱스로 해결 (카디널리티 높은 컬럼과 조합)
CREATE INDEX idx_users_gender_region ON users(region_id, gender);
SELECT * FROM users WHERE region_id = 5 AND gender = 'M';

    인덱스 미사용 케이스 요약

    케이스 예시 해결책
    함수 사용 YEAR(created_at) = 2024 범위 조건으로 변환
    묵시적 형변환 id = '123' (id가 INT) 타입 맞추기
    앞 와일드카드 LIKE '%검색어' ElasticSearch 사용
    OR (다른 컬럼) a = 1 OR b = 2 UNION ALL
    NOT/!= status != 'DELETED' 긍정 조건으로 변환
    낮은 카디널리티 WHERE gender = 'M' 복합 인덱스

    EXPLAIN 실행 계획 읽는 법

    EXPLAIN SELECT * FROM orders
WHERE user_id = 1 AND status = 'PAID'
ORDER BY created_at DESC;

    주요 컬럼 설명

    컬럼 의미 좋은 값 나쁜 값
    type 접근 방식 const, ref, range ALL, index
    key 실제 사용된 인덱스 인덱스명 NULL
    rows 예상 탐색 행 수 낮을수록 좋음 높을수록 나쁨
    Extra 추가 정보 Using index Using filesort, Using temporary

    type 값 순위 (좋음 → 나쁨)

    system > const > eq_ref > ref > range > index > ALL

system : 1행짜리 테이블
const  : PK/Unique로 1행 조회 (WHERE id = 1)
eq_ref : JOIN에서 PK/Unique 사용
ref    : 인덱스로 여러 행 조회 (WHERE user_id = 1)
range  : 인덱스 범위 스캔 (WHERE id BETWEEN 1 AND 100)
index  : 인덱스 전체 스캔 (Full Index Scan)
ALL    : 테이블 전체 스캔 (최악!)

    Extra 값 해석

    -- 실행 계획 예시
+----+-------------+--------+------+------------------+---------+---------+-------+------+-------------+
| id | select_type | table  | type | possible_keys    | key     | key_len | ref   | rows | Extra       |
+----+-------------+--------+------+------------------+---------+---------+-------+------+-------------+
|  1 | SIMPLE      | orders | ref  | idx_user_status  | idx_... |      8  | const |  150 | Using index |
+----+-------------+--------+------+------------------+---------+---------+-------+------+-------------+
    Extra 값 의미 성능
    Using index Covering Index 사용 (테이블 접근 없음) 매우 좋음
    Using where WHERE 조건 필터링 보통
    Using filesort ORDER BY를 메모리/디스크에서 정렬 나쁨
    Using temporary 임시 테이블 사용 (GROUP BY, DISTINCT) 나쁨
    Using index condition Index Condition Pushdown (ICP) 좋음

    인덱스 설계 원칙 및 실무 팁

    인덱스를 생성해야 할 때

    -- 1. WHERE 조건에 자주 사용되는 컬럼
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 2. JOIN ON 조건에 사용되는 컬럼 (외래키)
CREATE INDEX idx_order_items_order_id ON order_items(order_id);

-- 3. ORDER BY에 자주 사용되는 컬럼
CREATE INDEX idx_posts_created_at ON posts(created_at DESC);

-- 4. GROUP BY에 사용되는 컬럼
CREATE INDEX idx_orders_status ON orders(status);

    실무 설계 팁

    -- Tip 1: 범위 조건 컬럼은 복합 인덱스 마지막에
-- 잘못된 예: created_at이 범위인데 중간에 위치
CREATE INDEX idx_bad ON orders(user_id, created_at, status);
-- 올바른 예
CREATE INDEX idx_good ON orders(user_id, status, created_at);

-- Tip 2: WHERE + ORDER BY 함께 고려
-- 이 쿼리를 최적화하려면
SELECT * FROM posts WHERE user_id = 1 ORDER BY created_at DESC;
-- 이 인덱스가 효율적
CREATE INDEX idx_posts_user_created ON posts(user_id, created_at);

-- Tip 3: 카운트 쿼리 최적화 (Covering Index)
SELECT COUNT(*) FROM orders WHERE user_id = 1 AND status = 'PAID';
CREATE INDEX idx_covering ON orders(user_id, status);
-- Extra: Using index (테이블 접근 없음)

    인덱스 현황 확인

    -- 테이블의 인덱스 목록
SHOW INDEX FROM orders;

-- 인덱스 통계 갱신
ANALYZE TABLE orders;

-- 인덱스 힌트 (옵티마이저 무시하고 강제 지정)
SELECT * FROM orders USE INDEX (idx_user_status) WHERE user_id = 1;
SELECT * FROM orders FORCE INDEX (idx_user_status) WHERE user_id = 1;
SELECT * FROM orders IGNORE INDEX (idx_user_status) WHERE user_id = 1;

    과도한 인덱스의 단점

    쓰기 작업 시 인덱스 유지 비용:

INSERT 발생:
  테이블에 행 추가
  + 인덱스1 B-Tree 재구성
  + 인덱스2 B-Tree 재구성
  + 인덱스3 B-Tree 재구성
  + ... (인덱스 수만큼 추가 작업)

UPDATE, DELETE도 동일하게 인덱스 모두 갱신 필요
    인덱스 수 읽기 성능 쓰기 성능 저장 공간
    0개 최악 최고 최소
    적당 (3-5개) 좋음 좋음 보통
    과도 (10개+) 좋음 나쁨 많음

    실무 권장사항

    - 테이블당 인덱스는 5개 이하 권장
- 실제 쿼리 패턴 분석 후 생성 (추측으로 만들지 말 것)
- 불필요한 인덱스는 주기적으로 삭제
- 쓰기가 많은 테이블(로그, 이벤트)은 인덱스 최소화
- 읽기 전용 테이블은 인덱스를 많이 활용 가능

    -- 사용되지 않는 인덱스 찾기 (MySQL 8.0+)
SELECT * FROM sys.schema_unused_indexes
WHERE object_schema = 'your_database';

-- 중복 인덱스 찾기
SELECT * FROM sys.schema_redundant_indexes
WHERE table_schema = 'your_database';

    어떨 때 많이 쓰는가?

    인덱스가 효과적인 상황

    -- 1. 대용량 테이블에서 소수 행 조회
SELECT * FROM users WHERE email = 'kim@test.com';  -- 1행 반환

-- 2. 자주 조회되는 목록 페이징
SELECT * FROM posts WHERE category_id = 5
ORDER BY created_at DESC LIMIT 20 OFFSET 0;

-- 3. 집계 쿼리 최적화
SELECT user_id, COUNT(*) FROM orders
WHERE status = 'PAID' GROUP BY user_id;

-- 4. 범위 검색
SELECT * FROM products
WHERE price BETWEEN 10000 AND 50000;

-- 5. JOIN 성능 향상
SELECT u.name, o.total_amount
FROM users u
JOIN orders o ON u.id = o.user_id  -- user_id에 인덱스 필요!
WHERE u.region = 'Seoul';

    인덱스가 불필요한/역효과인 상황

    - 소량 데이터 테이블 (수백~수천 행): Full Scan이 더 빠를 수 있음
- 대부분의 행을 반환하는 쿼리 (WHERE 조건이 느슨함)
- 쓰기가 극도로 많은 테이블 (배치 인서트 로그 등)
- 임시 테이블 또는 일회성 쿼리

    정리

    인덱스 핵심 원칙:
1. 자주 조회하는 컬럼에 인덱스
2. 카디널리티가 높은 컬럼부터 복합 인덱스 구성
3. 범위 조건은 복합 인덱스 마지막에
4. EXPLAIN으로 실행 계획 반드시 확인
5. 인덱스는 읽기↑ 쓰기↓ 트레이드오프 인지
6. 과도한 인덱스는 오히려 독
