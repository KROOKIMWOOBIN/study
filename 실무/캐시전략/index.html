    캐시 전략 (Cache Strategy)

    캐시란? 왜 필요한가?

    캐시(Cache)는 자주 사용되는 데이터를 빠른 저장소에 임시로 보관하여 응답 속도를 높이는 기술이다.

    캐시 없는 구조:
  클라이언트 → 애플리케이션 → DB (매번 쿼리)
       응답: 100~300ms

캐시 있는 구조:
  클라이언트 → 애플리케이션 → Redis (캐시 히트)
       응답: 1~5ms

  클라이언트 → 애플리케이션 → Redis (캐시 미스) → DB → Redis 저장
       응답: 100~300ms (최초 1회)

    캐시 적용 효과

    구분 캐시 없음 캐시 있음
    DB 쿼리 횟수 요청마다 캐시 미스 시만
    응답 시간 100~500ms 1~10ms
    DB 부하 높음 낮음
    데이터 신선도 항상 최신 TTL에 따라 지연

    캐시 전략 종류

    1. Cache-Aside (Lazy Loading) - 가장 많이 사용

    애플리케이션이 직접 캐시와 DB를 관리하는 방식.

    [읽기 흐름]

        ┌─────────────┐
        │   Client    │
        └──────┬──────┘
               │ 요청
        ┌──────▼──────┐
        │    App      │
        └──────┬──────┘
               │ 1. 캐시 조회
        ┌──────▼──────┐     Cache Hit
        │    Redis    │ ──────────────→ 데이터 반환
        └──────┬──────┘
               │ Cache Miss
               │ 2. DB 조회
        ┌──────▼──────┐
        │     DB      │
        └──────┬──────┘
               │ 3. 캐시 저장
        ┌──────▼──────┐
        │    Redis    │
        └─────────────┘
               │ 4. 데이터 반환
        ┌──────▼──────┐
        │   Client    │
        └─────────────┘

[쓰기 흐름]
  App → DB 업데이트
  App → 캐시 삭제 (또는 업데이트)

    // Cache-Aside 구현 예시 (Spring + Redis)
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;
    private final RedisTemplate<String, User> redisTemplate;

    private static final String CACHE_KEY_PREFIX = "user:";
    private static final Duration TTL = Duration.ofMinutes(30);

    public User getUser(Long userId) {
        String cacheKey = CACHE_KEY_PREFIX + userId;

        // 1. 캐시 조회
        User cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;  // Cache Hit
        }

        // 2. DB 조회 (Cache Miss)
        User user = userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));

        // 3. 캐시 저장
        redisTemplate.opsForValue().set(cacheKey, user, TTL);

        return user;
    }

    public void updateUser(Long userId, UpdateUserRequest request) {
        // DB 업데이트
        userRepository.update(userId, request);

        // 캐시 무효화 (삭제)
        redisTemplate.delete(CACHE_KEY_PREFIX + userId);
    }
}

    장점: - 읽기 많은 서비스에 적합 - 캐시 장애 시 DB에서 직접 조회 (장애 격리) - 필요한 데이터만 캐시에 저장 (효율적)

    단점: - 최초 요청(Cache Miss)은 느림 (콜드 스타트) - 캐시와 DB 정합성 관리 필요 - 코드가 복잡해짐

    2. Write-Through

    데이터를 쓸 때 캐시와 DB를 동시에 업데이트.

    [쓰기 흐름]

  Client → App → Cache(Redis) → DB
              ↑ 동기적으로 둘 다 업데이트

[읽기 흐름]

  Client → App → Cache(Redis)  (항상 캐시에 있음)

    @Service
public class ProductService {

    public void updateProduct(Long id, Product product) {
        // 1. DB 업데이트
        productRepository.save(product);

        // 2. 캐시도 즉시 업데이트 (Write-Through)
        String cacheKey = "product:" + id;
        redisTemplate.opsForValue().set(cacheKey, product, Duration.ofHours(1));
    }

    public Product getProduct(Long id) {
        String cacheKey = "product:" + id;
        // 항상 캐시에 데이터가 있으므로 간단
        return redisTemplate.opsForValue().get(cacheKey);
    }
}

    장점: - 캐시와 DB 데이터 항상 일치 - 읽기 시 Cache Miss 거의 없음

    단점: - 쓰기 지연 증가 (캐시 + DB 모두 저장) - 읽지 않는 데이터도 캐시에 저장 (메모리 낭비) - 캐시 장애 시 쓰기 실패 가능

    3. Write-Behind (Write-Back)

    쓰기를 캐시에만 먼저 하고, 비동기로 DB에 반영.

    [쓰기 흐름]

  Client → App → Cache(Redis)  ← 즉시 응답
                    ↓ (비동기, 배치)
                   DB

[위험성]
  캐시 장애 시 DB에 반영되지 않은 데이터 손실 가능!

    @Service
public class ViewCountService {

    // Redis에 조회수 누적
    public void incrementViewCount(Long postId) {
        String key = "viewcount:" + postId;
        redisTemplate.opsForValue().increment(key);
    }

    // 배치로 DB에 반영 (Write-Behind)
    @Scheduled(fixedDelay = 60000)  // 1분마다
    public void flushViewCountsToDB() {
        Set<String> keys = redisTemplate.keys("viewcount:*");
        for (String key : keys) {
            Long count = Long.parseLong(redisTemplate.opsForValue().get(key));
            Long postId = extractPostId(key);
            postRepository.addViewCount(postId, count);
            redisTemplate.delete(key);
        }
    }
}

    장점: - 쓰기 응답 속도 매우 빠름 - DB 쓰기 부하 감소 (배치 처리)

    단점: - 데이터 유실 위험 (캐시 장애 시) - 구현 복잡도 높음 - 실시간 DB 정합성 보장 안 됨

    4. Read-Through

    캐시가 DB 앞에서 투명하게 동작. 앱은 캐시만 바라봄.

    [읽기 흐름]

  Client → App → Cache Provider (캐시 로직 포함)
                      ↓ Cache Miss 시 자동 DB 조회
                     DB
                      ↓ 자동 캐시 저장
                 Cache Provider
                      ↓ 데이터 반환
                    App → Client

    Cache-Aside vs Read-Through 차이:

    구분 Cache-Aside Read-Through
    DB 조회 주체 애플리케이션 캐시 라이브러리
    코드 복잡도 높음 낮음
    유연성 높음 낮음
    대표 구현 직접 구현 Ehcache, Spring Cache

    전략 비교 요약

    전략 읽기 성능 쓰기 성능 정합성 복잡도 적합한 상황
    Cache-Aside 높음 보통 보통 중간 읽기 많은 서비스
    Write-Through 높음 낮음 높음 낮음 정합성 중요
    Write-Behind 높음 매우 높음 낮음 높음 쓰기 폭발적
    Read-Through 높음 - 보통 낮음 단순 읽기

    캐시 문제점 및 해결 방법

    1. Cache Miss (콜드 스타트)

    서버 재시작 또는 TTL 만료 후 캐시가 비어있는 상태.

    문제:
  서버 배포 후 → 캐시 전체 비워짐
  → 모든 요청이 DB로 집중
  → DB 과부하 → 장애

해결책:
  1. Cache Warming: 배포 후 미리 캐시 채우기
  2. TTL 분산: 동시에 만료되지 않도록 랜덤 TTL

    // TTL 분산 (Cache Avalanche 방지도 겸함)
public void cacheWithRandomTtl(String key, Object value, Duration baseTtl) {
    long jitter = (long) (Math.random() * 60);  // 0~60초 랜덤
    Duration ttl = baseTtl.plusSeconds(jitter);
    redisTemplate.opsForValue().set(key, value, ttl);
}

// Cache Warming
@EventListener(ApplicationReadyEvent.class)
public void warmUpCache() {
    List<Product> hotProducts = productRepository.findTop100ByOrderBySalesDesc();
    hotProducts.forEach(p ->
        redisTemplate.opsForValue().set("product:" + p.getId(), p, Duration.ofHours(1))
    );
}

    2. Cache Stampede (Thunder Herd)

    인기 있는 캐시 키의 TTL이 만료될 때 동시에 수많은 요청이 DB로 몰리는 현상.

    문제 상황:
  인기 상품 캐시 만료 (TTL 0)
  동시에 1000개 요청 → 모두 DB 조회 시도
  → DB 과부하

해결 1: Mutex Lock (분산 락)
  첫 번째 요청만 DB 조회, 나머지는 대기
  Lock 해제 후 캐시에서 조회

해결 2: Probabilistic Early Expiration
  TTL 만료 전에 확률적으로 갱신

    // Mutex Lock으로 Cache Stampede 방지
@Service
public class HotItemService {

    private final RedisTemplate<String, Object> redisTemplate;
    private final ItemRepository itemRepository;

    public Item getHotItem(Long itemId) {
        String cacheKey = "item:" + itemId;
        String lockKey = "lock:item:" + itemId;

        // 1. 캐시 조회
        Item cached = (Item) redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) return cached;

        // 2. 분산 락 획득 시도
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", Duration.ofSeconds(3));

        if (Boolean.TRUE.equals(locked)) {
            try {
                // 락 획득 성공 → DB 조회
                Item item = itemRepository.findById(itemId).orElseThrow();
                redisTemplate.opsForValue().set(cacheKey, item, Duration.ofMinutes(10));
                return item;
            } finally {
                redisTemplate.delete(lockKey);
            }
        } else {
            // 락 획득 실패 → 잠시 대기 후 캐시 재조회
            try { Thread.sleep(100); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            return (Item) redisTemplate.opsForValue().get(cacheKey);
        }
    }
}

    3. Cache Penetration (캐시 침투)

    존재하지 않는 키를 지속적으로 요청하여 캐시를 우회하고 DB에 직접 부하를 주는 공격/현상.

    문제:
  GET /users/99999999 (없는 사용자)
  → 캐시에 없음 (당연히)
  → DB 조회 → 없음
  → 캐시에 저장 안 함 (null이니까)
  → 다음 요청도 DB로 직행
  → 악의적 대량 요청 시 DB 과부하

해결:
  1. Null 캐싱: 없는 결과도 캐시에 저장
  2. Bloom Filter: 존재 여부를 확률적으로 판단

    // Null 캐싱으로 Cache Penetration 방지
public Optional<User> getUser(Long userId) {
    String cacheKey = "user:" + userId;
    String nullMarker = "NULL";

    String cached = redisTemplate.opsForValue().get(cacheKey);

    if (nullMarker.equals(cached)) {
        return Optional.empty();  // 없는 데이터임을 캐시에서 확인
    }

    if (cached != null) {
        return Optional.of(deserialize(cached));
    }

    // DB 조회
    Optional<User> user = userRepository.findById(userId);

    if (user.isEmpty()) {
        // Null도 캐시에 저장 (짧은 TTL)
        redisTemplate.opsForValue().set(cacheKey, nullMarker, Duration.ofMinutes(5));
    } else {
        redisTemplate.opsForValue().set(cacheKey, serialize(user.get()), Duration.ofMinutes(30));
    }

    return user;
}

    4. Cache Avalanche (캐시 눈사태)

    대량의 캐시 키가 동시에 만료되어 DB에 갑작스러운 부하가 몰리는 현상.

    문제:
  오전 9시: 이벤트 시작, 수만 개 캐시를 TTL=1시간으로 저장
  오전 10시: 수만 개 캐시가 동시에 만료!
  → 모든 요청이 DB로 → DB 다운

해결:
  1. TTL 랜덤화 (jitter 추가)
  2. 캐시 사전 갱신 (만료 전 갱신)
  3. 이중 캐시 (L1: 로컬 캐시, L2: Redis)
  4. Circuit Breaker 패턴

    // TTL 랜덤화
public void saveToCache(String key, Object value) {
    // 기본 TTL 1시간 ± 최대 10분 랜덤
    long baseTtlSeconds = 3600;
    long jitter = (long) (Math.random() * 600) - 300;  // -300 ~ +300초
    Duration ttl = Duration.ofSeconds(baseTtlSeconds + jitter);

    redisTemplate.opsForValue().set(key, value, ttl);
}

    Spring Cache 추상화

    @Cacheable, @CachePut, @CacheEvict

    @Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory redisConnectionFactory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))
            .serializeKeysWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new GenericJackson2JsonRedisSerializer()));

        return RedisCacheManager.builder(redisConnectionFactory)
            .cacheDefaults(config)
            .build();
    }
}

    @Service
public class ProductService {

    // 캐시에서 먼저 조회, 없으면 메서드 실행 후 캐시 저장
    @Cacheable(value = "products", key = "#productId")
    public Product getProduct(Long productId) {
        return productRepository.findById(productId).orElseThrow();
    }

    // 항상 메서드 실행 + 결과를 캐시에 저장 (업데이트 시 사용)
    @CachePut(value = "products", key = "#product.id")
    public Product updateProduct(Product product) {
        return productRepository.save(product);
    }

    // 캐시 삭제
    @CacheEvict(value = "products", key = "#productId")
    public void deleteProduct(Long productId) {
        productRepository.deleteById(productId);
    }

    // 해당 캐시의 모든 항목 삭제
    @CacheEvict(value = "products", allEntries = true)
    public void clearAllProductCache() {
        // 빈 메서드여도 캐시 삭제됨
    }

    // 여러 캐시 동시 처리

redisTemplate.opsForHash().put("user:1", "name", "Kim");

redisTemplate.opsForList().leftPush("recent:user:1", productId);

    캐시 vs DB 정합성 문제

    데이터 불일치 시나리오:

  Thread A: 상품 가격 변경 → DB 업데이트 → 캐시 삭제 시도
  Thread B: 상품 조회 → 캐시 미스 → DB 조회 (아직 구버전) → 캐시 저장
  Thread A: 캐시 삭제 완료
  결과: 캐시에 구버전 데이터 잔존!

해결책 1: 짧은 TTL (데이터 일치 지연을 감수)
해결책 2: 쓰기 시 캐시 먼저 삭제 후 DB 업데이트
해결책 3: 분산 락으로 순서 보장
해결책 4: Event-Driven으로 캐시 갱신 (Kafka, Redis Pub/Sub)

    // Event-Driven 캐시 갱신 (Redis Pub/Sub)
@Service
public class CacheInvalidationListener {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @EventListener
    public void onProductUpdated(ProductUpdatedEvent event) {
        // 상품 업데이트 이벤트 발생 시 캐시 삭제
        redisTemplate.delete("product:" + event.getProductId());
        // 분산 환경에서는 Pub/Sub으로 다른 서버의 로컬 캐시도 무효화
        redisTemplate.convertAndSend("cache-invalidation",
            "product:" + event.getProductId());
    }
}

    캐시를 쓰는가 / 쓰면 안 되는가

    캐시를 써야 할 때

    1. 읽기 빈도가 쓰기보다 월등히 높음 (뉴스 기사, 상품 상세)
2. DB 조회가 느리고 자주 반복됨 (복잡한 JOIN 쿼리)
3. 데이터 변경이 드물고 일시적인 불일치 허용 가능
4. 실시간성이 중요하지 않은 데이터 (통계, 랭킹)
5. 외부 API 응답 캐싱 (날씨, 환율)

    캐시를 쓰면 안 될 때

    1. 데이터 실시간성이 절대적으로 중요한 경우 (재고, 잔액)
2. 개인화 데이터 (사용자마다 다른 데이터 → 캐시 효율 낮음)
3. 자주 변경되는 데이터 (빈번한 캐시 무효화 = 의미 없음)
4. 보안에 민감한 데이터 (인증 토큰, 개인정보)
5. 캐시 크기 대비 데이터가 너무 많은 경우

    정리

    캐시 전략 선택 기준:
  읽기 多, 쓰기 少, 일치성 보통  → Cache-Aside (가장 보편적)
  읽기 多, 정합성 중요            → Write-Through
  쓰기 폭발적, 손실 허용          → Write-Behind
  단순 읽기, 코드 단순화 원함     → Read-Through (@Cacheable)

핵심 주의사항:
  1. TTL 반드시 설정 (무한 캐시는 위험)
  2. TTL 분산으로 Avalanche 방지
  3. Null 캐싱으로 Penetration 방지
  4. 분산 락으로 Stampede 방지
  5. 정합성 요구사항 명확히 정의 후 전략 선택
