    낙관적 락 / 비관적 락 (Optimistic Lock / Pessimistic Lock)

    왜 필요한가? (동시성 문제 배경)

    데이터베이스는 여러 사용자가 동시에 같은 데이터를 읽고 수정할 수 있다. 이때 제어 없이 놔두면 다음과 같은 문제가 발생한다.

    동시성 문제 시나리오

    [재고: 5개]

사용자 A                    사용자 B
───────                     ───────
재고 조회: 5개              재고 조회: 5개
주문 처리 중...             주문 처리 중...
재고 = 5 - 1 = 4 저장      재고 = 5 - 1 = 4 저장

결과: 재고 4개 (실제로 2명이 주문했으므로 3개여야 함)

    이를 Lost Update(갱신 분실) 문제라고 한다.

    발생 가능한 동시성 문제

    문제 설명 예시
    Lost Update 나중 쓰기가 먼저 쓴 내용을 덮어씀 재고 차감, 포인트 적립
    Dirty Read 커밋되지 않은 데이터를 읽음 A가 수정 중인 데이터를 B가 읽음
    Non-Repeatable Read 같은 쿼리가 다른 결과를 반환 A가 두 번 조회하는 사이 B가 수정
    Phantom Read 같은 범위 쿼리가 다른 행을 반환 A가 목록 조회하는 사이 B가 INSERT

    낙관적 락 (Optimistic Lock)

    개념

    "충돌이 드물다"고 가정하고, 잠금 없이 작업한 뒤 커밋 시점에 충돌 여부를 확인한다.

    • 물리적 잠금을 걸지 않는다
    • 대신 버전(version) 컬럼으로 데이터가 변경됐는지 확인
    • 충돌이 감지되면 예외를 던지고 재시도
    읽기 시점: version=1
수정 후 저장: WHERE id=? AND version=1 → version=2로 업데이트

다른 트랜잭션이 먼저 수정했다면: version이 이미 2
→ WHERE 조건 불일치 → 0건 업데이트 → OptimisticLockException 발생

    동작 방식 다이어그램

    사용자 A                       DB                       사용자 B
────────                       ──                       ────────
                           [stock=5, ver=1]
SELECT (ver=1) ─────────→                ←───────── SELECT (ver=1)

재고 차감 계산...           재고 차감 계산...

UPDATE stock=4             UPDATE stock=4
WHERE ver=1 ──────────→   [ver=1 → ver=2]  ←── UPDATE stock=4
                           (A 성공)              WHERE ver=1
                                                (실패! ver이 이미 2)
                                                     │
                                              OptimisticLockException
                                                     │
                                              재시도 또는 사용자에게 안내

    @Version 어노테이션

    @Entity
public class Product {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private int stock;

    @Version  // JPA 낙관적 락
    private Long version;
    // 타입: Long, Integer, Short, Timestamp 모두 가능
    // JPA가 자동으로 관리 (직접 수정하면 안 됨)
}

    // Repository - 별도 설정 없이 그냥 사용하면 됨
public interface ProductRepository extends JpaRepository<Product, Long> {
}

    // Service
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;

    @Transactional
    public void decreaseStock(Long productId, int quantity) {
        Product product = productRepository.findById(productId)
            .orElseThrow(() -> new EntityNotFoundException("상품 없음"));

        if (product.getStock() < quantity) {
            throw new InsufficientStockException("재고 부족");
        }

        product.setStock(product.getStock() - quantity);
        // 트랜잭션 종료 시 자동으로 version 체크:
        // UPDATE product SET stock=?, version=version+1 WHERE id=? AND version=?
        // → 다른 트랜잭션이 먼저 수정했다면 OptimisticLockException
    }
}

    실행되는 SQL

    -- 조회
SELECT id, name, stock, version FROM product WHERE id = 1;
-- 결과: id=1, stock=5, version=1

-- 업데이트 (version 자동 체크)
UPDATE product
SET stock = 4, version = 2
WHERE id = 1 AND version = 1;  -- version 조건이 핵심!

-- 0건이 업데이트되면 OptimisticLockException

    OptimisticLockException 처리

    // LockModeType 명시적 지정
public interface ProductRepository extends JpaRepository<Product, Long> {

    @Lock(LockModeType.OPTIMISTIC)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithOptimisticLock(@Param("id") Long id);

    // OPTIMISTIC_FORCE_INCREMENT: 수정 없어도 version 증가 (읽기 일관성 강화)
    @Lock(LockModeType.OPTIMISTIC_FORCE_INCREMENT)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithForceIncrement(@Param("id") Long id);
}

    재시도 로직

    // 방법 1: @Retryable (Spring Retry)
@Service
public class OrderService {

    @Transactional
    @Retryable(
        value = OptimisticLockingFailureException.class,
        maxAttempts = 3,
        backoff = @Backoff(delay = 50, multiplier = 2) // 50ms, 100ms
    )
    public void placeOrder(Long productId, int quantity) {
        productService.decreaseStock(productId, quantity);
        // 주문 생성 로직...
    }

    @Recover
    public void recoverPlaceOrder(OptimisticLockingFailureException e,
                                   Long productId, int quantity) {
        throw new OrderFailedException("재고 경쟁으로 주문 실패. 다시 시도해주세요.");
    }
}

    // 방법 2: 수동 재시도
@Service
public class StockService {

    private static final int MAX_RETRY = 3;

    public void decreaseStockWithRetry(Long productId, int quantity) {
        for (int attempt = 0; attempt < MAX_RETRY; attempt++) {
            try {
                decreaseStock(productId, quantity);
                return; // 성공
            } catch (OptimisticLockingFailureException e) {
                if (attempt == MAX_RETRY - 1) {
                    throw new StockUpdateFailedException("재고 업데이트 실패 (재시도 초과)");
                }
                try {
                    Thread.sleep(50L * (attempt + 1)); // 점진적 대기
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new StockUpdateFailedException("인터럽트 발생");
                }
            }
        }
    }
}

    낙관적 락 장점

    • 실제 DB 잠금 없음 → 높은 처리량 (throughput)
    • 데드락 발생하지 않음
    • 충돌이 드문 경우 매우 효율적
    • 읽기 성능에 영향 없음

    낙관적 락 단점

    • 충돌이 많으면 재시도가 많아져 오히려 느림
    • 재시도 로직을 직접 구현해야 함
    • 이미 처리한 작업(외부 API 호출 등)을 롤백하기 어려울 수 있음
    • 사용자에게 "다시 시도하세요" 메시지를 보여줘야 할 수 있음

    비관적 락 (Pessimistic Lock)

    개념

    "충돌이 많다"고 가정하고, 데이터를 읽을 때부터 물리적 잠금(DB Lock)을 걸어 다른 트랜잭션의 접근을 차단한다.

    • SELECT FOR UPDATE 또는 SELECT FOR SHARE 사용
    • 잠금을 보유한 트랜잭션이 커밋/롤백할 때까지 다른 트랜잭션 대기

    동작 방식 다이어그램

    사용자 A                       DB                       사용자 B
────────                       ──                       ────────
                           [stock=5]
SELECT FOR UPDATE ───────→ [A가 잠금 획득]
                           ★ 잠금 상태 ★
                                             ←───── SELECT FOR UPDATE
                                             (대기! A가 잠금 보유 중)
재고 차감: stock=4
COMMIT ─────────────────→ [잠금 해제]
                           [stock=4, 잠금 없음]
                                             SELECT (stock=4) ──→ 재고 차감: 3
                                             COMMIT ──→ [stock=3]

결과: 정확하게 처리 (A: 5→4, B: 4→3)

    PESSIMISTIC_READ vs PESSIMISTIC_WRITE

    모드 SQL 다른 트랜잭션 읽기 다른 트랜잭션 쓰기 용도
    PESSIMISTIC_READ SELECT FOR SHARE O (공유 가능) X 다른 사람이 수정 못 하게
    PESSIMISTIC_WRITE SELECT FOR UPDATE X X 완전 독점 (가장 강력)
    PESSIMISTIC_FORCE_INCREMENT SELECT FOR UPDATE X X 버전도 같이 증가

    JPA로 비관적 락 사용

    // Repository
public interface ProductRepository extends JpaRepository<Product, Long> {

    // PESSIMISTIC_WRITE: 완전 독점 잠금
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithPessimisticLock(@Param("id") Long id);

    // PESSIMISTIC_READ: 다른 사람은 읽기만 가능
    @Lock(LockModeType.PESSIMISTIC_READ)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithReadLock(@Param("id") Long id);
}

    // Service
@Service
@RequiredArgsConstructor
public class ProductService {

    private final ProductRepository productRepository;

    @Transactional
    public void decreaseStockPessimistic(Long productId, int quantity) {
        // SELECT * FROM product WHERE id=? FOR UPDATE
        Product product = productRepository.findByIdWithPessimisticLock(productId)
            .orElseThrow(() -> new EntityNotFoundException("상품 없음"));

        // 이 시점부터 다른 트랜잭션은 이 상품에 접근 불가 (대기)
        if (product.getStock() < quantity) {
            throw new InsufficientStockException("재고 부족");
        }

        product.setStock(product.getStock() - quantity);
        // 트랜잭션 종료 시 잠금 해제 → 다음 트랜잭션이 접근 가능
    }
}

    실행되는 SQL

    -- PESSIMISTIC_WRITE
SELECT * FROM product WHERE id = 1 FOR UPDATE;

-- PESSIMISTIC_READ (MySQL)
SELECT * FROM product WHERE id = 1 FOR SHARE;

-- 잠금 타임아웃 설정
SELECT * FROM product WHERE id = 1 FOR UPDATE WAIT 3;  -- Oracle
SELECT * FROM product WHERE id = 1 FOR UPDATE SKIP LOCKED;  -- 잠금 걸린 행 건너뜀

    SKIP LOCKED 활용 (Queue 패턴)

    // 여러 워커가 동시에 작업 큐를 처리할 때
public interface JobRepository extends JpaRepository<Job, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @QueryHints({@QueryHint(name = "javax.persistence.lock.timeout", value = "-2")})
    // -2 = SKIP_LOCKED
    @Query("SELECT j FROM Job j WHERE j.status = 'PENDING' ORDER BY j.createdAt ASC")
    List<Job> findPendingJobsWithSkipLocked(Pageable pageable);
}

    비관적 락 장점

    • 충돌이 많은 환경에서 효율적
    • 재시도 로직 불필요
    • 일관성 보장이 확실함
    • 구현이 단순함 (예외 처리 코드 적음)

    비관적 락 단점

    • 잠금으로 인한 성능 저하 (대기 발생)
    • 데드락 위험 (순서가 다른 두 트랜잭션)
    • 읽기 성능에도 영향 (PESSIMISTIC_WRITE는 읽기도 차단)
    • 잠금 유지 시간이 길면 병목 발생

    언제 어떤 걸 선택하는가?

    상황 낙관적 락 비관적 락
    충돌 빈도 낮음 권장 과도한 잠금
    충돌 빈도 높음 재시도 많아 비효율 권장
    읽기 위주 권장 불필요한 잠금
    쓰기 위주 충돌 많음 권장
    데드락 위험 방지 권장 주의 필요
    정확한 재고/잔액 처리 재시도 필요 권장
    사용자가 많은 공유 데이터 상황에 따라 권장
    분산 환경 권장 (락이 DB 종속) DB 락은 분산에 비효율

    실무 선택 가이드

    Q: 같은 데이터를 동시에 수정하는 경우가 얼마나 자주 있는가?

자주 없음 (예: 사용자 프로필 수정)
└→ 낙관적 락 (@Version)

자주 있음 (예: 실시간 재고 차감, 포인트 사용)
└→ 비관적 락 (SELECT FOR UPDATE)
   단, 처리량이 중요하면:
   └→ 낙관적 락 + 재시도 or Redis 분산 락 검토

    실무 코드 예제: 재고 차감

    낙관적 락 버전

    @Entity
@Getter
public class Product {

    @Id @GeneratedValue
    private Long id;

    private String name;
    private int stock;

    @Version
    private Long version;

    public void decreaseStock(int quantity) {
        if (this.stock < quantity) {
            throw new InsufficientStockException("재고 부족: " + this.stock);
        }
        this.stock -= quantity;
    }
}

@Service
@RequiredArgsConstructor
public class StockServiceOptimistic {

    private final ProductRepository productRepository;

    @Transactional
    @Retryable(
        value = OptimisticLockingFailureException.class,
        maxAttempts = 5,
        backoff = @Backoff(delay = 100, random = true)
    )
    public void decreaseStock(Long productId, int quantity) {
        Product product = productRepository.findById(productId)
            .orElseThrow(() -> new EntityNotFoundException("상품 없음"));
        product.decreaseStock(quantity);
    }
}

    비관적 락 버전

    public interface ProductRepository extends JpaRepository<Product, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdForUpdate(@Param("id") Long id);
}

@Service
@RequiredArgsConstructor
public class StockServicePessimistic {

    private final ProductRepository productRepository;

    @Transactional
    public void decreaseStock(Long productId, int quantity) {
        Product product = productRepository.findByIdForUpdate(productId)
            .orElseThrow(() -> new EntityNotFoundException("상품 없음"));
        product.decreaseStock(quantity);
        // 메서드 종료 시 트랜잭션 커밋 → 잠금 해제
    }
}

    분산 환경에서의 한계

    비관적 락은 단일 DB에서는 잘 동작하지만, 분산 환경(다중 서버)에서는 한계가 있다.

    서버 A ──┐
          ├──→ DB (같은 DB면 비관적 락 유효)
서버 B ──┘

서버 A ──→ DB-1 (샤딩된 DB)
서버 B ──→ DB-2 (다른 DB)
→ DB 레벨 락으로는 분산 동시성 제어 불가

    분산 환경에서는 Redis 분산 락 사용

    // Redisson 라이브러리
@Service
@RequiredArgsConstructor
public class DistributedStockService {

    private final RedissonClient redissonClient;
    private final ProductRepository productRepository;

    @Transactional
    public void decreaseStock(Long productId, int quantity) throws InterruptedException {
        String lockKey = "stock:lock:" + productId;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 최대 5초 대기, 3초 보유
            boolean acquired = lock.tryLock(5, 3, TimeUnit.SECONDS);
            if (!acquired) {
                throw new LockAcquisitionException("잠금 획득 실패");
            }

            Product product = productRepository.findById(productId)
                .orElseThrow(() -> new EntityNotFoundException("상품 없음"));
            product.decreaseStock(quantity);

        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}

    격리 수준과의 관계

    격리 수준 Dirty Read Non-Repeatable Read Phantom Read 비고
    READ UNCOMMITTED 발생 발생 발생 거의 사용 안 함
    READ COMMITTED 방지 발생 발생 Oracle, PostgreSQL 기본값
    REPEATABLE READ 방지 방지 발생 MySQL InnoDB 기본값
    SERIALIZABLE 방지 방지 방지 성능 저하 큼

    낙관적/비관적 락은 격리 수준과 독립적으로 추가 제어 레이어를 제공한다.

    요약 정리

    동시성 제어가 필요한 이유: Lost Update (갱신 분실) 방지

낙관적 락 (Optimistic Lock)
├── 전략: 일단 하고, 나중에 확인
├── 방법: @Version (버전 컬럼)
├── 충돌 시: OptimisticLockException → 재시도
├── 적합: 충돌 드문 경우, 높은 처리량 필요
└── 주의: 재시도 로직 필수

비관적 락 (Pessimistic Lock)
├── 전략: 먼저 잠그고 작업
├── 방법: SELECT FOR UPDATE (@Lock + PESSIMISTIC_WRITE)
├── 충돌 시: 다른 트랜잭션 대기
├── 적합: 충돌 잦은 경우, 정확성 중요
└── 주의: 데드락, 성능 저하 가능
